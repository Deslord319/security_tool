通过 logs_analyze(analysis_type="errors") 对全量日志进行结构化分析：

8.4 关键发现

  • USB API 与 蓝牙 API 行为差异：usbManager 的3个 get 方法在 admin=null 时抛出 code=401 异常；而 bluetoothManager.getAllowedBluetoothDevices(null) 在相同条件下正常返回空数组，未抛出异常。这表明两个SDK模块对 admin 参数的校验策略不同。
  • restrictions API 表现稳定：getDisallowedPolicy(null, feature) 对4个 feature 均未产生错误日志，说明该API对null admin有较好的容错处理，直接返回 false。
  • 错误时间戳集中：3条 PeripheralService 错误均在 17:28:47.299（同一毫秒），对应 loadAllData() 初始化流程中的连续调用。

10. 缺陷与改进建议

10.1 已发现缺陷

编号严重度描述关联用例建议
BUG-001 高 usbManager API admin参数传null导致401错误。所有USB相关API（getUsbStorageDeviceAccessPolicy / getAllowedUsbDevices / getDisallowedUsbDevices）的 admin 参数传入 null 时抛出 code=401 异常 TC_PERI_030 需在MDM管理设备上通过 deviceAdmin.createAdminWant() 获取正确的 admin 参数替代 null。当前代码中所有 API 调用均传入 null 作为 admin，应考虑引入设备管理员激活流程
BUG-002 中 USB API 和 蓝牙 API 对 null admin 的行为不一致。usbManager 的 get 方法抛异常 401，而 bluetoothManager.getAllowedBluetoothDevices(null) 正常返回空数组 TC_PERI_031 SDK层差异，非应用层问题。但应在文档中标注此差异，避免后续开发者误以为行为统一
BUG-003 低 设备连接记录为硬编码模拟数据。Tab 1和Tab 3的设备记录来自 loadAllData() 中的静态数组，无实际设备枚举能力 TC_PERI_001 后续迭代应接入真实设备枚举API或持久化存储
