四层测试金字塔架构

参考 KuiTest 四层测试架构论文，SecurityTool 采用金字塔分层策略：越底层的测试数量越多、速度越快、成本越低；越顶层的测试越贴近用户、覆盖越完整、但执行越慢。

Layer 2

服务层测试 — FirewallService.test.ets

设计思路：通过 MockNetFirewall 替代真实系统 API，隔离测试防火墙服务层的业务逻辑，验证 CRUD 操作、状态管理、异常处理。

测试分组用例数测试内容

Complex scenarios2复合操作 (添加→删除/多操作序列)
Mock 设计

MockNetFirewall.ets 采用模块级变量替代 static 类（ArkTS 规范），提供：

  • resetMock() — 每个测试前重置状态
  • setFailMode() — 模拟 API 异常
  • setMockRules/Policy() — 预设数据
  • 分页逻辑模拟 (page/pageSize/totalCount)
  • 自增 ruleId 分配
设计亮点
  • beforeEach 严格隔离测试状态
  • 覆盖成功路径 + 异常路径
  • 复合场景测试验证操作序列正确性

优化方向
  • P0
    补充防火墙 UI 交互测试

    当前 L4 仅验证导航和页面加载。可增加：开关切换测试（点击 Toggle → 验证状态变化）、规则列表加载验证、添加规则弹窗交互。

  • P0
    为组件添加 testId 标识

    当前 UI 测试依赖 ON.text() 匹配中文文本，文案变更会导致测试失败。为关键组件添加 id() 属性，改用 ON.id('stat-firewall') 定位，提高稳定性。

  • P1
    增加外设管理服务层测试

    外设管理模块（USB/蓝牙）目前只有 L1 的数据解析测试，缺少 L2 的 Mock 服务测试。需要创建 MockUsbManager / MockBluetoothManager。

  • P1
    集成 CI 流水线

    将 L1-L2 本地测试集成到 Git pre-commit 或 CI 构建中，每次提交自动运行 hvigorw test，确保基础逻辑不被破坏。

  • P1
    主题切换回归测试

    应用支持浅色/深色主题切换，可在 L4 添加：点击主题按钮 → 验证界面颜色变化 → 切回原主题。防止主题适配遗漏。

  • P2
    扩展 L2 覆盖率报告

    当前仅 DataModels 有 100% 覆盖率数据。可扩展 hvigorw 配置，收集 FirewallService 等更多模块的覆盖率指标。

  • P2
    性能基线测试

    记录各页面加载时间基线，后续版本对比。利用 driver.delayMs() 前后的时间戳，检测页面渲染是否出现性能退化。
