## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 提供企业管理员场景下的外设接口总控、设备连接记录采集与单设备黑白名单策略管理能力。
- **入口路由 (Route Entry)**: `RouteIds.PERIPHERAL_MANAGE = 'peripheral-manage'`
- **对外暴露能力 (Public APIs)**:
  - `ApplicationRuntimeManager.ensureRuntimeServices()`
  - `PeripheralService`
  - `PeripheralConnectionRecordService`
  - `PeripheralDevicePolicyRepository`
- **业务边界**:
  - ✅ **包含**: USB、蓝牙、Wi-Fi、HDC、网络共享、打印机、麦克风、摄像头的接口管控；USB 接口应用侧默认管控策略；USB 存储访问策略；USB 插拔与蓝牙 ACL 连接记录采集、详情、导出、清理；基于 USB 策略状态库的 allow / deny 黑白名单策略；USB 默认策略为 deny 时，USB 设备首次连接自动下发 deny 并加入黑名单；黑白名单 USB 设备按 SN 或 `VID + PID + description` 弱指纹维护应用侧逻辑策略状态，拔出后内部恢复系统允许策略但 UI 仍显示逻辑策略。
  - ❌ **不包含**: 企业管理员激活流程本身；防火墙、日志管理、身份鉴别、工具设置等非外设模块能力；未在当前代码中接入的其它外设类型运行时采集；USB Hub 拓扑节点（`baseClass = 0x09`）作为可展示或可管控的外设资产。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `PeripheralViewModel.adminState: AdminActivationState`: 企业管理员激活状态及启用命令。
  - `PeripheralViewModel.reasonCode: string | null`: 最近一次外设业务失败原因码。
  - `PeripheralViewModel.currentTab: number`: 当前页签索引，取值由 `PeripheralTabIndexes` 统一定义，接口管控 / 设备连接记录 / 黑白名单分别保持 `0 / 1 / 2`。
  - `InterfaceControlViewModel.state: InterfaceControlStateModel`: USB、蓝牙、Wi-Fi、HDC、网络共享、打印机、麦克风、摄像头禁用状态，以及由 `USB_STORAGE_POLICY_OPTIONS` 映射的 USB 存储策略索引。`usbDisabled` 的页面文案保持“USB 接口禁用/启用”，底层含义改为应用侧 USB 默认管控策略：`true` 对应 `usb_default_policy = deny`，`false` 对应 `usb_default_policy = allow`；`networkDisabled` 的页面文案保持“网络共享”，底层含义为 restrictions `tethering` 特性禁用状态，不表示有线网络接口禁用状态。
  - `InterfaceControlViewModel.processingKey: string`: 当前正在处理的接口或 USB 存储策略 key；接口 feature key 由 `PeripheralModels.ets` 中的 `PeripheralInterfaceFeatures` 统一定义，页面和组件不得直接依赖 `PeripheralService` 静态常量。
  - `PeripheralRecordViewModel.state.records: PeripheralConnectionRecord[]`: 连接记录列表。
  - `PeripheralRecordViewModel.state.connectionDetailDialog: ConnectionDetailDialogStateModel`: 连接详情弹窗选中记录和同设备历史。
  - `PeripheralPolicyViewModel.state.records: PeripheralPolicyRecord[]`: 黑白名单页展示列表。
  - `PeripheralPolicyViewModel.state.updatingKeys: string[]`: 正在更新策略的设备标识集合。
- **关键流转路径**:
  - `进入外设页面` -> `MainPage` 传入共享 `PeripheralViewModel` -> `PeripheralPage.initViewModel()` -> 接口管控 Tab 进入短暂读取态 -> `PeripheralViewModel.initialize(context)` -> `PeripheralTraceMaintenanceService.initialize(context)` + `PeripheralDevicePolicyRepository.init(context)` -> `reloadAllData()` -> `InterfaceControlViewModel.reloadState()` 读取接口真实状态；若 Wi-Fi / 蓝牙策略状态为启用，则委托 `PeripheralService.restoreEnabledInterfacePowerWithResult(...)` 做一次幂等硬件开启恢复 -> 读取完成后首次创建 `InterfaceControlTab` 并按 ViewModel 当前状态渲染，设备连接记录 / 黑白名单 Tab 仍按原页面结构展示
  - `页面显示` -> `MainPage.handlePageActivated()` -> `ApplicationRuntimeManager.ensureRuntimeServices()` -> `RuntimeProducerAdapter + Pipeline + Consumers` -> `PeripheralTraceMaintenanceService.appendEntries()` -> `PeripheralTraceEntryRepository`
  - `切换接口或 USB 存储策略` -> `InterfaceControlTab` -> `PeripheralPage` -> `PeripheralViewModel.ensureAdminReady()` -> `InterfaceControlViewModel` -> `USB 接口写入 PeripheralDevicePolicyRepository.usb_default_policy / 其它接口或 USB 存储委托 PeripheralService`；USB 默认策略保存成功后由 `PeripheralPolicySnapshotTraceService` 枚举当前真实在场 USB 设备并写入 `policy_snapshot` 快照，枚举不到设备时不写设备记录；蓝牙全局禁用下发前由 `BluetoothPolicySnapshotTraceService` 读取本进程 ACL `CONNECTED` 当前连接缓存，禁用策略下发成功后对缓存中的设备补写 `policy_snapshot`；Wi-Fi / 蓝牙从禁用切回启用时，`PeripheralService` 在撤销 restrictions 策略后主动调用对应开启能力 -> `state 或 reasonCode 更新`
  - `USB 运行时事件进入` -> `PeripheralConnectionRecordRuntimeProducerAdapter` -> `PeripheralConnectionRecordPipelineImpl` -> `PeripheralConnectionRecordUsbConsumer` -> `解析 VID/PID/baseClass/SN/description` -> `UsbDeviceIdentityResolver` 生成强标识或弱指纹 -> `按统一 USB 名称规则生成 deviceName` -> `过滤 baseClass = 0x09 的 USB Hub 拓扑噪声` -> `UsbDevicePolicyStateService` 更新 `usb_device_policy_states` 在线态与逻辑策略 -> `按应用侧 USB 默认策略 / USB 存储策略 / 单设备逻辑策略 / 默认允许顺序生成基础判定` -> `connect 命中 deny 时下发 deny 并将 active_policy 标记为 deny` -> `disconnect 命中 active deny 时在无其它同 baseClass 在线 deny 设备的情况下内部恢复 allow 并将 active_policy 标记为 none` -> `插拔触发的内部恢复不补 policy_snapshot 或额外连接记录` -> `trace 落库`。
  - `蓝牙 ACL 事件进入` -> API 26 `connection.onAclStateChange()` -> `PeripheralConnectionRecordRuntimeProducerAdapter` 使用 `GET_BLUETOOTH_PEERS_MAC` 获取真实远端 MAC，并把 `AclState.STATE_CONNECTED / STATE_DISCONNECTED` 显式映射为内部 `connect / disconnect` -> `PeripheralConnectionRecordPipelineImpl` -> `PeripheralConnectionRecordBluetoothAclConsumer` -> `按真实 MAC 查询名称并维护连接缓存` -> `按允许接入生成连接记录` -> `trace 落库`；不再订阅 `COMMON_EVENT_BLUETOOTH_REMOTEDEVICE_ACL_STATE_CHANGE`，不再读取 `linktype`，不保留旧公共事件回退路径。
  - `黑白名单列表生成流` -> `USB 策略状态库变更` -> `PeripheralViewModel listener` -> `PeripheralPolicyViewModel.reloadRecords(...)` -> `PeripheralPolicyService.buildPolicyRecords(...)` -> `只读取 usb_device_policy_states` -> `PeripheralPolicyRecord[]` -> `PolicyList 展示 desired_policy 和 present 在线态`
  - `单设备黑白名单更新流` -> `PolicyList.onPolicyChange` -> `PeripheralPage` -> `PeripheralViewModel.handlePeripheralPolicyChange()` -> `PeripheralPolicyViewModel.setDevicePolicy()` -> `UsbDevicePolicyStateService.setPolicy()` -> `在线设备先 dispatch 再保存 desired_policy/active_policy`；离线设备不允许修改 -> 用户主动策略变更成功后仍按现有规则补写 `policy_snapshot`，插拔触发的内部 allow 恢复不得补写 -> `PeripheralPolicyViewModel.reloadRecords(...)` 从 USB 策略状态库回读真实列表和 `hasRestorableRecords` -> `PolicyList 与头部还原按钮刷新`

黑白名单页按应用侧 USB 策略状态库展示逻辑策略，不再从连接记录反推候选，也不兼容旧 `device_policies` 单设备策略。USB 设备优先使用 SN 生成 `USB-SN:<serial>` 强标识；没有 SN 时使用公共事件 `description` 与 VID/PID 生成 `USB-WEAK:<vid>:<pid>:<descriptionHash>` 弱指纹。唯一标识生成由 `UsbDeviceIdentityResolver` 负责，RDB 仓储只按 fingerprint key 持久化，不耦合 fingerprint 规则。运行时首次发现 USB 设备连接时，若状态库不存在该 fingerprint 且 `usb_default_policy = deny`，则写入 desired_policy=deny 并同步下发 deny；系统下发成功后将 active_policy 标记为 deny。`usb_default_policy = allow` 时未知 USB 首次连接写入 allow 状态但不自动下发 deny。已有 allow 或 deny 逻辑策略不被运行时自动覆盖；已有 deny 命中时在 connect 阶段确保系统 deny 下发。USB detach 时若该设备 active_policy=deny，则在没有其它同 baseClass 且 present=1/active_policy=deny 的 USB 设备时内部恢复 allow；该恢复只修正系统状态和 active_policy，不补 policy_snapshot，不补额外连接记录，UI 仍按 desired_policy 显示禁止。离线 USB 设备 present=0 时黑白名单控件禁用，不允许修改策略。清空连接记录只清理 trace，不删除在线或离线 USB 黑白名单状态，避免清记录动作误删用户维护的本地指纹策略。还原策略只把已有 USB 黑白名单状态恢复为默认允许接入：保留卡片和 fingerprint，desired_policy 置为 allow，active_policy=deny 的设备先内部下发 allow，成功后将 active_policy 置为 none；该操作不补 policy_snapshot，不补连接记录，也不删除状态行。黑白名单页头部按钮必须使用“还原”语义，保留红色风险提示样式，但不使用删除图标或删除确认文案；仅当列表存在禁止接入记录时可点击，全部允许接入或无记录时保持不可点击态。USB Hub 拓扑节点（`baseClass = 0x09`）在 USB Consumer 解析后、策略判定前被视为运行时噪声过滤，不生成连接记录，不进入黑白名单候选，不触发首次自动拉黑，也不尝试系统策略下发。USB_STORAGE 设备在 USB 存储总策略为 `DISABLED` 时属于账户级禁止访问场景，USB Consumer 不执行首次自动拉黑，不下发设备级 deny，也不写入本地 deny；连接记录仍按 USB 存储已禁用语义落库。蓝牙设备仍进入设备连接记录，但不进入黑白名单候选列表，不触发 allow / deny 策略下发，也不写入 USB 策略状态库。策略变更产生的 `policy_snapshot` 只表示用户主动策略变更时通过当前可信来源发现的真实在场或连接设备，底层动作为 `discover`，页面展示为“策略变更”，不得伪装为 `connect` 或 `disconnect`；插拔触发的内部 allow 恢复和还原策略不得补快照。

## 4. 模块结构与组件设计 (Module Components)

### 【核心层】(Core MVVM & Domain Layers)

#### 4.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `PeripheralType`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 定义外设类型枚举。
    * **关键字段**: `USB_STORAGE / USB_CAMERA / USB_PRINTER / USB_HID / USB_OTHER / BLUETOOTH` (当前支持的外设类型)
    * **展示职责**: 设备类型、事件来源和动作文案由 `PeripheralRecordPresentationMapper` 输出，`PeripheralModels.ets` 不直接依赖模块字符串常量。
  * `UsbStoragePolicy`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 表示 USB 存储设备访问策略。
    * **关键字段**: `READ_WRITE` (读写), `READ_ONLY` (只读), `DISABLED` (禁止访问)
  * `PeripheralInterfaceFeature` / `PeripheralInterfaceFeatures`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 定义接口管控 feature key，作为页面、组件、ViewModel 和 Service 之间的共享领域常量。
    * **边界约束**: UI 层只引用模型常量，不直接引用 `PeripheralService.FEATURE_*`；`PeripheralService` 仅消费这些 key 并负责系统策略读写。
  * `PeripheralConnectionRecord`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 页面层连接记录展示模型。
    * **关键字段**: `deviceId` (设备标识), `action` (connect / disconnect), `result` (`SecurityEventResult`), `policyHit` (allow / deny / unknown), `effectivePolicyLabel` (列表策略展示), `matchedPolicyKind` (命中类型)
  * `PeripheralTraceRecord`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: RDB 持久化的外设运行记录模型。
    * **关键字段**: `recordId` (唯一记录键), `occurAt` (事件发生时间), `result` (`SecurityEventResult`), `source` (runtime_event / audit_event / policy_snapshot), `rawPayload` (原始事件快照), `usbPolicy` (USB 存储策略快照)
  * `PeripheralTraceQuery`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 外设 trace 仓库查询条件。
    * **关键字段**: `startTime/endTime`, `deviceId`, `keyword`, `deviceTypes/actions/decisions/results/sources`；分页由调用方通过仓库方法参数或上层 ViewModel 控制，不在 Query 模型中定义未消费字段。
  * `PeripheralTraceStats`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 安全总览卡片使用的外设 trace 汇总。
    * **关键字段**: `total`, `connects`, `disconnects`, `policyChanges`；安全总览外设卡片统计 `runtime_event` 的 connect / disconnect 和 `policy_snapshot` 的 discover，策略变更必须单独展示为策略变更次数，不得混入连接或断开次数。
  * `PeripheralDevicePolicySnapshotRequest`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 用户主动修改 USB 黑白名单后用于补写策略快照的输入模型；不是持久化黑白名单模型。
    * **关键字段**: `deviceId` (VID/PID 展示标识), `policy` (allow / deny), `updatedAt` (最后更新时间), `deviceName`, `deviceType`
  * `UsbDeviceIdentity`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: USB 设备唯一标识解析结果，隔离 fingerprint 生成规则和数据库持久化。
    * **关键字段**: `fingerprintKey` (SN 强标识或弱指纹), `fingerprintType` (`serial / weak`), `vendorId`, `productId`, `description`, `baseClass`, `deviceName`
  * `UsbDevicePolicyStateRecord`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: USB 黑白名单应用侧逻辑策略和当前系统下发态持久化模型。
    * **关键字段**: `fingerprintKey`, `desiredPolicy`, `present`, `activePolicy`, `lastSeenAt`, `updatedAt`；不保存连接记录 ID、lastError、lastDispatchAt 等可推导或诊断字段。
  * `PeripheralPolicyRecord`
    * **所在文件**: `entry/src/main/ets/models/PeripheralModels.ets`
    * **业务作用**: 黑白名单页组合展示模型。
    * **关键字段**: `deviceId` (候选设备键 / USB fingerprintKey), `lastSeenAt` (最近连接时间), `policyIndex` (由 `PeripheralPolicyOptionIndexes` 定义，`0` 允许 / `1` 禁止), `present` (是否在线), `editable` (是否允许修改)

#### 4.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/peripheral/**`
* **核心业务用例 (Use Cases)**:
  * `PeripheralService.setInterfaceDisabledWithResult(feature, disallow)`: 下发非 USB 接口启用 / 禁用策略；`feature` 必须来自 `PeripheralInterfaceFeatures`。
    * **副作用**: 调用 restrictions 系统 API；USB 接口不再通过该路径调用 `restrictions.setDisallowedPolicy(..., 'usb', ...)`。网络共享通过 restrictions `tethering` 特性读写，不再使用 `networkManager.setNetworkInterfaceDisabledSync(...)` 禁用有线网络接口。Wi-Fi / 蓝牙从禁用切回启用时，在 restrictions 策略撤销成功后继续调用 `restoreEnabledInterfacePowerWithResult(...)` 主动开启硬件能力。
    * **失败策略**: 返回 `OperationResult`，保留错误码、管理员缺失标志和冲突 stage。
  * `PeripheralService.restoreEnabledInterfacePowerWithResult(feature)`: 对策略已启用的 Wi-Fi / 蓝牙执行启动恢复。
    * **副作用**: Wi-Fi 调用 `wifiManager.turnOnWifi(ADMIN_WANT, false)`，不使用强制开启；蓝牙调用 `bluetoothManager.turnOnBluetooth(ADMIN_WANT)`，依赖 `ENTERPRISE_MANAGE_BLUETOOTH`；其它接口直接返回成功。
    * **失败策略**: 返回 `OperationResult`，并通过 `turn_on_wifi` / `turn_on_bluetooth` stage 标识恢复失败；页面初始化时仅记录日志并保持接口管控可展示，用户主动切换启用失败时按接口切换失败提示处理。
  * `PeripheralService.setUsbStoragePolicyWithCleanupResult(policy)`: 设置 USB 存储访问策略。
    * **副作用**: 调用 `usbManager.setUsbStorageDeviceAccessPolicy(...)`。
    * **失败策略**: 系统返回 `9200010` 时映射为 USB 冲突类失败。
  * `PeripheralConnectionRecordService.queryRecords(filter?)`: 查询连接记录。
    * **副作用**: 读取 `PeripheralTraceEntryRepository`。
    * **失败策略**: 记录错误并返回空数组。
  * `PeripheralTraceMaintenanceService.initialize(context)`: 初始化 trace 仓库并对齐 retention / trim。
    * **副作用**: 初始化 RDB、删除过期记录、裁剪最大条数。
    * **失败策略**: 返回 `storage_init_failed`。
  * `PeripheralTraceMaintenanceService.appendEntries(records)`: 标准化并写入 trace。
    * **副作用**: `upsertBatch` 写入 RDB；若本批新增蓝牙记录带有有效设备名，则按同一规范化 MAC 刷新历史记录中 `device_name = 未命名蓝牙设备` 的记录；最后触发变更监听。
    * **失败策略**: 仓库未初始化返回 `not_initialized`，写入失败返回 `save_failed`。
  * `PeripheralConnectionRecordRuntimeProducerAdapter.start()`: 启动外设运行时事件订阅。
    * **副作用**: USB 继续订阅 attach / detach common event；蓝牙直接注册 API 26 `connection.onAclStateChange()`，回调仅转换真实 `deviceId` 和最终连接状态，不依赖蓝牙开关状态重复注册。回调在 runtime stop 前保持注册，保证策略禁用蓝牙时随后投递的断开最终态仍可进入管线。
    * **失败策略**: 无 context、`ACCESS_BLUETOOTH` 未授权、`GET_BLUETOOTH_PEERS_MAC` 未生效或 ACL 回调注册失败时记录降级状态并令 `bluetoothRuntimeEvents=false`；不回退旧 ACL common event，也不订阅私有 `ACL_CONNECTED_REALMAC / ACL_DISCONNECTED_REALMAC` 事件。
  * `PeripheralConnectionRecordPipelineImpl.push(event)`: 分发运行时事件并写入 trace。
    * **副作用**: 按 channel 分发给 USB / Bluetooth Consumer，对 USB 事件执行短时间去重，并调用 `PeripheralTraceMaintenanceService.appendEntries(...)` 落库；外设模块不再承载身份鉴别 UKey 注册、删除或 CustomAuth 凭据生命周期副作用。
    * **失败策略**: trace 仓库未就绪时仅跳过需要落库的 trace record；Consumer 无法处理或写入失败时记录日志并返回未持久化结果。
  * `PeripheralConnectionRecordUsbConsumer.consume(event)`: 将 USB common event 转换为外设 trace。
    * **副作用**: 解析 payload 中的 VID/PID、baseClass、SN、description、设备名称和时间；USB 设备名统一按 `manufacturerName + productName` 优先，其次 `deviceName`，最后按 VID/PID 是否有效回退为“USB 设备”或“未识别 USB 设备”；`name` 字段不作为展示设备名来源，避免系统枚举名或泛化 payload 字段进入连接记录；对 USB Hub 拓扑节点（`baseClass = 0x09`）直接返回空消费结果；通过 `UsbDevicePolicyStateService` 更新 USB 状态库并按需下发 deny 或内部恢复 allow；USB_STORAGE 且 USB 存储策略为 `DISABLED` 时跳过该自动拉黑动作；最终生成 `PeripheralTraceRecord`。
    * **失败策略**: 不支持的 USB 事件、无法解析的事件或 USB Hub 拓扑噪声返回 `null`。
  * `UsbDeviceIdentityResolver`: USB 设备唯一标识解析服务。
    * **副作用**: 无；只根据事件或枚举数据生成 `UsbDeviceIdentity`，不读写数据库，不调用 MDM。
    * **失败策略**: VID/PID 无效时返回 `null`，调用方按未识别 USB 连接记录处理，不进入黑白名单策略状态。
  * `UsbDevicePolicyStateRepository`: USB 黑白名单策略状态 RDB 仓储。
    * **副作用**: 维护 `usb_device_policy_states` 表，字段限定为 fingerprint、VID/PID、description、baseClass、deviceName、desiredPolicy、present、activePolicy、lastSeenAt、updatedAt。
    * **失败策略**: RDB 未初始化或写入失败时返回失败并记录日志，不阻断连接记录基础落库。
  * `UsbDevicePolicyStateService`: USB 策略状态领域服务。
    * **副作用**: connect 时创建/刷新策略状态并按 desiredPolicy 下发 deny；disconnect 时按 activePolicy 和同 baseClass 在线 deny 设备数量决定是否内部恢复 allow；用户主动修改策略时拒绝离线设备修改；还原策略时保留状态行并恢复 desired_policy=allow。
    * **失败策略**: 策略下发失败时不更新 activePolicy 为目标状态；内部恢复失败只记录错误并保留 activePolicy，后续重试仍有依据。
  * `PeripheralPolicySnapshotTraceService`: 策略变更后的真实 USB 在场设备快照补偿服务。
    * **副作用**: 通过 `@kit.BasicServicesKit.usbManager.getDevices()` 枚举当前 USB 设备；设备类型只使用当前枚举结果中的 `USBDevice.clazz` 或 `configs[].interfaces[].clazz`，不使用策略记录、历史连接记录或审计事件兜底；设备名复用统一 USB 名称规则，不单独把 manufacturer、product 或 `USBDevice.name` 当作完整名称；过滤 USB Hub、无合法 VID/PID 和无有效 class 的设备；USB 默认策略、单设备策略或 USB 存储策略变更时为真实在场设备写入 `source = policy_snapshot`、`action = discover` 的 trace；单设备策略变更时仅在当前枚举结果中精确匹配同一 VID/PID 后写入快照；USB 存储策略变更时只针对当前枚举到的 USB_STORAGE 设备补写策略快照；快照写入委托 `PeripheralTraceMaintenanceService.appendEntries(...)`。
    * **失败策略**: 枚举失败、trace 仓储未就绪或写入失败只记录日志并返回失败结果，不回滚已经成功下发或保存的策略；枚举为空、Hub、无 VID/PID、单设备未匹配当前在场设备时返回成功且不写设备记录，避免伪造连接或在场数据。
  * `BluetoothPolicySnapshotTraceService`: 蓝牙全局禁用策略变更后的真实连接快照补偿服务。
    * **副作用**: 在禁用策略下发前读取本进程 ACL `CONNECTED` 当前连接缓存并去重；设备名称优先通过 `connection.getRemoteDeviceName(mac, false)` 读取真实远端名，读取到有效名称时同步刷新本进程 MAC 名称缓存；读取失败或为空时可回退使用同 MAC 最近一次有效名称，仍无有效名称时使用“未命名蓝牙设备”；不使用 ACL payload 名称字段或 alias 名称；策略下发成功后对缓存设备写入 `source = policy_snapshot`、`action = discover`、`decision = deny`、`result = BLOCKED`、`rawEventName = bluetooth_global_policy`、`matchedPolicyKind = bluetooth_disabled` 的 trace。
    * **失败策略**: 名称查询失败、无 ACL 当前连接缓存、trace 仓储未就绪或写入失败均不回滚已经成功下发的 restrictions 策略；缓存为空时返回成功且不写设备记录；策略快照不抑制后续真实 ACL `DISCONNECTED`，断开事件仍由蓝牙 Consumer 按普通运行时事件处理。
  * `PeripheralConnectionRecordBluetoothAclConsumer.consume(event)`: 将已规范化的蓝牙 ACL 内部事件转换为外设 trace。
    * **副作用**: 直接读取 Producer 提供的真实 MAC 和 `connect / disconnect` 动作；通过 `connection.getRemoteDeviceName(mac, false)` 查询名称，连接时刷新 `BluetoothPolicySnapshotTraceService` 的本进程当前连接缓存，断开时先使用同一真实 MAC 名称缓存兜底再移除当前连接缓存；不读取 `linktype`，不读取旧 common event payload，不执行时间窗口去重，不调用黑白名单策略下发。
    * **失败策略**: 真实 MAC 缺失或内部动作不是 `connect / disconnect` 时返回 `null`；名称查询失败时允许按“未命名蓝牙设备”落库，不使用随机 MAC 再构造第二设备。
  * `PeripheralDevicePolicyDispatchService.dispatch(request)`: 将 USB 黑白名单策略下发到系统；USB 按设备类型映射为 MDM USB baseClass 下发。
    * **副作用**: 调用 `usbManager` 的 disallow API；USB 下发粒度为 `PeripheralType` 对应的 `UsbDeviceType`，不是 VID/PID 单设备。
    * **失败策略**: 检查全局接口禁用、USB 存储冲突和系统下发冲突，失败时返回 `PeripheralPolicyDispatchReason`。
  * `PeripheralDevicePolicyRepository.setUsbDefaultPolicy(policy)`: 保存 USB 接口应用侧默认管控策略。
    * **副作用**: 写入 `peripheral_device_policy` store 下的 `usb_default_policy` key，并触发 change listener。
    * **失败策略**: 记录错误并返回 `false`；默认读取值为 `allow`，避免升级后未知 USB 设备被默认拦截。
  * `PeripheralPolicyService.buildPolicyRecords(usbPolicyStates)`: 生成黑白名单页展示列表。
    * **副作用**: 纯内存转换；只接纳 `usb_device_policy_states`，不从连接记录反推候选，不叠加旧 `PeripheralDevicePolicyRepository.device_policies`。
    * **失败策略**: 跳过状态异常或无法展示的 USB 策略状态。
* **内部架构设计**:
  * **Repository**: `entry/src/main/ets/services/peripheral/connection-record/PeripheralTraceEntryRepository.ets` - 外设运行记录的 RDB 真源，表名为 `peripheral_trace_entries`。
  * **Repository**: `entry/src/main/ets/services/peripheral/device-policy/PeripheralDevicePolicyRepository.ets` - USB 默认管控策略的 Preferences 真源，是 `peripheral_device_policy` store 的唯一 owner；旧单设备策略键不再参与黑白名单链路。
  * **Mapper/Ports**: `entry/src/main/ets/services/peripheral/connection-record/PeripheralConnectionRecordPayloadUtils.ets` - USB 运行时 common event payload 提取工具；蓝牙链路不再使用该工具。
  * **Mapper/Ports**: `entry/src/main/ets/viewmodels/peripheral/connection-record/PeripheralRecordPresentationMapper.ets` - 连接记录展示语义转换，包括设备类型、事件来源、动作和策略结果；时间格式委托公共 `DateTimeFormatUtils`，不依赖日志管理展示层。

设备连接记录的详细流程如下：

- `PeripheralConnectionRecordRuntimeProducerAdapter` 负责系统事件入口。USB 继续订阅 attach / detach common event；蓝牙使用 API 26 `connection.onAclStateChange()`，依赖 `ACCESS_BLUETOOTH + GET_BLUETOOTH_PEERS_MAC` 获取真实远端地址。Producer 显式把新接口 `AclState.STATE_CONNECTED = 0` 映射为 `connect`、`AclState.STATE_DISCONNECTED = 1` 映射为 `disconnect`，不得复用旧 common event 的 `0 = disconnect / 1 = connect` 数值规则。ACL 回调注册后只在 runtime stop 时释放，蓝牙关闭过程不主动注销，确保策略禁用触发的断开仍可进入 Consumer。
- `PeripheralConnectionRecordProducerEvent.rawData` 继续只承载 USB `commonEventManager.CommonEventData`；蓝牙通过专用 `bluetoothAcl` 内部载荷传递真实 `deviceId`、规范化动作和采集时间，不伪造 common event。蓝牙回调注册失败时直接令 `bluetoothRuntimeEvents=false`，不兼容旧公共事件、不订阅私有 REALMAC 事件、不通过 `addPersistentDeviceId()` 维持虚拟地址。
- `PeripheralConnectionRecordPipelineImpl` 负责事件接力。它绑定 context、挂载 Producer、连接 Consumers，根据 `channel` 分发到 USB 或蓝牙 Consumer；USB 事件会按 `deviceId + action + rawEventName` 做短时间去重。
- `PeripheralConnectionRecordPipelineImpl` 只处理外设连接记录采集与外设策略派生，不再挂载身份鉴别或 UKey 凭据生命周期 Consumer。UKey 插拔、首把绑定和 CustomAuth 凭据注册/删除已迁移到独立 `ukey/` 系统应用，由该应用自行订阅或探测 USB/DDK 状态。
- `PeripheralConnectionRecordUsbConsumer` 负责 USB 事件业务判定。它从 payload 中提取 VID/PID、baseClass、SN、description、设备名称和时间；USB 设备名与策略快照共用同一规则：`manufacturerName + productName` 两者同时存在时作为设备名，其次使用明确的 `deviceName`，否则按 VID/PID 是否有效显示“USB 设备”或“未识别 USB 设备”，不单独使用 manufacturer、product 或 `name` 作为完整设备名；若 `baseClass = 0x09`，该事件被识别为 USB Hub 拓扑噪声并直接返回 `null`，不进入 trace 落库、黑白名单候选或自动拉黑链路；其它 USB 事件继续推断 `PeripheralType`，并按应用侧 USB 默认策略、USB 存储禁止访问、USB 状态库 desired_policy、USB 存储只读、默认允许的顺序生成基础策略命中结果。
- `PeripheralConnectionRecordUsbConsumer` 在非 Hub USB connect 事件中基于 USB 策略状态库执行首次连接自动拉黑：若本次基础命中不是 `usb_storage_disabled`，`usb_device_policy_states` 不存在该 fingerprint，且 `PeripheralDevicePolicyRepository.isUsbDefaultDeny()` 为 true，则调用 `PeripheralDevicePolicyDispatchService.dispatch({ policy: 'deny' })`；下发成功后写入 desired_policy=deny / active_policy=deny；下发失败不更新 active_policy，并在本次 trace 详情中保留失败原因。应用侧 USB 默认策略为 allow 时，未知 USB 首次连接写入 allow 状态但不自动拉黑。
- `PeripheralConnectionRecordUsbConsumer` 对 USB_STORAGE 且 USB 存储总策略为 `DISABLED` 的连接事件不执行首次自动拉黑。该场景代表账户级 USB 存储禁止访问已生效，记录仍按 `usb_storage_disabled` 展示，不额外制造设备级 deny，避免后续恢复 USB 存储总策略或清空黑白名单时被设备级策略叠加阻塞。
- `PeripheralConnectionRecordBluetoothAclConsumer` 负责蓝牙 ACL 事件业务判定。它只消费 Producer 的真实 MAC 和规范化最终动作，并按允许接入语义生成连接记录；连接时刷新本进程当前连接缓存和真实 MAC 名称缓存，供蓝牙全局禁用前置快照和后续断开展示使用，断开时先按同一真实 MAC 读取名称缓存再移除当前连接缓存并正常落库。设备名称优先读取 `connection.getRemoteDeviceName(realMac, false)`，读取失败或为空时回退同一真实 MAC 最近一次有效名称，仍无有效名称时使用“未命名蓝牙设备”。蓝牙设备不参与黑白名单候选构建，不读取本地单设备策略，不执行首次连接自动拉黑。
- 蓝牙链路不再读取 `linktype`，不根据名称或传输类型过滤事件，也不对同 MAC、同状态增加短时间去重。应用卸载重装后从真实 MAC 新记录开始，不迁移、不合并旧版本已经落库的虚拟 MAC 记录。
- `PeripheralTraceMaintenanceService` 负责落库前标准化。它补齐缺省字段、规范化 action / decision / source / result，并写入 `PeripheralTraceEntryRepository`；同一 MAC 后续蓝牙记录解析到有效设备名时，会刷新历史记录中仍为“未命名蓝牙设备”的同 MAC 蓝牙记录名称；最后触发仓库变更通知。
- `PeripheralConnectionRecordService` 负责页面查询。它从 `PeripheralTraceEntryRepository` 查询 runtime / audit 记录，并映射为 `PeripheralConnectionRecord[]`。

USB 冲突与单设备策略冲突是当前领域设计的关键规则：

- USB 接口总控不再使用 restrictions 的系统级 USB 全局禁用能力，也不再通过 USB 接口开关自动恢复 USB 存储策略。`接口管控 > USB接口` 仅维护应用侧 `usb_default_policy`，用于控制未知 USB 首次连接是否自动拉黑。
- USB 接口总控保存成功后会补写策略快照，但快照数据必须来自 `usbManager.getDevices()` 当前枚举结果，不能用历史连接记录、策略仓库或审计事件推断设备当前在场状态，也不能用这些历史数据兜底设备类型。快照动作为 `discover`，来源为 `policy_snapshot`，表示“策略变更时发现当前在场设备”，不表示真实插入连接。
- 蓝牙全局禁用保存成功后会补写策略快照，但快照数据只来自禁用下发前本进程自启动后收到的 ACL `CONNECTED` 当前连接缓存；不再调用 Classic profile 的 a2dp/hfp/hid/pan `getConnectedDevices()` 或 BLE `getConnectedBLEDevices()` 做前后枚举。审计事件 `disabled_bluetooth` 只用于日志管理表达策略变更发生，不进入外设连接记录的设备快照补偿；系统 profile 枚举、历史连接记录、配对记录或审计事件不得用于伪造设备 MAC 或名称。
- USB 存储策略保存成功后会补写当前 USB_STORAGE 设备的策略快照。快照数据同样必须来自 `usbManager.getDevices()` 当前枚举结果，action 固定为 `discover`，页面展示为“策略变更”；`DISABLED / READ_ONLY / READ_WRITE` 只影响策略结论、结果状态、USB 存储策略快照和命中类型，不伪装成真实连接或断开。没有当前在场 USB_STORAGE 设备时不补记录。
- `PeripheralService.setUsbStoragePolicyWithCleanupResult(...)` 设置 USB 存储策略时同样先读取当前 USB 策略快照，系统返回 `9200010` 时映射为 USB 冲突失败，而不是简单未知错误。
- 黑白名单策略不是“先保存本地，再尝试系统下发”。当前实现是 `PeripheralPolicyViewModel` 先调用 `PeripheralDevicePolicyDispatchService.dispatch(...)`，只有系统下发成功后才写入 `PeripheralDevicePolicyRepository`。
- 单设备黑白名单策略保存成功后只在 `usbManager.getDevices()` 当前枚举结果中精确匹配到同一 VID/PID，且能从当前枚举数据取得有效 class 时补写 `policy_snapshot`；未匹配或无有效 class 时只更新策略，不写当前设备快照。
- 首次连接自动拉黑同样遵循“先系统下发，再保存本地”。USB Consumer 只在非 Hub USB connect 事件、本地无既有策略且 `usb_default_policy = deny` 时触发自动 deny；已有 allow 或 deny 均不覆盖，已有 deny 命中不重复调用 `addDisallowedUsbDevices`，disconnect 事件不触发。USB Hub 拓扑节点（`baseClass = 0x09`）在策略判定前过滤，不触发自动 deny。USB_STORAGE 在 USB 存储总策略 `DISABLED` 下跳过首次自动拉黑，保留账户级禁用的连接记录语义。蓝牙 Consumer 不触发自动 deny。
- USB 黑白名单在本地按规范化 deviceId 管理和回显，但系统侧下发粒度是设备类型：`PeripheralDevicePolicyDispatchService.toUsbDeviceType(...)` 将 `PeripheralType` 映射为 MDM `UsbDeviceType.baseClass` 后调用 `usbManager`。当前代码没有按 VID/PID 对单个 USB 物理设备下发系统策略。因此 USB 首次连接自动拉黑可能影响同类 USB 设备，该误伤风险在当前规格中可接受。
- USB 黑白名单下发不再检查 restrictions 的 USB 全局禁用状态；USB 存储设备设置 allow 或 deny 前仍会检查 USB 存储账户级策略是否为 `DISABLED`。当 USB 存储总策略为 `DISABLED` 时，黑白名单仍按本地单设备策略展示，但 USB_STORAGE 单设备策略不允许继续下发 allow 或 deny，避免账户级禁止访问和设备级策略叠加。

黑白名单列表生成与策略下发的详细流程如下：

- `PeripheralPolicyService.buildPolicyRecords(...)` 只从 `usb_device_policy_states` 构建黑白名单页面记录，连接记录不再作为候选来源。
- `PeripheralPolicyService.normalizeDeviceIdForPolicy(...)` 只规范化 fingerprintKey，非 `USB-SN:` / `USB-WEAK:` 的旧 VID/PID 标识不再允许作为黑白名单修改入口。
- fingerprintKey 是黑白名单策略的本地业务 key，决定策略状态查询、策略保存和列表回显；其中 USB 的系统下发仍按设备类型/baseClass 执行。
- `PeripheralDevicePolicyDispatchService.dispatch(...)` 负责真实系统策略下发。USB 按 `PeripheralType` 映射的 `UsbDeviceType.baseClass` 下发。下发成功后才允许 `UsbDevicePolicyStateRepository` 保存 active_policy；下发失败时 active_policy 不更新为目标状态。
- USB 首次连接自动拉黑成功后，连接记录应展示为 deny / blocked，并保留“首次接入已自动加入黑名单”类详情。自动拉黑下发失败时，连接记录仍应落库，保留原策略判定结果，并在详情中追加失败原因，避免设备接入事实因策略下发失败而丢失。USB_STORAGE + USB 存储 `DISABLED` 场景不展示自动拉黑成功或失败语义，只展示 USB 存储已禁用。蓝牙连接记录不展示自动拉黑语义，策略结论记录为允许接入。
- USB_STORAGE + USB 存储 `READ_ONLY` 场景下若 USB 状态库 desired_policy=deny，连接事件应优先展示 `device_deny` 和已禁用，不得被 `usb_storage_read_only` 提前返回覆盖；断开事件和黑白名单同样以 desired_policy=deny 展示已禁用 / 禁止接入。

#### 4.3 ViewModel (视图模型层)
* **真实文件路径**: `entry/src/main/ets/viewmodels/peripheral/**`
* **状态分发逻辑**:
  * `PeripheralViewModel.initialize(context)`: 初始化 trace 维护服务和设备策略仓库，加载接口状态、连接记录与黑白名单，并注册 trace / policy listener。
  * `PeripheralViewModel.reloadAllData()`: 刷新管理员状态、接口状态、连接记录和黑白名单数据。
  * `PeripheralViewModel.toggleInterface(feature, disallow)`: 校验企业管理员状态后委托 `InterfaceControlViewModel` 下发接口策略。
  * `PeripheralViewModel.setUsbStoragePolicy(index)`: 校验企业管理员状态后下发 USB 存储策略，并在成功后刷新展示状态。
  * `PeripheralViewModel.handlePeripheralPolicyChange(...)`: 先标记 updating，再触发 `PeripheralPolicyViewModel.setDevicePolicy(...)`，成功后刷新记录和策略列表，最后清理 updating 状态。
  * `PeripheralViewModel.clearConnectionHistory()`: 通过 `PeripheralConnectionRecordClearUsecase` 清空 trace，再刷新连接记录；不删除在线或离线 USB 黑白名单状态。
  * `PeripheralViewModel.clearPeripheralPolicyRecords()`: 还原 USB 黑白名单策略状态并重建黑白名单列表。
  * `InterfaceControlViewModel.reloadState()`: 从 `PeripheralService` 读取接口状态和 USB 存储策略。
  * `PeripheralRecordViewModel.reloadRecords()`: 从 `PeripheralConnectionRecordService` 加载连接记录，并维护详情弹窗历史。
  * `PeripheralPolicyViewModel.reloadRecords()`: 基于 USB 策略状态库重建黑白名单页展示列表，并同步 `PeripheralPolicyState.hasRestorableRecords`。

界面展示设备时的策略判定由 ViewModel 组合链路保证：

- 连接记录页展示的是 trace 中已经落库的策略结果。`PeripheralRecordViewModel` 不会重新计算当前策略，而是直接展示 `PeripheralConnectionRecord` 中的 `effectivePolicyLabel`、`policyHit` 和 `matchedPolicyKind`。
- trace 变化后，`PeripheralViewModel.refreshTraceDrivenData()` 只调用 `PeripheralRecordViewModel.reloadRecords()` 刷新连接记录；黑白名单不再依赖连接记录候选，不因清空 trace 删除本地策略状态。
- 策略仓库变化后，`PeripheralViewModel.refreshPolicyDrivenData()` 只需要基于 USB 策略状态库重建黑白名单列表，不需要重新采集运行时事件。
- `PeripheralRecordViewModel.openConnectionDetailDialog(record)` 基于当前 `records` 构建同设备历史；`exportConnectionHistory()` 将当前连接记录导出为 CSV，表头、字段和 BOM 由公共 CSV 服务统一生成。
- 黑白名单页展示的是 `usb_device_policy_states` 中的 USB 逻辑策略状态。`PeripheralPolicyViewModel` 通过 `PeripheralPolicyService.buildPolicyRecords(...)` 将状态库记录转换为页面记录；蓝牙连接记录和历史 USB 连接记录不进入该列表。
- `PeripheralPolicyViewModel.setDevicePolicy(...)` 负责系统下发与本地状态更新；`clearAllPolicies()` 还原策略时只处理 USB 状态库记录，将卡片恢复为默认允许接入，不删除状态行，也不回滚或兼容旧 `device_policies`。

#### 4.4 View / Page (页面视图层)
* **真实文件路径**: `entry/src/main/ets/views/peripheral/overview/PeripheralPage.ets`
* **结构说明**:
  * `PeripheralPage`: 外设管理页组合层。接收 `MainPage` 持有的共享 `PeripheralViewModel`，负责页面初始化、Tab 切换、USB loading dialog、导出提示、清理确认弹窗和连接详情弹窗，并把用户操作转发给 `PeripheralViewModel`。接口管控 Tab 不再维护页面本地 ready/loading 门控，进入页面时直接创建 `InterfaceControlTab` 并绑定共享 ViewModel 当前状态，初始化完成后由 `InterfaceControlViewModel.reloadState()` 的状态提交驱动控件刷新，避免接口管控内容区出现左上角独立加载动画。
  * 页面包含三个 Tab：`接口管控`、`设备连接记录`、`黑白名单`。
  * 页面不负责启动运行时采集。运行时采集由 `MainPage` 通过 `ApplicationRuntimeManager.ensureRuntimeServices()` 启动。

#### 4.5 Components (可复用组件层)
* **真实文件路径**: `entry/src/main/ets/components/peripheral/**`
* **结构说明**:
  * `InterfaceControlTab`: 接口管控 UI。**入参/事件**: 接收各接口禁用状态、USB 存储策略索引、processingKey；抛出 `onInterfaceToggle`、`onUsbInterfaceToggleAsync`、`onUsbStoragePolicyChange`。该组件直接绑定 `PeripheralPage` 传入的 ViewModel 当前状态，组件内部不负责读取系统状态或维护初始化门控。接口行左侧保留标题和描述文本，右侧行内选择器使用固定宽度，不允许占满整行挤压左侧文案。USB 接口和 USB 存储策略行复用公共 `AsyncSelectRow`，但行背景由 `InterfaceControlTab` 外层接口分组卡片统一绘制，`AsyncSelectRow` 不再自行绘制 `BG_CARD`，避免深浅色主题切换时父子背景更新节奏不一致导致短暂黑色闪烁。

  * `ConnectionDetailDialog`: 连接详情弹窗。**入参/事件**: 接收选中记录和同设备历史；抛出 `onClose`。
  * `PolicyList`: 黑白名单列表 UI。**入参/事件**: 接收 `PeripheralPolicyRecord[]`、策略选项与 updating 查询函数；抛出 `onPolicyChange`。
  * 黑白名单页头部动作区：导出名单按钮保持主操作样式；还原策略按钮使用圆环还原图标和红色风险提示样式，确认弹窗按钮文案为“还原”。还原按钮直接绑定 `PeripheralPolicyViewModel.state.hasRestorableRecords`，该字段由 `reloadRecords()` 根据真实 USB 策略状态库计算；字段为 false 时还原策略按钮置灰且不可点击。

界面展示设备时的策略判定规则如下：

- `DeviceRecordList` 展示 `PeripheralConnectionRecord[]`，包含设备名称、设备类型、连接事件、策略、连接时间和详情入口。策略列由 `getPeripheralConnectionListPolicyLabel(record)` 生成：若 `record.effectivePolicyLabel` 有值，优先展示该值；若命中 `device_deny` 或 `policyHit === 'deny'`，展示禁用；否则展示启用。列表不重新计算当前策略。
- `ConnectionDetailDialog` 展示选中记录的关键信息和同设备历史连接记录。同设备历史由 `buildConnectionHistoryItems(records, selectedRecord)` 生成；“策略结论 / 执行策略 / 命中类型 / 结果状态”由 `PeripheralRecordPresentationMapper` 读取 trace 记录语义生成，不重新回查当前策略仓库。
- `PolicyList` 展示 `PeripheralPolicyRecord[]`，当前状态由 `policyIndex` 决定，索引取值来自 `PeripheralPolicyOptionIndexes` 并与 `PeripheralOptionStrings.POLICY_OPTIONS` 保持同序。该列表来自 USB 策略状态库，展示的是基于 fingerprintKey 合并后的业务记录，不直接展示原始事件 payload 中的设备标识。
- `PolicyList` 无候选设备时只展示常规字重的主空态文案，不再附加“重新插拔设备或检查账号权限”类兜底副标题。
- 用户在 `PolicyList` 中选择 allow / deny 后触发 `onPolicyChange`，由页面和 ViewModel 进入单设备黑白名单更新流；系统策略下发成功后再从 USB 策略状态库回读列表和头部按钮状态，不提前改写 `PeripheralPolicyRecord[]`。清空连接记录只清理连接记录 trace，在线和离线 USB 策略状态均保留；还原策略保留 USB 状态库卡片，将已有记录的 desired_policy 恢复为 allow，active_policy=deny 的记录在系统 allow 成功后置为 none。若当前黑白名单全部为允许接入或没有记录，头部还原策略按钮不可点击。

## 5. 异常处理与系统依赖 (Dependencies & Errors)
- **关键系统 API**:
  - `@ohos.enterprise.restrictions`: 接口总控策略设置与读取。
  - `@kit.MDMKit.usbManager`: USB 存储策略和 USB 设备禁用策略下发。
  - `@kit.MDMKit.wifiManager`: Wi-Fi 启用恢复，使用非强制开启模式。
  - `@kit.MDMKit.bluetoothManager`: 蓝牙启用恢复。
  - `@kit.BasicServicesKit.commonEventManager`: USB 插拔公共事件订阅。
  - `@kit.ConnectivityKit.connection`: API 26 蓝牙 ACL 状态回调和真实远端设备名称查询。
  - `@kit.AbilityKit.abilityAccessCtrl`: 蓝牙运行时权限检查与申请。
- **系统权限**:
  - `ohos.permission.ENTERPRISE_MANAGE_USB`
  - `ohos.permission.ENTERPRISE_MANAGE_RESTRICTIONS`
  - `ohos.permission.ENTERPRISE_MANAGE_NETWORK`
  - `ohos.permission.ENTERPRISE_MANAGE_WIFI`
  - `ohos.permission.ENTERPRISE_MANAGE_BLUETOOTH`
  - `ohos.permission.ENTERPRISE_ADMIN_MANAGE`
  - `ohos.permission.ENTERPRISE_SUBSCRIBE_MANAGED_EVENT`
  - `ohos.permission.ACCESS_BLUETOOTH`
  - `ohos.permission.GET_BLUETOOTH_PEERS_MAC`
- **异常兜底策略**:
  - 企业管理员未激活时，`PeripheralViewModel.ensureAdminReady()` 返回 `PeripheralReasonCodes.ADMIN_REQUIRED`，页面统一提示先激活企业管理员。
  - USB 接口默认管控策略写入失败时返回接口切换失败提示；USB 接口开关不再映射 restrictions 全局禁用冲突。USB 存储策略的系统冲突仍映射为 USB 冲突类 reasonCode，并通过 `resolvePeripheralReason(...)` 显示提示。
  - Wi-Fi / 蓝牙启用恢复失败时，用户主动切换启用返回接口切换失败且不更新本地开关状态；页面初始化时恢复失败只记录日志，不阻断接口状态读取和页面展示，页面仍展示策略读取结果，不额外声明硬件已开启。Wi-Fi 开启使用 `isForce=false`，避免把“启用接口管控策略”扩大为强制用户不可关闭 Wi-Fi；蓝牙恢复仅使用 `turnOnBluetooth`，不恢复蓝牙单设备 MDM 黑白名单策略。
  - 单设备策略下发失败时，不更新 USB 状态库 active_policy 为目标状态；页面根据 dispatch 返回的冲突或失败原因提示用户。USB_STORAGE + USB 存储 `DISABLED` 时，allow / deny 两个方向都返回 USB 存储账户级冲突，不更新 active_policy。蓝牙设备不进入单设备策略下发链路。
  - USB 首次连接自动拉黑下发失败时，不更新 active_policy=deny，连接记录继续落库并在详情中追加失败原因；已有 allow 或 deny 逻辑策略时运行时自动拉黑跳过，不覆盖管理员决策；已有 deny 命中在 connect 阶段确保系统 deny 下发。USB Hub 拓扑节点（`baseClass = 0x09`）在策略判定前过滤，不进入失败提示、连接记录或本地策略保存链路。USB_STORAGE + USB 存储 `DISABLED` 场景同样不下发设备级 deny，避免账户级禁止访问和设备级 deny 叠加造成恢复链路阻塞。蓝牙连接记录始终按允许接入记录。
  - 蓝牙运行时权限被拒绝、真实 MAC 系统权限未生效或 API 26 ACL 回调注册失败时，`bluetoothRuntimeEvents=false`；本次 runtime 启动不回退旧公共事件，后续由 runtime 重新启动再尝试注册。蓝牙关闭过程中不得注销既有 ACL 回调，以免丢失策略禁用触发的断开事件。
  - 蓝牙禁用策略快照无 ACL 当前连接缓存或快照写入失败时，不回滚已成功下发的蓝牙禁用策略；策略快照不抑制 ACL 断开记录，不使用设备名、系统 profile 枚举或历史记录模糊匹配来决定是否丢弃断开事件。
  - trace RDB 未初始化时，pipeline 在进入 consumer 前跳过当前运行时事件并记录日志，避免状态更新或策略下发先发生但 trace 无法落库；trace 写入失败时记录日志，不阻塞页面基本展示。
  - 导出和清理失败时，页面通过统一失败弹窗提示用户重试；CSV 导出写文件失败必须返回失败结果，不得留下部分成功状态。

  - Wi-Fi / 蓝牙在策略状态为启用时，页面初始化和用户主动切换启用都必须触发一次硬件开启恢复；Wi-Fi 恢复不得使用强制开启，蓝牙恢复必须使用 `ENTERPRISE_MANAGE_BLUETOOTH` 支持的企业开启接口。
  - USB Hub 拓扑节点（`baseClass = 0x09`）不生成连接记录，不进入黑白名单候选，不触发首次自动拉黑，也不尝试单设备策略下发。
  - 非 Hub USB 首次连接在状态库无该 fingerprint 且 `usb_default_policy = deny` 时自动下发 deny 并写入黑名单；`usb_default_policy = allow` 时未知 USB 首次连接写入 allow 状态但不自动下发 deny；下发失败不更新 active_policy=deny，已有 allow 或 deny 不被自动覆盖，已有 deny 命中在 connect 阶段确保 deny 下发。蓝牙连接记录保留，但策略结论为允许接入，不进入黑白名单；API 26 ACL 回调只接收并映射连接、断开最终态。
  - USB 黑白名单策略状态必须按 SN 或 `VID + PID + description` 弱指纹维护；唯一标识解析逻辑不得耦合 RDB 仓储。黑名单 USB 插入后必须按逻辑策略下发 deny，拔出后在无其它同 baseClass 在线 deny 设备时内部恢复 allow；该内部恢复不得补连接记录或 `policy_snapshot`，UI 仍按 desiredPolicy 显示禁止。离线 USB 设备在黑白名单页不得修改策略；清空连接记录只清理 trace，不得删除在线或离线 USB 策略状态；还原策略不得删除黑白名单卡片，只能把已有 USB 状态恢复为默认允许接入。全部黑白名单均为允许接入时，还原策略按钮必须置灰不可点击。
  - USB 默认策略、USB 存储策略或单设备策略变更成功后，只对 `usbManager.getDevices()` 当前枚举到且非 Hub、具备合法 VID/PID 和有效 class 的 USB 设备补写 `policy_snapshot`；单设备策略必须精确匹配 VID/PID 才补写；策略快照 action 必须为 `discover`，不得写成 `connect` 或 `disconnect`；USB 存储策略快照只针对 USB_STORAGE，并通过 decision、result、usbPolicy、matchedPolicyKind 体现 `DISABLED / READ_ONLY / READ_WRITE`；枚举不到设备、未匹配当前设备或无有效 class 时不得生成设备快照。USB 运行时记录和策略快照的设备名必须共用统一解析规则：`manufacturerName + productName` 优先，其次 `deviceName`，最后兜底通用 USB 名称；不得使用 `name` 兜底。
  - 蓝牙全局禁用策略变更成功后，只对本进程 ACL `CONNECTED` 当前连接缓存中的真实 MAC 设备补写 `policy_snapshot`；补写记录必须为 `source = policy_snapshot`、`action = discover`、`decision = deny`、`result = BLOCKED`、`rawEventName = bluetooth_global_policy`、`matchedPolicyKind = bluetooth_disabled`，页面展示为“策略变更”；无 ACL 当前连接缓存时不得生成设备快照；随后同一真实 MAC 的 ACL `DISCONNECTED` 不得被策略快照抑制，必须保留为“断开”。
  - 蓝牙 ACL 采集必须通过 API 26 `connection.onAclStateChange()` 和 `GET_BLUETOOTH_PEERS_MAC` 使用真实 MAC；不得继续使用普通 ACL common event 的虚拟地址，不得订阅私有 REALMAC action，不得读取 `linktype` 或 `isRetention`，不得增加时间窗口去重，也不得在新接口失败时回退旧链路。
  - 同一真实 MAC 后续蓝牙记录解析到有效设备名时，必须刷新本进程 MAC 名称缓存，并刷新历史记录中仍显示“未命名蓝牙设备”的同 MAC 蓝牙记录；已有有效名称不得被覆盖。断开事件无法从系统读取远端名称时，必须优先使用同一真实 MAC 名称缓存，避免策略禁用或真实断开后设备名退化为“未命名蓝牙设备”。
  - USB_STORAGE + USB 存储 `DISABLED` 时，连接记录优先展示设备级禁止接入语义，allow / deny 两个方向均不下发单设备策略。
  - 运行时采集失败、蓝牙权限拒绝、API 26 ACL 回调注册失败或 RDB 写入失败时不阻断页面基本展示；状态上报必须反映真实可采集能力，导出和清理失败需保留可重试提示。
  - 外设连接记录和黑白名单导出同秒同名覆盖时，不得残留旧文件尾部内容。
  - 设备连接记录列表必须优先保证设备名称和设备类型可读：设备名列宽高于策略、连接时间等辅助列，设备名最多允许两行显示；设备类型列不得被压缩到标签文字截断，连接事件、策略、连接时间和详情操作列保持紧凑但不得出现文字重叠或按钮内容溢出。
