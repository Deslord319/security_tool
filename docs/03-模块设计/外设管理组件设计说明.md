## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 提供企业管理员场景下的外设接口总控、设备连接记录采集与单设备黑白名单策略管理能力。
- **入口路由 (Route Entry)**: `RouteIds.PERIPHERAL_MANAGE = 'peripheral-manage'`
- **对外暴露能力 (Public APIs)**:
  - `ApplicationRuntimeManager.ensureRuntimeServices()`
  - `PeripheralService`
  - `PeripheralConnectionRecordService`
  - `PeripheralDevicePolicyRepository`
- **业务边界**:
  - ✅ **包含**: USB、蓝牙、Wi-Fi、HDC、网络共享、打印机、麦克风、摄像头的接口管控；USB 存储访问策略；USB 插拔与蓝牙 ACL 连接记录采集、详情、导出、清理；基于候选设备的 allow / deny 黑白名单策略；USB / 蓝牙设备首次连接时自动下发 deny 并加入黑名单。
  - ❌ **不包含**: 企业管理员激活流程本身；防火墙、日志管理、身份鉴别、工具设置等非外设模块能力；未在当前代码中接入的其它外设类型运行时采集。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `PeripheralViewModel.adminState: AdminActivationState`: 企业管理员激活状态及启用命令。
  - `PeripheralViewModel.reasonCode: string | null`: 最近一次外设业务失败原因码。
  - `PeripheralViewModel.currentTab: number`: 当前页签索引，`0` 为接口管控，`1` 为设备连接记录，`2` 为黑白名单。
  - `InterfaceControlViewModel.state: InterfaceControlStateModel`: USB、蓝牙、Wi-Fi、HDC、网络共享、打印机、麦克风、摄像头禁用状态，以及 USB 存储策略索引。
  - `InterfaceControlViewModel.processingKey: string`: 当前正在处理的接口或 USB 存储策略 key。
  - `PeripheralRecordViewModel.state.records: PeripheralConnectionRecord[]`: 连接记录列表。
  - `PeripheralRecordViewModel.state.connectionDetailDialog: ConnectionDetailDialogStateModel`: 连接详情弹窗选中记录和同设备历史。
  - `PeripheralPolicyViewModel.state.records: PeripheralPolicyRecord[]`: 黑白名单页展示列表。
  - `PeripheralPolicyViewModel.state.updatingKeys: string[]`: 正在更新策略的设备标识集合。
- **关键流转路径**:
  - `进入外设页面` -> `MainPage` 传入共享 `PeripheralViewModel` -> `PeripheralPage.initViewModel()` -> `PeripheralViewModel.initialize(context)` -> `PeripheralTraceMaintenanceService.initialize(context)` + `PeripheralDevicePolicyRepository.init(context)` -> `reloadAllData()` -> `三个 Tab 同步展示`
  - `页面显示` -> `MainPage.handlePageActivated()` -> `ApplicationRuntimeManager.ensureRuntimeServices()` -> `RuntimeProducerAdapter + Pipeline + Consumers` -> `PeripheralTraceMaintenanceService.appendEntries()` -> `PeripheralTraceEntryRepository`
  - `切换接口或 USB 存储策略` -> `InterfaceControlTab` -> `PeripheralPage` -> `PeripheralViewModel.ensureAdminReady()` -> `InterfaceControlViewModel` -> `PeripheralService` -> `系统 MDM/restrictions API` -> `state 或 reasonCode 更新`
  - `USB 运行时事件进入` -> `PeripheralConnectionRecordRuntimeProducerAdapter` -> `PeripheralConnectionRecordPipelineImpl` -> `PeripheralConnectionRecordUsbConsumer` -> `按接口总控 / USB 存储策略 / 单设备策略 / 默认允许顺序先生成基础判定` -> `非 USB_STORAGE 禁止访问场景下 connect 事件首次发现设备时自动 dispatch deny` -> `成功后写入 PeripheralDevicePolicyRepository` -> `自动拉黑结果叠加到 trace 语义` -> `trace 落库`
  - `蓝牙 ACL 事件进入` -> `PeripheralConnectionRecordRuntimeProducerAdapter` -> `PeripheralConnectionRecordPipelineImpl` -> `PeripheralConnectionRecordBluetoothAclConsumer` -> `connect 事件首次发现设备时自动 dispatch deny` -> `成功后写入 PeripheralDevicePolicyRepository` -> `结合单设备策略生成连接记录` -> `trace 落库`
  - `黑白名单列表生成流` -> `连接记录变更 / 策略仓库变更` -> `PeripheralViewModel listener` -> `PeripheralPolicyViewModel.reloadRecords(...)` -> `PeripheralPolicyService.buildPolicyRecords(...)` -> `规范化 deviceId` -> `叠加 PeripheralDevicePolicyRepository 当前策略` -> `PeripheralPolicyRecord[]` -> `PolicyList 展示`
  - `单设备黑白名单更新流` -> `PolicyList.onPolicyChange` -> `PeripheralPage` -> `PeripheralViewModel.handlePeripheralPolicyChange()` -> `PeripheralPolicyViewModel.setDevicePolicy()` -> `PeripheralDevicePolicyDispatchService.dispatch()` -> `PeripheralDevicePolicyRepository.set()` -> `PeripheralPolicyViewModel.reloadRecords(...)` -> `PolicyList 刷新`

黑白名单页不是独立设备资产库。候选设备来自连接记录，同一设备通过规范化后的 deviceId 合并。运行时首次发现 USB / 蓝牙设备连接时，若本地策略仓库不存在该 deviceId 的 allow / deny 策略，则自动同步下发 deny；系统下发成功后再写入本地 deny 策略。已有 allow 或 deny 策略不被运行时自动覆盖。USB_STORAGE 设备在 USB 存储总策略为 `DISABLED` 时属于账户级禁止访问场景，USB Consumer 不执行首次自动拉黑，不下发设备级 deny，也不写入本地 deny；连接记录仍按 USB 存储已禁用语义落库。黑白名单候选设备在生成列表前会先规范化 deviceId：USB 设备优先解析 VID/PID 并格式化为统一大写格式；蓝牙设备识别 MAC 地址并格式化为统一 MAC 表示。规范化后的 deviceId 是策略仓库和列表合并的业务 key，连接记录中的原始 deviceId 不直接作为策略 key 使用。

## 3. 模块结构与组件设计 (Module Components)

### 【核心层】(Core MVVM & Domain Layers)

#### 3.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `PeripheralType`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 定义外设类型枚举。
    * **关键字段**: `USB_STORAGE / USB_CAMERA / USB_PRINTER / USB_HID / USB_OTHER / BLUETOOTH` (当前支持的外设类型)
  * `UsbStoragePolicy`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 表示 USB 存储设备访问策略。
    * **关键字段**: `READ_WRITE` (读写), `READ_ONLY` (只读), `DISABLED` (禁止访问)
  * `PeripheralConnectionRecord`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 页面层连接记录展示模型。
    * **关键字段**: `deviceId` (设备标识), `action` (connect / disconnect), `policyHit` (allow / deny / unknown), `effectivePolicyLabel` (列表策略展示), `matchedPolicyKind` (命中类型)
  * `PeripheralTraceRecord`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: RDB 持久化的外设运行记录模型。
    * **关键字段**: `recordId` (唯一记录键), `occurAt` (事件发生时间), `source` (runtime_event / audit_event / policy_snapshot), `rawPayload` (原始事件快照), `usbPolicy` (USB 存储策略快照)
  * `PeripheralTraceQuery`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 外设 trace 仓库查询条件。
    * **关键字段**: `startTime/endTime`, `deviceId`, `keyword`, `deviceTypes/actions/decisions/results/sources`；分页由调用方通过仓库方法参数或上层 ViewModel 控制，不在 Query 模型中定义未消费字段。
  * `PeripheralTraceStats`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 安全总览卡片使用的外设 trace 汇总。
    * **关键字段**: `total`, `connects`, `disconnects`；统计实现只按 `action` 分组，不保留页面未展示的来源/决策分组字段和对应查询参数。
  * `PeripheralDevicePolicyRecord`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 单设备策略持久化模型。
    * **关键字段**: `deviceId` (规范化设备标识), `policy` (allow / deny), `updatedAt` (最后更新时间)
  * `PeripheralPolicyRecord`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 黑白名单页组合展示模型。
    * **关键字段**: `deviceId` (候选设备键), `lastSeenAt` (最近连接时间), `policyIndex` (0 允许 / 1 禁止)

#### 3.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/peripheral/**`
* **核心业务用例 (Use Cases)**:
  * `PeripheralService.setInterfaceDisabledWithResult(feature, disallow)`: 下发接口启用 / 禁用策略。
    * **副作用**: 调用 restrictions、networkManager 或 usbManager 系统 API。
    * **失败策略**: 返回 `OperationResult`，保留错误码、管理员缺失标志和冲突 stage。
  * `PeripheralService.setUsbStoragePolicyWithCleanupResult(policy)`: 设置 USB 存储访问策略。
    * **副作用**: 调用 `usbManager.setUsbStorageDeviceAccessPolicy(...)`。
    * **失败策略**: 系统返回 `9200010` 时映射为 USB 冲突类失败。
  * `PeripheralConnectionRecordService.queryRecords(filter?)`: 查询连接记录。
    * **副作用**: 读取 `PeripheralTraceEntryRepository`。
    * **失败策略**: 记录错误并返回空数组。
  * `PeripheralTraceMaintenanceService.initialize(context)`: 初始化 trace 仓库并对齐 retention / trim。
    * **副作用**: 初始化 RDB、删除过期记录、裁剪最大条数。
    * **失败策略**: 返回 `storage_init_failed`。
  * `PeripheralTraceMaintenanceService.appendEntries(records)`: 标准化并写入 trace。
    * **副作用**: `upsertBatch` 写入 RDB，并触发变更监听。
    * **失败策略**: 仓库未初始化返回 `not_initialized`，写入失败返回 `save_failed`。
  * `PeripheralConnectionRecordRuntimeProducerAdapter.start()`: 启动外设运行时事件订阅。
    * **副作用**: 订阅 USB attach / detach common event，监听蓝牙状态变化，并在蓝牙可用时订阅蓝牙 ACL 状态事件。
    * **失败策略**: 无 context 或订阅失败时记录降级状态；蓝牙权限未授权、ACL 订阅失败或 ACL 订阅未就绪时，`bluetoothRuntimeEvents` 不得标记为可用，并进入可恢复阻塞态。
  * `PeripheralConnectionRecordPipelineImpl.push(event)`: 分发运行时事件并写入 trace。
    * **副作用**: 按 channel 分发给 USB / Bluetooth Consumer，对 USB 事件执行短时间去重，并调用 `PeripheralTraceMaintenanceService.appendEntries(...)` 落库。
    * **失败策略**: trace 仓库未就绪时跳过当前事件；Consumer 无法处理或写入失败时记录日志并返回未持久化结果。
  * `PeripheralConnectionRecordUsbConsumer.consume(event)`: 将 USB common event 转换为外设 trace。
    * **副作用**: 解析 payload 中的 VID/PID、baseClass、设备名称和时间；读取 USB 接口状态、USB 存储策略与单设备策略快照并生成基础判定；对首次 USB connect 且本地无策略的设备调用 `PeripheralDevicePolicyDispatchService.dispatch(... deny ...)`，成功后写入 `PeripheralDevicePolicyRepository`；USB_STORAGE 且 USB 存储策略为 `DISABLED` 时跳过该自动拉黑动作；最终生成 `PeripheralTraceRecord`。
    * **失败策略**: 不支持的 USB 事件或无法解析的事件返回 `null`。
  * `PeripheralConnectionRecordBluetoothAclConsumer.consume(event)`: 将蓝牙 ACL common event 转换为外设 trace。
    * **副作用**: 解析蓝牙设备 ID、设备名称、连接/断开状态；对首次蓝牙 connect 且本地无策略的设备先调用 `PeripheralDevicePolicyDispatchService.dispatch(... deny ...)`，成功后写入 `PeripheralDevicePolicyRepository`；读取本地单设备策略，并生成 `PeripheralTraceRecord`。
    * **失败策略**: 设备标识缺失或无法识别连接动作时返回 `null`。
  * `PeripheralDevicePolicyDispatchService.dispatch(request)`: 将黑白名单策略下发到系统；蓝牙按设备 MAC 下发，USB 按设备类型映射为 MDM USB baseClass 下发。
    * **副作用**: 调用 `usbManager` 或 `bluetoothManager` 的 disallow API；USB 下发粒度为 `PeripheralType` 对应的 `UsbDeviceType`，不是 VID/PID 单设备。
    * **失败策略**: 检查全局接口禁用、USB 存储冲突和系统下发冲突，失败时返回 `PeripheralPolicyDispatchReason`。
  * `PeripheralDevicePolicyRepository.set(record)`: 保存单设备策略。
    * **副作用**: 写入 preferences，并触发 change listener。
    * **失败策略**: 记录错误并返回 `false`。
  * `PeripheralPolicyService.buildPolicyRecords(connectionRecords, devicePolicies)`: 生成黑白名单页展示列表。
    * **副作用**: 无。
    * **失败策略**: 跳过无法规范化的设备标识。
* **内部架构设计**:
  * **Repository**: `entry/src/main/ets/services/peripheral/connection-record/PeripheralTraceEntryRepository.ets` - 外设运行记录的 RDB 真源，表名为 `peripheral_trace_entries`。
  * **Repository**: `entry/src/main/ets/services/peripheral/device-policy/PeripheralDevicePolicyRepository.ets` - 单设备策略的 Preferences 真源。
  * **Mapper/Ports**: `entry/src/main/ets/services/peripheral/connection-record/peripheral_connection_record_payload_utils.ets` - 运行时 common event payload 提取工具。
  * **Mapper/Ports**: `entry/src/main/ets/viewmodels/peripheral/connection-record/PeripheralRecordPresentationMapper.ets` - 连接记录展示语义转换。

设备连接记录的详细流程如下：

- `PeripheralConnectionRecordRuntimeProducerAdapter` 负责系统事件入口。它订阅 USB attach / detach common event，监听蓝牙 stateChange，并在蓝牙处于可用状态时订阅蓝牙 ACL 状态事件；系统事件进入后统一转换为内部 `PeripheralConnectionRecordProducerEvent`。`bluetoothRuntimeEvents` 只表示蓝牙状态监听可持续工作且当前不处于 ACL 权限/订阅阻塞；蓝牙已开启但 ACL 订阅失败时必须降级为不可用。
- `PeripheralConnectionRecordProducerEvent.rawData` 明确定义为可选 `commonEventManager.CommonEventData`；只有系统 common event 原始数据存在时 Consumer 才解析，缺失时返回 `null`，不再把原始数据建模为泛化 `Object`。
- `PeripheralConnectionRecordPipelineImpl` 负责事件接力。它绑定 context、挂载 Producer、连接 Consumers，根据 `channel` 分发到 USB 或蓝牙 Consumer；USB 事件会按 `deviceId + action + rawEventName` 做短时间去重。
- `PeripheralConnectionRecordUsbConsumer` 负责 USB 事件业务判定。它从 payload 中提取 VID/PID、baseClass、设备名称和时间，推断 `PeripheralType`，并按接口总控、USB 存储禁止访问、本地设备 deny、USB 存储只读、默认允许的顺序生成基础策略命中结果。
- `PeripheralConnectionRecordUsbConsumer` 在 USB connect 事件中基于基础策略判定执行首次连接自动拉黑：若本次基础命中不是 `usb_storage_disabled`，且 `PeripheralDevicePolicyRepository.get(info.deviceId)` 为空，则调用 `PeripheralDevicePolicyDispatchService.dispatch({ policy: 'deny' })`；下发成功后写入本地 deny；下发失败不写入本地策略，并在本次 trace 详情中保留失败原因。
- `PeripheralConnectionRecordUsbConsumer` 对 USB_STORAGE 且 USB 存储总策略为 `DISABLED` 的连接事件不执行首次自动拉黑。该场景代表账户级 USB 存储禁止访问已生效，记录仍按 `usb_storage_disabled` 展示，不额外制造设备级 deny，避免后续恢复 USB 存储总策略或清空黑白名单时被设备级策略叠加阻塞。
- `PeripheralConnectionRecordBluetoothAclConsumer` 负责蓝牙 ACL 事件业务判定。它解析蓝牙设备 ID、设备名称和连接/断开状态，规范化蓝牙设备 ID，并结合本地单设备策略生成记录语义。
- `PeripheralConnectionRecordBluetoothAclConsumer` 在蓝牙 connect 事件中先执行首次连接自动拉黑：若 `PeripheralDevicePolicyRepository.get(eventInfo.deviceId)` 为空，则调用 `PeripheralDevicePolicyDispatchService.dispatch({ policy: 'deny' })`；下发成功后写入本地 deny；下发失败不写入本地策略，并在本次 trace 详情中保留失败原因。
- `PeripheralTraceMaintenanceService` 负责落库前标准化。它补齐缺省字段、规范化 action / decision / source / result，并写入 `PeripheralTraceEntryRepository`，同时触发仓库变更通知。
- `PeripheralConnectionRecordService` 负责页面查询。它从 `PeripheralTraceEntryRepository` 查询 runtime / audit 记录，并映射为 `PeripheralConnectionRecord[]`。

USB 冲突与单设备策略冲突是当前领域设计的关键规则：

- USB 接口总控与 USB 存储策略存在系统冲突风险。`PeripheralService.setUsbInterfaceDisabledWithCleanup(...)` 在设置 USB 接口前，会先读取 USB 存储策略；若当前策略不是 `READ_WRITE`，会先恢复到 `READ_WRITE` 再继续设置接口状态。
- `PeripheralService.setUsbStoragePolicyWithCleanupResult(...)` 设置 USB 存储策略时同样先读取当前 USB 策略快照，系统返回 `9200010` 时映射为 USB 冲突失败，而不是简单未知错误。
- 黑白名单策略不是“先保存本地，再尝试系统下发”。当前实现是 `PeripheralPolicyViewModel` 先调用 `PeripheralDevicePolicyDispatchService.dispatch(...)`，只有系统下发成功后才写入 `PeripheralDevicePolicyRepository`。
- 首次连接自动拉黑同样遵循“先系统下发，再保存本地”。USB / 蓝牙 Consumer 只在 connect 事件且本地无既有策略时触发自动 deny；已有 allow 或 deny 均不覆盖，disconnect 事件不触发。USB_STORAGE 在 USB 存储总策略 `DISABLED` 下跳过首次自动拉黑，保留账户级禁用的连接记录语义。
- USB 黑白名单在本地按规范化 deviceId 管理和回显，但系统侧下发粒度是设备类型：`PeripheralDevicePolicyDispatchService.toUsbDeviceType(...)` 将 `PeripheralType` 映射为 MDM `UsbDeviceType.baseClass` 后调用 `usbManager`。当前代码没有按 VID/PID 对单个 USB 物理设备下发系统策略。因此 USB 首次连接自动拉黑可能影响同类 USB 设备，该误伤风险在当前规格中可接受。
- USB 策略下发前会检查 USB 全局禁用状态；USB 存储设备设置 allow 或 deny 前还会检查 USB 存储账户级策略是否为 `DISABLED`。当 USB 存储总策略为 `DISABLED` 时，黑白名单仍按本地单设备策略展示，但 USB_STORAGE 单设备策略不允许继续下发 allow 或 deny，避免账户级禁止访问和设备级策略叠加。
- 蓝牙单设备策略下发前会检查蓝牙全局禁用状态。任一冲突成立时，本地策略不保存，页面收到对应 reasonCode。

黑白名单列表生成与策略下发的详细流程如下：

- `PeripheralPolicyService.buildPolicyRecords(...)` 从连接记录中提取候选设备，同一设备取最近一次连接记录。
- `PeripheralPolicyService.normalizeDeviceIdForPolicy(...)` 负责黑白名单链路的设备 ID 规范化。USB 设备优先解析 VID/PID 并格式化为统一大写格式；蓝牙设备识别 MAC 地址并格式化为统一 MAC 表示；无法规范化为空标识的设备不会进入策略候选列表。
- 规范化结果是黑白名单策略的本地业务 key，决定同设备合并、策略仓库查询、策略保存和列表回显；其中 USB 的系统下发仍按设备类型/baseClass 执行。
- `PeripheralPolicyService.buildPolicyRecords(...)` 会叠加 `PeripheralDevicePolicyRepository.getAll()` 中的 allow / deny 状态，未配置策略时默认允许，最终输出 `PeripheralPolicyRecord[]`。
- `PeripheralDevicePolicyDispatchService.dispatch(...)` 负责真实系统策略下发。蓝牙按规范化 MAC 下发；USB 按 `PeripheralType` 映射的 `UsbDeviceType.baseClass` 下发。下发成功后才允许 `PeripheralDevicePolicyRepository.set(...)` 保存本地策略；下发失败时本地策略不保存。
- 首次连接自动拉黑成功后，连接记录应展示为 deny / blocked，并保留“首次接入已自动加入黑名单”类详情。自动拉黑下发失败时，连接记录仍应落库，保留原策略判定结果，并在详情中追加失败原因，避免设备接入事实因策略下发失败而丢失。USB_STORAGE + USB 存储 `DISABLED` 场景不展示自动拉黑成功或失败语义，只展示 USB 存储已禁用。
- USB_STORAGE + USB 存储 `READ_ONLY` 场景下若本地已存在设备级 deny，连接事件应优先展示 `device_deny` 和已禁用，不得被 `usb_storage_read_only` 提前返回覆盖；断开事件和黑白名单同样以本地 deny 展示已禁用 / 禁止接入。

#### 3.3 ViewModel (视图模型层)
* **真实文件路径**: `entry/src/main/ets/viewmodels/peripheral/**`
* **状态分发逻辑**:
  * `PeripheralViewModel.initialize(context)`: 初始化 trace 维护服务和设备策略仓库，加载接口状态、连接记录与黑白名单，并注册 trace / policy listener。
  * `PeripheralViewModel.reloadAllData()`: 刷新管理员状态、接口状态、连接记录和黑白名单数据。
  * `PeripheralViewModel.toggleInterface(feature, disallow)`: 校验企业管理员状态后委托 `InterfaceControlViewModel` 下发接口策略。
  * `PeripheralViewModel.setUsbStoragePolicy(index)`: 校验企业管理员状态后下发 USB 存储策略，并在成功后刷新展示状态。
  * `PeripheralViewModel.handlePeripheralPolicyChange(...)`: 先标记 updating，再触发 `PeripheralPolicyViewModel.setDevicePolicy(...)`，成功后刷新记录和策略列表，最后清理 updating 状态。
  * `PeripheralViewModel.clearConnectionHistory()`: 通过 `PeripheralConnectionRecordClearUsecase` 清空 trace，再刷新连接记录和黑白名单候选集。
  * `PeripheralViewModel.clearPeripheralPolicyRecords()`: 清空单设备策略仓库并重建黑白名单列表。
  * `InterfaceControlViewModel.reloadState()`: 从 `PeripheralService` 读取接口状态和 USB 存储策略。
  * `PeripheralRecordViewModel.reloadRecords()`: 从 `PeripheralConnectionRecordService` 加载连接记录，并维护详情弹窗历史。
  * `PeripheralPolicyViewModel.reloadRecords(connectionRecords)`: 基于连接记录与策略仓库重建黑白名单页展示列表。

界面展示设备时的策略判定由 ViewModel 组合链路保证：

- 连接记录页展示的是 trace 中已经落库的策略结果。`PeripheralRecordViewModel` 不会重新计算当前策略，而是直接展示 `PeripheralConnectionRecord` 中的 `effectivePolicyLabel`、`policyHit` 和 `matchedPolicyKind`。
- trace 变化后，`PeripheralViewModel.refreshTraceDrivenData()` 会先调用 `PeripheralRecordViewModel.reloadRecords()` 刷新连接记录，再调用 `PeripheralPolicyViewModel.reloadRecords(records.state.records)` 重建黑白名单候选集。因此连接记录变化会同时影响设备连接记录页和黑白名单页。
- 策略仓库变化后，`PeripheralViewModel.refreshPolicyDrivenData()` 只需要基于当前连接记录重建黑白名单列表，不需要重新采集运行时事件。
- `PeripheralRecordViewModel.openConnectionDetailDialog(record)` 基于当前 `records` 构建同设备历史；`exportConnectionHistory()` 将当前连接记录导出为 CSV。
- 黑白名单页展示的是“候选设备当前策略状态”。`PeripheralPolicyViewModel` 通过 `PeripheralPolicyService.buildPolicyRecords(...)` 将连接记录中的候选设备与 `PeripheralDevicePolicyRepository` 当前存量策略进行叠加。运行时首次连接自动拉黑成功的设备会以本地 deny 策略进入列表；仍未配置策略的历史候选设备默认视为允许。
- `PeripheralPolicyViewModel.setDevicePolicy(...)` 负责系统下发与本地保存；`clearAllPolicies()` 清空策略时会逐条尝试恢复系统策略，失败时尝试回滚已恢复的策略。

#### 3.4 View / Page (页面视图层)
* **真实文件路径**: `entry/src/main/ets/views/peripheral/overview/PeripheralPage.ets`
* **结构说明**:
  * `PeripheralPage`: 外设管理页组合层。接收 `MainPage` 持有的共享 `PeripheralViewModel`，负责页面初始化、Tab 切换、USB loading dialog、导出提示、清理确认弹窗和连接详情弹窗，并把用户操作转发给 `PeripheralViewModel`。
  * 页面包含三个 Tab：`接口管控`、`设备连接记录`、`黑白名单`。
  * 页面不负责启动运行时采集。运行时采集由 `MainPage` 通过 `ApplicationRuntimeManager.ensureRuntimeServices()` 启动。

#### 3.5 Components (可复用组件层)
* **真实文件路径**: `entry/src/main/ets/components/peripheral/**`
* **结构说明**:
  * `InterfaceControlTab`: 接口管控 UI。**入参/事件**: 接收各接口禁用状态、USB 存储策略索引、processingKey；抛出 `onInterfaceToggle`、`onUsbInterfaceToggleAsync`、`onUsbStoragePolicyChange`。
  * `DeviceRecordList`: 连接记录列表 UI。**入参/事件**: 接收 `PeripheralConnectionRecord[]`；抛出 `onRecordClick` 打开详情。
  * `ConnectionDetailDialog`: 连接详情弹窗。**入参/事件**: 接收选中记录和同设备历史；抛出 `onClose`。
  * `PolicyList`: 黑白名单列表 UI。**入参/事件**: 接收 `PeripheralPolicyRecord[]`、策略选项与 updating 查询函数；抛出 `onPolicyChange`。

界面展示设备时的策略判定规则如下：

- `DeviceRecordList` 展示 `PeripheralConnectionRecord[]`，包含设备名称、设备类型、连接事件、策略、连接时间和详情入口。策略列由 `getPeripheralConnectionListPolicyLabel(record)` 生成：若 `record.effectivePolicyLabel` 有值，优先展示该值；若命中 `device_deny` 或 `policyHit === 'deny'`，展示禁用；否则展示启用。列表不重新计算当前策略。
- `ConnectionDetailDialog` 展示选中记录的关键信息和同设备历史连接记录。同设备历史由 `buildConnectionHistoryItems(records, selectedRecord)` 生成；“策略结论 / 执行策略 / 命中类型 / 结果状态”由 `PeripheralRecordPresentationMapper` 读取 trace 记录语义生成，不重新回查当前策略仓库。
- `PolicyList` 展示 `PeripheralPolicyRecord[]`，当前状态由 `policyIndex` 决定，`0` 表示允许、`1` 表示禁止。该列表来自“连接记录候选设备 + 本地策略仓库”的组合结果，展示的是基于规范化 deviceId 合并后的业务记录，不直接展示原始事件 payload 中的设备标识。
- 用户在 `PolicyList` 中选择 allow / deny 后触发 `onPolicyChange`，由页面和 ViewModel 进入单设备黑白名单更新流。因此清空连接记录后候选设备会消失，而清空策略后设备仍可保留但恢复默认允许；后续若该设备再次 connect 且本地策略已被清空，会按首次发现规则再次自动下发 deny。

  - RDB 未初始化或写入失败时，pipeline 跳过当前事件写入并记录日志，不阻塞页面基本展示。
  - 导出和清理失败时，页面通过统一失败弹窗提示用户重试。

### 4.1 实施步骤与测试验收 (Implementation & Acceptance)

- **实施步骤**:
  - 涉及接口总控、USB 存储、单设备黑白名单、连接记录、运行时采集、自动拉黑、导出清理或权限变化时，先更新本文档的目标范围、数据模型、状态流、职责边界、异常兜底和验收口径。

  - 接口总控统一经 `InterfaceControlService.ets` 和 `InterfaceControlViewModel.ets`；单设备策略统一经设备策略仓储、dispatch 和 MDM adapter；页面不得直接下发 MDM 策略。
  - 运行时采集变化先更新 `ApplicationRuntimeManager.ets`、Producer、Pipeline、Trace Repository 和 retention/max entries 策略，再调整页面展示。
  - 新增或调整企业管理、蓝牙或网络权限时，同步 `entry/src/main/module.json5`、`hapsigner/UnsgnedDebugProfileTemplate.json` 并按签名流程重新生成 p7b。
