## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 采集设备安全相关日志，提供本地查询、过滤、分页、详情查看、清理、留存配置和 CSV 导出能力，为安全审计和演示验收提供可追溯证据。
- **入口路由 (Route Entry)**: `log-manage` -> `entry/src/main/ets/views/log-manage/overview/LogManagePage.ets`
- **对外暴露能力 (Public APIs)**:
  - 页面入口由主导航路由到 `LogManagePage`。
  - 业务层主要通过 `LogCollectorService`、`LogEntryRepository`、`LogStorageMaintenanceService`、`LogExportService` 对模块内部提供采集、存储、维护和导出能力。
  - 当前没有面向其他业务模块的公共 SDK 或跨模块写入接口。
- **业务边界**:
  - ✅ **包含**: 系统审计日志订阅、崩溃日志采集、应用前后台电源生命周期日志、本地 RDB 存储、日志查询过滤、分页展示、详情弹窗、日志清理、留存配置、CSV 导出。
  - ❌ **不包含**: 测试代码设计、测试用例说明、远端日志上报、云端审计平台、用户自定义审计规则、审计事件策略下发、系统审计框架自身能力实现。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `LogManageState.initialized: boolean`: 页面是否完成存储初始化、设置加载和首屏快照构建。
  - `LogManageState.loading: boolean`: 页面初始化或关键异步操作中的加载态。
  - `LogManageState.active: boolean`: 列表页是否处于活跃态；非活跃时自动刷新被跳过。
  - `LogManageState.autoRefreshEnabled: boolean`: 是否响应存储变更自动刷新。
  - `LogManageState.mode: 'list' | 'settings'`: 当前展示列表视图或存储设置视图。
  - `LogCollectorState`: 采集器运行态，包含 `running`、`auditReady`、`crashReady`、`pendingCount`、`degraded` 和 `degradedReason`。
  - `LogSummaryState`: 总数、过滤后数量、今日数量、拦截数量和当前页数量。
  - `LogFilterState`: 事件类型筛选、结果筛选和关键字。
  - `LogPaginationState`: 当前页、页大小、页码输入和总页数。
  - `LogListState.pagedEntries: LogEntry[]`: 当前页展示日志。
  - `LogDetailState`: 详情弹窗可见性和当前日志 `entryId`。
  - `LogStorageSettingsState`: 最大本地条数、留存天数、脏状态和保存态。
- **关键流转路径**:
  - 页面初始化 -> `LogManageViewModel.initialize()` -> 初始化 RDB、注册变更监听、加载设置、构建快照 -> 更新 `initialized / summary / list / pagination`。
  - 页面激活 -> `activate()` -> `active = true` -> 存储变更可触发 `refreshIfActive()`。
  - 切换到设置页 -> `setMode('settings')` -> 加载设置、关闭详情、暂停列表活跃态和自动刷新。
  - 修改筛选条件 -> `setEventTypeIndex()` / `setResultIndex()` / `setKeyword()` -> 重置到第一页 -> 重新按查询快照刷新。
  - 分页操作 -> `setPageSizeIndex()` / `jumpToInputPage()` / `goToPreviousPage()` / `goToNextPage()` -> 更新分页状态 -> 重建当前页列表。
  - 日志采集 -> `LogCollectorService` 从审计/崩溃/生命周期源接收 `LogEntry` -> `LogStorageMaintenanceService.appendEntries()` -> `LogEntryRepository.upsertEntries()` -> 通知 ViewModel 刷新。
  - 清理日志 -> `clearLogs()` -> `LogStorageMaintenanceService.clearAll()` -> 关闭详情并重建快照。
  - 导出日志 -> `exportLogs()` -> `LogExportService.export()` -> 查询当前过滤快照并通过 CSV 导出服务写出文件。

#### 3.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `LogEventType`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 定义日志事件类型枚举。
    * **关键字段**: `CRASH`、`FILE`、`ACCOUNT`、`PERMISSION`、`POWER`。
  * `LogEventResult`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 定义日志结果枚举，用于列表筛选、统计和详情展示。
    * **关键字段**: `SUCCESS`、`BLOCKED`、`FAILED`。
  * `LogEntry`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 日志管理模块的核心记录实体，是采集、存储、查询、详情和导出的统一数据结构。
    * **关键字段**: `id`、`timestamp`、`eventType`、`result`、`detail`、`processName`、`pid`、`uid`、`filePath`、`sourceEventId`、`rawEventName`、`rawOutcome`、`rawAction`、`detailRows`。
  * `LogStorageConfig`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 控制本地最大日志条数和保留天数。
    * **关键字段**: `maxLocalEntries` 默认 `20000`，`retentionDays` 默认 `30`。
    * **配置选项**: 
      * 最大存储条数选项：`[5000, 10000, 20000, 50000, 100000]`，最大值 `100000`
      * 保留天数选项：`[7, 14, 30, 90, 180]`，最大值 `180`
  * `LogSubscriptionConfig`
    * **所在文件**: `entry/src/main/ets/models/DataModels.ets`
    * **业务作用**: 记录日志订阅配置。
    * **关键字段**: `subscribeAll` 默认 `true`，`subscribedTypes` 默认 `ALL_LOG_EVENT_TYPES`。
  * `LogManageState` 与子状态
    * **所在文件**: `entry/src/main/ets/models/LogManageModels.ets`
    * **业务作用**: 页面状态树，承载采集器、摘要、筛选、分页、列表、详情和模式状态。
    * **关键字段**: `collector`、`summary`、`filter`、`pagination`、`list`、`detail`、`mode`。
  * `LogStorageSettingsState`
    * **所在文件**: `entry/src/main/ets/models/LogManageModels.ets`
    * **业务作用**: 存储设置页状态。
    * **关键字段**: `maxLocalEntries`、`retentionDays`、`dirty`、`saving`。

#### 3.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/log-manage`
* **核心业务用例 (Use Cases)**:
  * `LogCollectorService.start(context)`: 启动日志采集。
    * **副作用**: 初始化存储、订阅系统审计事件、启动崩溃源、记录采集器状态。
    * **失败策略**: 审计或崩溃源失败时进入降级态，保留可用采集源。
  * `LogCollectorService.stop()`: 停止采集。
    * **副作用**: 取消审计订阅、停止崩溃源。
    * **失败策略**: 返回状态结果并记录错误日志。
  * `LogAuditSource.subscribe(onEntry, eventTypes?)`: 订阅系统审计事件并将审计事件映射为 `LogEntry`。
    * **副作用**: 创建 `securityAudit` 客户端、按事件逐个订阅、为文件事件挂载前缀过滤器。
    * **失败策略**: 全部失败返回 `unsupported` 或 `permission_denied`，部分失败返回 `partial`，保留已成功订阅的事件。
  * `LogStorageMaintenanceService.appendEntries(entries)`: 批量写入日志并执行维护。
    * **副作用**: 写入 RDB，按留存天数删除过期日志，按最大条数裁剪旧日志。
    * **失败策略**: 返回 `MutationResult`，由调用方决定是否刷新 UI。
  * `LogEntryRepository.listByQuery(query, limit, offset)`: 按筛选条件分页查询日志。
    * **副作用**: 无，仅查询 RDB。
    * **失败策略**: Repository 层记录错误，调用方返回查询失败状态。
  * `LogEntryRepository.getStatistics()`: 统计总数、今日数量、拦截数量和类型分布。
    * **副作用**: 无。
    * **失败策略**: 查询失败时由上层刷新流程返回失败结果。
  * `LogExportService.export(context, query, fileNamePrefix?)`: 导出当前查询条件下的日志。
    * **副作用**: 查询日志快照并调用公共 CSV 文件导出服务。
    * **失败策略**: 返回 `ExportLogsResult`，区分空数据、查询失败和写文件失败。
  * `LogStorageSettingsViewModel.save()`: 保存存储配置。
    * **副作用**: 写入 Preferences，触发后续维护策略使用新配置。
    * **失败策略**: 保存失败保持 `dirty` 状态，便于用户重试。
* **审计事件订阅设计**:
  * **订阅入口**: `entry/src/main/ets/services/log-manage/source/LogAuditSource.ets`
  * **订阅来源**: `@kit.DeviceSecurityKit` 的 `securityAudit`。
  * **类型到审计事件映射**:

| 日志类型 | 审计事件 | 说明 |
|---|---|---|
| `LogEventType.FILE` | `securityAudit.NotifyEvent.FILE` | 文件访问事件 |
| `LogEventType.FILE` | `securityAudit.NotifyEvent.FILE_INTERCEPTED` | 文件访问被拦截事件 |
| `LogEventType.FILE` | `securityAudit.NotifyEvent.KIA_CREATE` | KIA 创建类文件事件 |
| `LogEventType.FILE` | `securityAudit.NotifyEvent.KIA_READ` | KIA 读取类文件事件 |
| `LogEventType.FILE` | `securityAudit.NotifyEvent.KIA_VARIANT` | KIA 变体类文件事件 |
| `LogEventType.FILE` | `securityAudit.NotifyEvent.KIA_INTERCEPT` | KIA 拦截类文件事件 |
| `LogEventType.ACCOUNT` | `securityAudit.NotifyEvent.ACCOUNT` | 账户事件 |
| `LogEventType.PERMISSION` | `securityAudit.NotifyEvent.PERMISSION` | 权限事件 |

  * **未订阅说明**: `CRASH` 由 `LogCrashSource` 采集，`POWER` 由 `LogPowerLifecycleSource` 采集，不走 `securityAudit.NotifyEvent` 订阅。
* **文件事件前缀过滤器设计**:
  * **挂载事件**: 仅挂载到 `FILE` 类型对应的 6 个审计事件：`FILE`、`FILE_INTERCEPTED`、`KIA_CREATE`、`KIA_READ`、`KIA_VARIANT`、`KIA_INTERCEPT`。
  * **过滤器类型**: `securityAudit.FilterType.FILE_PATH_PREFIX`。
  * **过滤器语义**: `isInclude = true`，只接收指定路径前缀下的文件审计事件。
  * **前缀来源**: `LogAuditSource.prepareFileFilterContext()` 获取前台系统用户 ID，再调用 `LogPathUtils.buildHmdfsAccountFilesPrefix(userId)`。
  * **前缀格式**: `/data/service/el2/{foregroundUserId}/hmdfs/account/files/`。
  * **用户 ID 获取**: `osAccount.getAccountManager().getForegroundOsAccountLocalId()`；获取失败时回退 `DEFAULT_FOREGROUND_USER_ID = 100`。
  * **典型值**: 当前用户为 `100` 时，过滤前缀为 `/data/service/el2/100/hmdfs/account/files/`。
  * **挂载顺序**: 每个文件审计事件先 `client.subscribe([eventId])`，再 `client.addFilter(eventId, filter)`；若过滤器挂载失败，会立即取消该事件订阅并将该事件计入失败列表。
  * **失败策略**: 文件过滤上下文不可用、客户端无 `addFilter`、或系统返回 `FILE_FILTER_UNSUPPORTED_CODE = 1012000005` 时，该文件事件不进入有效订阅；若仍有其他事件成功，整体返回 `partial`。
* **内部架构设计**:
  * **Collector**: `entry/src/main/ets/services/log-manage/collector/LogCollectorService.ets` - 统一协调审计源、崩溃源和生命周期源，并维护采集状态。
  * **Audit Source**: `entry/src/main/ets/services/log-manage/source/LogAuditSource.ets` - 系统审计订阅、前缀过滤器挂载和审计回调入口。
  * **Audit Mapper**: `entry/src/main/ets/services/log-manage/source/LogAuditEventMapper.ets` - 将审计事件解析为 `LogEntry`，抽取事件名、结果、动作、路径和详情行。
  * **Crash Source / Mapper**: `entry/src/main/ets/services/log-manage/source/LogCrashSource.ets`、`entry/src/main/ets/services/log-manage/source/LogCrashEventMapper.ets` - 采集并转换崩溃日志。
  * **Power Lifecycle Source**: `entry/src/main/ets/services/log-manage/source/LogPowerLifecycleSource.ets` - 记录应用生命周期类日志。
  * **Timestamp Resolver**: `entry/src/main/ets/services/log-manage/source/LogTimestampResolver.ets` - 统一审计事件时间戳解析。
  * **Repository**: `entry/src/main/ets/services/log-manage/repository/LogEntryRepository.ets` - RDB 表结构、查询、统计、去重写入、过期删除和容量裁剪。
  * **Settings Repository**: `entry/src/main/ets/services/log-manage/repository/LogSettingsRepository.ets` - Preferences 读写本地留存配置。
  * **Maintenance**: `entry/src/main/ets/services/log-manage/maintenance/LogStorageMaintenanceService.ets` - 存储初始化、批量追加、清理和留存维护。
  * **Export**: `entry/src/main/ets/services/log-manage/export/LogExportService.ets` - 日志导出用例。
  * **Ports**: `entry/src/main/ets/services/log-manage/common/LogManagePorts.ets` - ViewModel 与服务层之间的查询、快照、结果和端口契约。
  * **Value Picker**: `entry/src/main/ets/utils/ValuePicker.ets` - 审计、崩溃和持久化扩展字段的公共容错抽取。

#### 3.3 ViewModel (视图模型层)
* **真实文件路径**:
  * `entry/src/main/ets/viewmodels/log-manage/overview/LogManageViewModel.ets`
  * `entry/src/main/ets/viewmodels/log-manage/storage-settings/LogStorageSettingsViewModel.ets`
* **状态分发逻辑**:
  * `initialize(context)`: 初始化存储、注册变更监听、加载设置、进入列表模式并构建首屏快照。
  * `activate()` / `deactivate()` / `destroy()`: 控制页面活跃态、自动刷新和监听释放。
  * `refreshAll()` / `refreshIfActive()`: 从 Repository 快照重建摘要、分页和列表状态。
  * `setMode(mode)`: 切换列表和设置视图；进入设置视图时暂停列表自动刷新。
  * `setEventTypeIndex(index)` / `setResultIndex(index)` / `setKeyword(keyword)`: 更新过滤条件，重置第一页并刷新。
  * `setPageSizeIndex(index)` / `jumpToInputPage()` / `goToPreviousPage()` / `goToNextPage()`: 驱动分页状态和当前页列表；分页刷新内部捕获查询异常并返回 `RefreshResult`，即使页面层以 fire-and-forget 方式触发也不得产生未处理异步错误。
  * `openDetail(entryId)` / `closeDetail()`: 控制详情弹窗。
  * `clearLogs()`: 清空日志并重建快照。
  * `exportLogs(context)`: 按当前查询条件导出日志。
  * `LogStorageSettingsViewModel.load()` / `save()`: 加载和保存最大条数、留存天数配置。

## 4. 异常处理与系统依赖 (Dependencies & Errors)
- **关键系统 API**:
  - `@kit.DeviceSecurityKit.securityAudit`: 系统审计事件客户端、事件订阅、事件取消订阅和文件前缀过滤器。
  - `@ohos.account.osAccount`: 获取前台系统用户 ID，用于构造 HMDFS 文件审计前缀。
  - `@ohos.data.relationalStore`: 本地 RDB 存储。
  - Ability `common.UIAbilityContext`: 初始化存储和导出文件时使用。
- **系统权限**:
  - `ohos.permission.QUERY_AUDIT_EVENT`: 订阅系统审计事件。
  - `ohos.permission.GET_LOCAL_ACCOUNTS`: 获取前台系统用户 ID。
  - `ohos.permission.PREPARE_APP_TERMINATE`: 应用终止前准备能力。
  - 其他企业管理权限以 `entry/src/main/module.json5` 当前声明为准，不在日志管理模块内独立扩展。
- **异常兜底策略**:
  - 审计客户端不可用：`LogAuditSource.subscribe()` 返回 `unsupported`，采集器进入降级态。
  - 审计权限不足或订阅失败：返回 `permission_denied`，记录错误码，保留其他可用采集源。
  - 部分审计事件失败：返回 `partial`，`failedEventIds` 记录失败事件，成功事件继续工作。
  - 文件前缀过滤器不可用：文件事件不保留裸订阅，避免接收未限定路径范围的文件审计事件。
  - 前台用户 ID 获取失败：回退到用户 `100` 构造前缀。
  - 审计事件解析失败：丢弃无法映射为 `LogEntry` 的事件，不写入存储。
  - 扩展 JSON 解析失败：返回空扩展字段，保留主记录展示。
  - 查询刷新失败：返回 `query_failed`，页面保持现有状态并记录错误日志。
  - 分页刷新失败：返回 `query_failed`，保留最近一次成功分页列表和页码状态，并记录错误日志，不向 UI 事件回调抛出未处理异常。
  - 清理失败：返回 `clear_failed`，不主动清空 UI 列表。
  - 导出无数据：返回空数据结果，由 UI 给出提示。
  - 设置保存失败：保持脏状态，允许用户重试。

### 4.1 实施步骤与测试验收 (Implementation & Acceptance)

- **实施步骤**:
  - 涉及采集源、过滤条件、分页、详情、清理、留存配置、导出、RDB schema 或权限变化时，先更新本文档的目标范围、数据模型、状态流、职责边界、异常兜底和验收口径。
  - 数据结构和枚举先同步 `LogModels.ets`、`LogSourcePorts.ets`、相关字符串和筛选常量，再调整采集、仓储或页面。
  - 系统审计、崩溃、生命周期等输入先落在 Source / Collector / Maintenance 边界，再由 Repository 和 ViewModel 统一提供页面状态；页面不得直接解析审计事件或访问 RDB。
  - RDB 字段、索引、扩展 JSON 或留存策略变化时，需要同步迁移/兼容策略、清理策略、导出字段和历史数据降级展示。
  - 新增或调整审计权限时，同步 `entry/src/main/module.json5`、`hapsigner/UnsgnedDebugProfileTemplate.json` 并按签名流程重新生成 p7b。
- **核心伪码**:
  ```text
  initializeLogRuntime(context):
    LogCollectorService.initialize(context)
    subscribe audit / crash / power sources
    if audit source unsupported:
      mark collector degraded and keep other sources running

  refreshLogList(query):
    snapshot = LogEntryRepository.query(query)
    if query fails:
      keep last successful list and return query_failed
    map snapshot to LogManageState

  exportLogs(query):
    entries = LogEntryRepository.queryAll(query)
    if entries empty:
      return empty_result
    return LogExportService.exportCsv(entries)

  saveStorageSettings(settings):
    persist settings
    run maintenance trim by maxEntries and retentionDays
    keep dirty state if save fails
  ```
- **测试覆盖**:
  - UT: `entry/src/test/log-manage/*.test.ets` 覆盖采集源、审计映射、仓储查询、留存配置、导出、ViewModel 状态和异常分支。
  - ohosTest: `entry/src/ohosTest/ets/test/log-manage/*.test.ets` 覆盖页面组件和基础流程。
  - E2E: `scripts/e2e/cases/logs/` 下 `page_details.json`、`filters_visible.json`、`stats_visible.json`、`storage_settings_visible.json`、`export.json`、`event_after_policy_change.json` 覆盖查询、筛选、统计、存储配置、导出和联动事件。
- **验收口径**:
  - 审计能力不可用或权限不足时，采集器进入可解释降级态，页面仍可查看已有日志和其它来源日志。
  - 筛选、分页、详情和统计之间状态一致；分页失败保留最近一次成功列表，不向 UI 抛出未处理异常。
  - 留存配置保存失败保持脏状态，导出无数据给出明确提示，清理失败不主动清空 UI 列表。
