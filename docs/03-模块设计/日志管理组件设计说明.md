## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 采集设备安全相关日志，提供本地查询、过滤、分页、详情查看、清理、留存配置、CSV 导出和离线日志导入能力，为安全审计和演示验收提供可追溯证据。
- **入口路由 (Route Entry)**: `log-manage` -> `entry/src/main/ets/views/log-manage/overview/LogManagePage.ets`
- **对外暴露能力 (Public APIs)**:
  - 页面入口由主导航路由到 `LogManagePage`。
  - 业务层主要通过 `LogCollectorService`、`LogEntryRepository`、`LogStorageMaintenanceService`、`LogExportService`、`LogImportService` 对模块内部提供采集、存储、维护、导出和离线导入能力。
  - 当前没有面向其他业务模块的公共 SDK 或跨模块写入接口。
- **业务边界**:
  - ✅ **包含**: 系统审计日志订阅、崩溃日志采集、应用前后台电源生命周期日志、本地 RDB 存储、日志查询过滤、分页展示、详情弹窗、日志清理、留存配置、CSV 导出、zip 离线日志导入、内核日志导入规则、faultlogger 崩溃日志导入规则。
  - ❌ **不包含**: 测试代码设计、测试用例说明、远端日志上报、云端审计平台、用户自定义审计规则、审计事件策略下发、系统审计框架自身能力实现、`remoteLog/hilog/hilog.*` 普通 hilog 文本导入。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `LogManageState.initialized: boolean`: 页面是否完成存储初始化、设置加载和首屏快照构建。
  - `LogManageState.loading: boolean`: 页面初始化或关键异步操作中的加载态。
  - `LogManageState.active: boolean`: 列表页是否处于活跃态；非活跃时自动刷新被跳过。
  - `LogManageState.autoRefreshEnabled: boolean`: 是否响应存储变更自动刷新。
  - `LogManageState.mode: 'list' | 'settings'`: 当前展示列表视图或存储设置视图。
  - `LogSummaryState`: 总数、过滤后数量、今日数量和当前页数量。
  - `LogFilterState`: 事件类型筛选、时间范围筛选和关键字；时间范围默认 `0` 表示“全部时间”，其它选项由展示层统一配置为最近 1 小时、最近 12 小时、最近 24 小时和最近 3 天。
  - `LogPaginationState`: 当前页、页大小、页码输入和总页数。
  - `LogListState.pagedEntries: LogEntry[]`: 当前页展示日志。
  - `LogDetailState`: 详情弹窗可见性和当前日志 `entryId`。
  - `LogStorageSettingsState`: 最大本地条数、留存天数、脏状态和保存态。
- **关键流转路径**:
  - 页面初始化 -> `LogManageViewModel.initialize()` -> 初始化 RDB、注册变更监听、加载设置、构建快照 -> 更新 `initialized / summary / list / pagination`，初始化失败时记录 error 日志并保持未初始化态。
  - 页面激活 -> `activate()` -> `active = true` -> 存储变更可触发 `refreshIfActive()`。
  - 切换到设置页 -> `setMode('settings')` -> 加载设置、关闭详情、暂停列表活跃态和自动刷新。
  - 修改筛选条件 -> `setEventTypeIndex()` / `setTimeRangeIndex()` / `setKeyword()` -> 重置到第一页 -> ViewModel 将时间范围选项转换为 `LogQuery.startTimestamp` -> 先计算过滤后总数，再复用该计数构建查询快照并刷新当前页。
  - 分页操作 -> `setPageSizeIndex()` / `jumpToInputPage()` / `goToPreviousPage()` / `goToNextPage()` -> 更新分页状态 -> 复用过滤后总数重建当前页列表。
  - 日志采集 -> `LogCollectorService` 从审计/崩溃/生命周期源接收 `LogEntry` -> `LogStorageMaintenanceService.appendEntries()` -> `LogEntryRepository.upsertEntries()` -> 通知 ViewModel 刷新。
  - 开关机日志采集 -> `EntryAbility.onCreate()` 调用 `LogPowerLifecycleSource.init()` -> 存储初始化成功后先处理上次关机 pending marker，再直接写入本次应用冷启动 `POWER` 日志 -> 订阅 `usual.event.SHUTDOWN`，收到关机事件后先写 pending marker，再写入关机 `POWER` 日志。
  - 清理日志 -> `clearLogs()` -> `LogStorageMaintenanceService.clearAll()` -> 关闭详情并重建快照。
  - 导出日志 -> `exportLogs()` -> `LogExportService.export()` -> 查询当前过滤快照并通过 CSV 导出服务写出文件。
  - 添加日志 -> `importLogs()` -> `LogImportService.import()` / `importSelected()` -> 选择外层 zip，服务层自动使用应用私有临时工作根目录 -> `taskpool.execute(runLogImportParseTask)` 调度 `LogImportBackgroundTask` -> 后台任务调用 `LogArchiveExtractor` 解压外层 zip / 嵌套 zip 并枚举文件元数据 -> 可插拔导入规则处理候选文件，其中 `KernelLogImportRule` 过滤 `hilog/hilog_kmsg*`、按需同目录预解压 `.gz`、逐行解析内核日志，`CrashLogImportRule` 过滤 `faultlogger/*.log`、按文件解析崩溃日志，`SystemLogImportRule` 按白名单路径解析系统日志 -> `LogImportHash` 生成稳定 ID -> 后台任务先按确定性 ID 批内去重并返回解析统计 -> `LogImportService.importParsedEntries()` 在主线程按过期、库内重复顺序过滤 -> `LogStorageMaintenanceService.appendEntries()` 入库 -> 刷新列表与统计 -> 无论成功、失败或异常均清理本次临时工作区。

## 4. 模块结构与组件设计 (Module Components)

### 【核心层】(Core MVVM & Domain Layers)

#### 4.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `LogEventType`
    * **所在文件**: `entry/src/main/ets/models/LogModels.ets`
    * **业务作用**: 定义日志事件类型枚举。
    * **关键字段**: `CRASH`、`FILE`、`ACCOUNT`、`PERMISSION`、`POWER`、`SYSTEM`、`KERNEL`、`AUDIT`。
    * **导入说明**: `SYSTEM`、`KERNEL` 仅由离线导入链路产生，不接入 `LogCollectorService` 实时采集；`AUDIT` 由实时审计订阅链路产生，用于承接当前项目未归入文件、账号/认证、权限、崩溃等专用类型的其它 `securityAudit.NotifyEvent`。
    * **筛选值顺序**: `ALL_LOG_EVENT_TYPES` 是 ViewModel 查询映射和页面展示选项共同使用的业务顺序；展示层只能基于该数组派生 label，不得另建业务顺序。
  * `LogEntry`
    * **所在文件**: `entry/src/main/ets/models/LogModels.ets`
    * **业务作用**: 日志管理模块的核心记录实体，是采集、存储、查询、详情和导出的统一数据结构。
    * **关键字段**: `id`、`timestamp`、`eventType`、`detail`、`processName`、`pid`、`uid`、`filePath`、`sourceEventId`、`rawEventName`、`rawOutcome`、`rawAction`、`detailRows`。
    * **规范化规则**: `LogEntryNormalizer` 对详情、文件路径、远端地址、原始事件名、原始结果、动作和详情行做长度限制；截断后的最终字符串长度不得超过对应上限，追加省略号时必须先为省略号预留长度。
  * `LogStorageConfig`
    * **所在文件**: `entry/src/main/ets/models/LogModels.ets`
    * **业务作用**: 控制本地最大日志条数和保留天数。
    * **关键字段**: `maxLocalEntries` 默认 `20000`，`retentionDays` 默认 `30`。
    * **页面默认**: 存储设置面板未收到 ViewModel 入参时必须使用 `DEFAULT_LOG_STORAGE_CONFIG.retentionDays`，不得退回到保留天数选项首项。
    * **配置选项**: 
      * 最大存储条数选项：`[5000, 10000, 20000, 50000, 100000]`，最大值 `100000`
      * 保留天数选项：`[7, 14, 30, 90, 180]`，最大值 `180`
  * `LogManageState` 与子状态
    * **所在文件**: `entry/src/main/ets/models/LogManageModels.ets`
    * **业务作用**: 页面状态树，承载摘要、筛选、分页、列表、详情和模式状态。
    * **关键字段**: `summary`、`filter`、`pagination`、`list`、`detail`、`mode`。
    * **时间筛选**: `filter.timeRangeIndex` 记录页面时间范围筛选选项，`0` 为“全部时间”，`1` 为最近 1 小时，`2` 为最近 12 小时，`3` 为最近 24 小时，`4` 为最近 3 天；非 `0` 时由 `LogManageViewModel.buildQuery()` 转换为 `startTimestamp = Date.now() - durationMs`。
  * `LogStorageSettingsState`
    * **所在文件**: `entry/src/main/ets/models/LogManageModels.ets`
    * **业务作用**: 存储设置页状态。
    * **关键字段**: `maxLocalEntries`、`retentionDays`、`dirty`、`saving`。
  * `ImportLogsCode`、`ImportLogsResult`、`LogImportSelectionResult`
    * **所在文件**: `entry/src/main/ets/models/LogImportModels.ets`
    * **业务作用**: 定义离线日志导入在 View、ViewModel 与 Service 之间共享的选择结果和导入结果契约。
    * **边界约束**: 页面和 ViewModel 只依赖模型层结果，不从 `LogImportService.ets` 导入 DTO；Picker/Workspace 端口仍由服务层持有。

#### 4.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/log-manage`
* **核心业务用例 (Use Cases)**:
  * `LogCollectorService.start(context)`: 启动日志采集。
    * **副作用**: 初始化存储、按当前内置映射订阅全部系统审计事件、启动崩溃源、记录采集器状态。
    * **失败策略**: 审计或崩溃源失败时进入降级态，保留可用采集源。
    * **文件日志噪音过滤**: 文件类审计事件进入 `LogCollectorService.handleIncomingEntry()` 后、写入 `pendingEntries` 和 RDB 前，必须先执行内部噪音过滤。过滤范围包含本应用设置、主题、HiAppEvent、应用数据库、日志导出目录以及日志导入过程中在用户文件空间 `Docs/.Trash/.../security_log_import_*` 目录下产生的解压/扫描文件事件。该规则只影响新增文件操作日志，不迁移、不删除历史已入库记录。
  * `LogCollectorService.stop()`: 停止采集。
    * **副作用**: 取消审计订阅、停止崩溃源。
    * **失败策略**: 返回状态结果并记录错误日志。
  * `LogAuditSource.subscribe(onEntry, eventTypes?)`: 订阅系统审计事件并将审计事件映射为 `LogEntry`。
    * **副作用**: 创建 `securityAudit` 客户端、按事件逐个订阅、为文件事件挂载前缀过滤器。
    * **失败策略**: 全部失败返回 `unsupported` 或 `permission_denied`，部分失败返回 `partial`，保留已成功订阅的事件。
  * `LogStorageMaintenanceService.appendEntries(entries)`: 批量写入日志并执行维护。
    * **副作用**: 写入 RDB，按留存天数删除过期日志，按最大条数裁剪旧日志。
    * **失败策略**: 返回 `MutationResult`，由调用方决定是否刷新 UI。
  * `LogEntryRepository.listByQuery(query, limit, offset)`: 按筛选条件分页查询日志。
    * **副作用**: 无，仅查询 RDB。
    * **时间范围**: 当 `LogQuery.startTimestamp` 存在时，查询条件追加 `timestamp >= startTimestamp`；该条件与事件类型和关键字使用 `AND` 组合。
  * **失败策略**: Repository 层记录错误，日志列表查询、计数和统计失败必须向上暴露为查询失败，不得以空数组或 0 条静默伪装成功。
  * `LogEntryRepository.getStatistics()`: 统计总数、今日数量和类型分布。
    * **副作用**: 无。
    * **失败策略**: 查询失败时由上层刷新流程返回失败结果，页面保留可解释的失败态或最近一次成功状态。
  * `LogExportService.export(context, query, fileNamePrefix?)`: 导出当前查询条件下的日志。
    * **副作用**: 查询日志快照并调用公共 CSV 文件导出服务；导出内容必须写入 UTF-8 BOM，所有字段统一经 CSV 转义后输出；公共导出服务必须以统一文件打开/关闭流程写入，并在覆盖同名文件时截断旧内容。
    * **失败策略**: 返回 `ExportLogsResult`，区分空数据、查询失败和写文件失败。
  * `LogImportService.import(context)`: 从用户选择的外层 zip 包导入离线日志。
    * **副作用**: 通过 Picker 选择 zip，自动使用应用私有临时工作根目录解压外层 zip、递归处理嵌套 zip、枚举解压后的文件元数据，调用导入规则生成 `LogEntry`，过滤重复记录后复用维护服务入库；导入结束后必须清理临时工作根目录。
    * **失败策略**: 区分用户取消、非法压缩包、解压失败、规则解析失败和入库失败；规则无法解析完整字段的记录直接丢弃，不入库。
  * `LogStorageSettingsViewModel.save()`: 保存存储配置。
    * **副作用**: 写入 Preferences，触发后续维护策略使用新配置。
    * **失败策略**: 保存失败保持 `dirty` 状态，便于用户重试。

  * **API26 文件事件来源**: 首批 API26 文件事件仅接入已完成自维护 API 文档验证的 `FILE_CREATE`、`FILE_CLOSE`、`FILE_COPY`、`FILE_SETOWNER`、`FILE_SETMODE`、`FILE_SETEXTATTR`、`FILE_DELETEEXTATTR`。`FILE_OPEN`、`FILE_DELETE`、`FILE_RENAME`、`FILE_WRITE`、`FILE_SHARE`、`DATA_DRAG`、`DLP_FILE_ACCESS` 当前不接入；原因是尚未稳定确认触发条件与日志详情，避免订阅后无法解释日志含义。验证依据维护在 `docs/06-自维护API文档/API26文件审计事件技术穿刺.md`。
  * **审计日志详情**: `LogEventType.AUDIT` 的详情弹窗默认展示两类结构化字段：`事件说明` 和 `事件原始JSON`。事件说明来自事件 ID 到官方说明的固定映射；事件原始 JSON 由回调中的 `eventId`、`timestamp`、`userId`、`content`、`metadata` 组装。字段仍进入 `LogEntryNormalizer`，超出详情行值长度上限时按既有规则截断并追加省略号。企业设备管理事件（`securityAudit.NotifyEvent.EDM`）从原始 JSON 的 `objectInfo` 增量展示“操作对象”；当 `objectInfo` 命中已实机确认映射时，额外展示“操作备注”。外设管理映射覆盖 `disable_usb`、`usb_read_only`、`disabled_bluetooth`、`disable_wifi`、`disabled_printer`、`disable_microphone`、`disable_camera`、`disabled_hdc`、`disallowed_tethering`、`get_network_interfaces`、`disallowed_usb_devices`、`allowed_usb_devices` 和 `set_switch_status`，统一使用“管理 / 查询 / 策略”中性语义，不把策略对象名解释为已执行启用或禁用。`allowed_usb_devices` 按公开 API 语义展示为“USB 设备信任列表”；无公开 API 和项目调用依据的 `disallowed_permissive_usb_devices` 不做中文推断，按未知对象回退显示原始值。实机样本中的 `targettInfo`、`extra` 均为空且 `subType` 固定为 `0`，当前不生成无业务信息的详情行；其它未知 `objectInfo` 仍只展示原始操作对象，不展示中文备注。
  * **EDM 审计列表摘要**: `securityAudit.NotifyEvent.EDM` 新入库记录的 `LogEntry.detail` 必须基于同次解析生成的 `detailRows` 构造，不再直接使用通用审计 `content` / `metadata` 摘要。摘要字段优先级为 `audit.edm.objectRemark` -> `audit.edm.objectInfo`，以及 `audit.edm.operationTypeRemark` -> `audit.edm.operationType`，按 `对象语义 | 操作类型语义` 拼接；例如 `Wi-Fi 管理 | 读取/查询策略状态`、`disable_usb | 读取/查询策略状态`、`Wi-Fi 管理 | type=9`。`audit.description` 和 `audit.rawJson` 不进入 `LogEntry.detail`；当对象和操作类型均缺失时，才回退到通用审计摘要。该规则只影响新采集或新导入后进入仓储的 EDM 记录，不迁移历史已入库日志。
  * **文件审计路径解析**: `LogAuditEventMapper` 必须只保留一个文件审计路径解析入口，统一产出 `primaryPath`、`oldPath`、`newPath`。`primaryPath` 同时写入 `LogEntry.filePath` 和详情行 `file.filePath`，用于列表搜索、导出数据源和详情展示的一致口径；不得在详情行构造时另写一套路径字段优先级。路径归一化继续复用 `LogPathUtils.normalizePathCandidate()`，将 `/data/service/el2/{userId}/hmdfs/account/files/Docs/Download|Documents|Desktop/...` 转为 `/storage/Users/currentUser/...`；结构化审计字段中的路径必须保留文件名内部空格，例如 `新建 文本文档.txt` 不得被截断。
  * **文件重命名路径语义**: 当文件审计事件同时包含旧路径和新路径时，`primaryPath` 取目标/新路径，详情展示 `文件路径 = primaryPath`、`原文件路径 = oldPath`、`新文件路径 = newPath`。典型字段包括 `file_oldpath` / `file_newpath`、`oldPath` / `newPath`、`srcPath` / `targetPath|dstPath`。普通文件事件只有 `file_path`、`filePath` 或 `path` 时，`primaryPath` 保持该直接路径语义。
  * **API26 文件事件详情**: 文件事件详情在现有“操作类型、文件路径、原文件路径、新文件路径、进程名、PID、UID、事件时间、事件 ID”基础上，若事件 ID 命中已验证 API26 文件事件说明映射，则展示“事件说明”，格式为“官方枚举名：中文说明”，例如 `FILE_SETMODE：文件权限模式变更事件`。
  * **API26 文件可选字段展示规则**: 对 `file_mode`、`file_newmode`、`file_owneruid`、`file_ownergid`、`file_newuid`、`file_newgid`、`file_xattrname`、`file_size` 采用“有值才展示”的结构化详情规则；历史 `FILE` 事件或系统未返回字段时，不展示 `未知`、`-`、`空` 等占位值。
  * **路径解析腐败代码清理**: 文件审计映射不得同时保留 `LogEntry.filePath` 路径解析和 `detailRows.file.filePath` 路径解析两套入口。实现时必须删除旧的详情路径 fallback、旧的独立原始路径解析函数和 `ParsedAuditPayload.rawFilePath` 中间字段，避免后续再次出现列表可搜索但详情缺路径的分叉。
* **开关机生命周期设计**:
  * **启动日志**: `LogPowerLifecycleSource.init(context)` 在 RDB/Preferences 初始化成功后，每个应用冷启动周期直接写入一条 `LogEventType.POWER` / `设备启动` 记录，`rawEventName = app.boot`，`detailRows.source` 按是否存在上次关机恢复场景标记为 `app_start` 或 `startup_recovery`。
  * **关机事件**: `LogPowerLifecycleSource` 通过 `commonEventManager` 动态订阅 `commonEventManager.Support.COMMON_EVENT_SHUTDOWN` 对应的 `usual.event.SHUTDOWN`，收到事件后先写入 `power_lifecycle.pending_shutdown_at` 和 `pending_shutdown_logged = 0`，再写入 `LogEventType.POWER` / `设备关闭` 记录。
  * **恢复补偿**: 下次启动若读取到 `pending_shutdown_at > 0`，表示上次关机事件至少进入过本应用处理链路；若 `pending_shutdown_logged = 0`，先按该时间补写一条来源为 `startup_recovery` 的关机日志；若已为 `1`，不重复补写关机日志。补偿处理完成后清理 marker。
  * **离线类型说明**: `KERNEL` 不走 `securityAudit.NotifyEvent` 订阅，只由“添加日志”离线导入产生；`CRASH` 既可由运行时 `LogCrashSource` 采集，也可由“添加日志”解析 `faultlogger` 离线文件产生；`SYSTEM` 由“添加日志”按白名单系统日志规则离线导入产生。
* **离线导入设计**:
  * **入口**: `entry/src/main/ets/views/log-manage/overview/LogManagePage.ets` 在“导出日志”旁提供“添加日志”按钮；页面先调用 `LogManageViewModel.pickImportTargets(context)` 完成 zip 选择并由服务层生成默认 `targetDirectory`，选择完成后展示页面内等待遮罩，再调用 `LogManageViewModel.importSelectedLogs(archivePath, targetDirectory)` 执行导入。
  * **线程模型**: Picker、等待遮罩、结果弹窗、库内重复过滤、RDB 入库和页面刷新保留在主线程；zip 解压、嵌套 zip 枚举、gzip 解压、日志文件读取、规则解析和批内去重通过 `@Concurrent` 后台任务执行，避免大文件解析持续占用 UI 主线程导致等待动画卡顿。
  * **职责边界**: `LogImportService` 只负责导入用例编排、Picker 选择、后台解析任务调度、后台解析结果持久化、过期过滤和库内重复过滤；不直接持有或调用 `LogArchiveExtractor`、`LogImportRule`、`KernelLogImportRule`、`CrashLogImportRule`、`SystemLogImportRule`。`LogImportBackgroundTask` 是离线导入解析职责边界，统一负责 zip 解压、嵌套 zip 枚举、gzip 解压、日志文件读取、规则解析和批内去重。
  * **外层格式**: 用户 Picker 选择的外层压缩包仅支持 `.zip`；生产代码不得硬编码开发机 demo 路径，只使用 Picker 返回路径。
  * **临时工作区**: `LogImportService` 使用 `context.filesDir/log_import_tmp` 作为默认临时工作根目录，不再要求用户选择解压位置。每次导入前先清理该根目录下历史残留；`LogArchiveExtractor` 继续在根目录下创建 `security_log_import_<timestamp>` 子目录承载本次解压内容。
  * **清理要求**: `LogImportService.importSelected()` 必须在 `finally` 中清理本次传入的临时工作根目录。导入成功、解析失败、入库失败或异常退出后均必须执行清理；清理失败只记录错误日志，不覆盖导入主结果。
  * **解压规则**: `LogArchiveExtractor` 只负责解压外层 zip、递归处理嵌套 zip、枚举文件元数据；嵌套 zip 解压目录固定为 `<zipPath>_extracted`。`LogArchiveExtractor` 不读取日志全文，不判断 `hilog_kmsg*`，也不处理 `.gz` 日志载体。
  * **安全策略**: 解压时拒绝 `../`、绝对路径和逃逸目标目录的条目，避免路径穿越。
  * **规则端口**: `LogImportRule` 定义可插拔导入规则，输入解压后的文件元数据，输出 `LogEntry[]`、丢弃数和失败数；规则自行判断是否能处理该文件。
  * **内核规则**: `KernelLogImportRule` 只处理解压后 `hilog` 目录下文件名以 `hilog_kmsg` 开头的文件；若载体文件名带 `.gz`，按剥离 `.gz` 后的逻辑文件名判断，并将 `.gz` 按块解压到所在目录的同名去 `.gz` 文本文件后逐行解析。
  * **内核内存与调度策略**: `KernelLogImportRule` 不将完整 `.gz` 或完整文本文件读入内存，不对全文执行 `split()`；解析时按块/按行处理，命中规则后才构造 `LogEntry`。大文件解析必须按固定行数批次让出事件循环，避免在 UI 主线程连续执行字符串解析触发 `THREAD_BLOCK_6S`。
  * **内核热路径策略**: `parseModulePrefix`、正文规范化等逐行高频逻辑优先使用轻量字符串扫描，避免在大日志批量导入中频繁执行全局正则替换。
  * **内核文件偏移策略**: `fileIo.readSync/writeSync` 按文件偏移读写时必须维护累计 `readOffset/writeOffset`；禁止在循环中固定传入 `offset: 0`，避免反复读写同一 64KB 分片导致解压文件被覆盖或解析循环无法结束。
  * **内核诊断日志策略**: 导入链路只输出批次级、文件级和进度级日志，不逐行输出。日志至少包含导入批次 ID、候选文件数、当前规则和文件、`.gz` 解压字节进度、解析字节/行数/命中/丢弃进度、去重数量、入库数量和耗时，便于从设备日志定位卡在选择、解压、解析、去重或入库阶段。
  * **内核行过滤**: 仅处理包含 `comm=` 的行；不含 `comm=` 的行直接丢弃。
  * **内核保留规则**:

| 规则 ID | 类别 | 条件 | 结果 |
|---|---|---|---|
| `kernel_hmseh_denied` | 安全访问控制拒绝 | `modulePrefix == hmseh` 且包含 `avc: denied` 且包含 `permissive=0` 且不包含 `path="/sys/power/last_sr"` | 导入 |
| `kernel_binder_perm_failed` | Binder/IPC 权限失败 | `modulePrefix == binderfs_write` 且包含 `translate_binder failed` 且包含 `err=E_HM_PERM` | 导入 |
| `kernel_hmdec_rule` | DEC 规则设置 | `modulePrefix == hmdec` | 导入 |
| `kernel_fs_devfs_dec` | DEC 设备文件事件 | `modulePrefix == fs_devfs` | 导入 |

  * **确定性 ID**: `LogImportHash` 基于 `eventType + rawEventName + normalizedDetail` 生成稳定 `LogEntry.id`；来源压缩包、来源文件、行号、时间、PID、UID 不参与 hash，因此相同正文不同来源视为重复。
  * **崩溃规则**: `CrashLogImportRule` 只处理解压后 `faultlogger` 目录下文件名匹配 `appfreeze-*.log`、`jscrash-*.log`、`cppcrash-*.log` 的文本文件；一个 faultlogger 文件只生成一条 `LogEventType.CRASH` 记录，不按堆栈行拆分多条日志。
  * **崩溃字段映射**: 崩溃规则从文件名和文本头部抽取 `crashType`、`processName`、`pid`、`uid`、`timestamp`、`reason`、`errorName`、`errorMessage`、`stringId` 和首个有效堆栈摘要。`detail` 只保存短摘要，完整文件不写入数据库；排查字段通过 `detailRows` 展示来源文件、崩溃类型、原因、错误名称、错误消息、STRINGID、Top stack 等结构化信息；不再派生 `rawOutcome = freeze/crash`。
  * **崩溃确定性 ID**: 崩溃导入 ID 基于 `eventType + crashType + processName + crashTimestamp + reason摘要 + topStack摘要` 生成；来源压缩包、来源文件路径不参与 hash，相同崩溃内容从不同来源导入视为重复；崩溃时间参与 hash，同一应用同一原因在不同时间发生时保留为不同事件。
  * **崩溃内存与调度策略**: `CrashLogImportRule` 按块/按行读取 faultlogger 文件，仅保留摘要字段和有限 detailRows，不将完整文件写入 `detail`。解析多个大文件时按固定行数批次让出事件循环，避免阻塞 UI 主线程。
  * **系统日志规则**: `SystemLogImportRule` 严格按白名单路径处理系统日志，不处理 `remoteLog/hilog/hilog.*`。路径匹配统一转小写并将 `\` 归一为 `/`。
  * **系统日志处理范围**:

| 路径规则 | 系统日志类型 | 切分规则 |
|---|---|---|
| `remoteLog/eventlog/SERVICE_TIMEOUT_WARNING-*.log` | 服务超时异常 | 1 文件生成 1 条 `LogEntry` |
| `remoteLog/eventlog/WINDOW_EXCEPTION_DETECTION-*.log` | 窗口异常检测 | 1 文件生成 1 条 `LogEntry` |
| `remoteLog/hisi_logs/hisi_diaginfo.log` | 硬件/SoC 诊断 | 每条 `Time [...]` 生成 1 条 |
| `remoteLog/f2fs-tools/fsck.log` | 文件系统检查 | 每个 `=== yyyy/MM/dd HH:mm:ss ===` 段生成 1 条 |
| `remoteLog/f2fs-tools/resize.log` | 文件系统扩容/维护 | 每个 `=== yyyy/MM/dd HH:mm:ss ===` 段生成 1 条 |
| `remoteLog/memory/proc_meminfo*.txt` | 内存资源快照 | 1 文件生成 1 条摘要 |
| `remoteLog/memory/proc_memview*.txt` | 内存视图快照 | 1 文件生成 1 条摘要 |
| `remoteLog/memory_leak/*.txt` | 内存泄漏诊断 | 1 文件生成 1 条报告摘要，低价值短文件丢弃 |
| `remoteLog/updater/updater_log` | 系统升级/恢复日志 | 命中关键行时每行生成 1 条 |
| `remoteLog/updater/updater_stage_log` | 系统升级阶段日志 | 命中关键行时每行生成 1 条 |
| `remoteLog/log/sys_installer.log` | 系统安装/参数更新日志 | 命中关键行时每行生成 1 条 |
| `remoteLog/log/HSLd.log` | 系统服务启动/挂载日志 | 命中关键行时每行生成 1 条 |

  * **系统关键行规则**: 仅用于 `updater_log`、`updater_stage_log`、`sys_installer.log`、`HSLd.log`。空行、纯分隔符行、长度小于 8 的行丢弃；仅包含错误、告警、失败、成功、升级阶段、恢复模式或非零返回码等关键语义的行生成 `SYSTEM` 记录，普通 `INFO/I` 行只有包含关键语义时才保留，不设置单文件命中上限。
  * **系统 memory_leak 保留规则**: `remoteLog/memory_leak/*.txt` 满足任一条件才保留：包含 `Timestamp:`；包含 `Pid:`；包含 `LOGGER_MEMCHECK`；包含 `memoryName:` 且文件行数不少于 20；文件名或内容包含 `smaps` / `leakstack` / `sample` 且有有效进程名或内存名。否则丢弃。
  * **系统字段映射**: 系统日志统一写入 `eventType = SYSTEM`，`filePath = file.relativePath`，`rawEventName = 事件名或规则 ID`，`processName = PROCESS_NAME / 日志 tag / 模块名 / 文件来源`，`detail = 200-300 字短摘要`，`detailRows` 保存系统日志类型、来源文件、规则 ID、时间和片段摘要等结构化字段。
  * **系统结果语义**: 日志管理不维护全局结果字段；系统日志关键词仅用于识别关键行，不再映射或展示成功、失败、告警等全局结果类型。
  * **系统时间优先级**: 日志内容完整时间 -> 文件名时间 -> 段落标题时间 -> 内容中的 `MM-dd HH:mm:ss(.SSS)` 按兜底年份补齐 -> `fileIo.stat(file.path).mtime` 归一化毫秒 -> 当前导入时间。文件名时间兼容已有崩溃日志的 `yyyyMMddHHmmssSSS`，并补充系统日志常见的 `yyyyMMddHHmmss`；`mtime` 进入业务前必须统一为毫秒，避免秒级时间戳被当作毫秒写入后触发过期清理。
  * **系统无年份时间补齐**: `updater_log`、`updater_stage_log`、`sys_installer.log` 等关键行日志常见 `MM-dd HH:mm:ss(.SSS)` 格式，解析时使用兜底时间所在年份补齐；若补齐后的时间明显晚于兜底时间，则回退到上一年，避免跨年日志排序到未来。
  * **系统确定性 ID**: 系统日志导入 ID 基于 `eventType + rawEventName/ruleId + timestamp + normalizedDetail` 生成；来源压缩包和来源文件路径不参与 hash，避免相同内容不同来源重复入库；时间参与 hash，保留同类系统事件在不同时间的多次发生。
  * **系统内存与调度策略**: `SystemLogImportRule` 按块/按行读取文件，分段时只保留当前段摘要和必要字段，不把大文件全文写入 `detail`；解析大文件按固定行数批次让出事件循环。
  * **导入审计副作用**: 导入 zip、解压、读取 `faultlogger` 和 `hilog` 文件可能触发系统文件操作审计事件；该副作用属于系统审计链路的真实事件，本模块不屏蔽导入目录产生的文件操作日志。
  * **导入 IO 异常边界**: 离线导入中的 `fileIo`、`zlib`、gzip 和 UTF-8 解码调用必须在导入层显式捕获并转换为导入规则失败或解压失败结果；文件关闭或 gzip 关闭失败只记录 `warn`，不得覆盖正在传播的读取、解压或解析失败。分块读取 UTF-8 文本时必须使用流式解码，避免多字节字符跨 chunk 边界后产生替换字符。
  * **gzip 解压目标策略**: 内核 `.gz` 文件解压到导入目录下的唯一临时文本文件，解析完成后清理该临时文件；禁止无校验复用同名明文文件，避免 zip 同时包含 `.gz` 和同名文本时解析错源。
  * **导入分类顺序**: 后台解析任务先对规则解析得到的记录按确定性 ID 做批内去重，并返回批内重复数量；主线程收到解析结果后继续按“过期跳过 -> 库内重复 -> 新增”分类。过期判断使用当前日志存储配置的 `retentionDays` 和 `LogStorageMaintenanceService.buildLogStorageNaturalDayCutoff()` 的自然日 cutoff；库内重复调用 `LogEntryRepository.hasEntry(id)` 判断。
  * **过期处理**: `timestamp < cutoff` 的离线日志在入库前跳过，不调用 `LogStorageMaintenanceService.appendEntries()`，不计入新增，计入 `skippedExpiredCount`。该口径避免导入服务先计新增、随后维护服务按留存策略删除而导致 UI 新增数量虚高。
  * **重复处理**: 后台解析任务先按 ID 做批内去重；导入服务收到去重后的解析结果后过滤过期记录，再调用 `LogEntryRepository.hasEntry(id)` 过滤历史已存在记录；重复记录不调用入库链路，不更新数据库。
  * **单测口径**: `LogImportService` 单测聚焦 Picker 取消/成功、后台解析结果持久化、批内重复统计透传、过期跳过、库内重复跳过和入库失败统计；规则解析行为由 `KernelLogImportRule`、`CrashLogImportRule`、`SystemLogImportRule` 等规则单测，以及后台任务/实机验收覆盖。
  * **结果反馈**: 导入结束后返回解压文件数、扫描文件数、识别条数、新增条数、重复跳过条数、过期跳过条数、丢弃条数和失败条数，并刷新列表与统计；UI 成功提示只展示“日志添加成功”，不展示统计明细。导入统计明细继续通过后台日志输出，至少保留新增、重复跳过、过期跳过、丢弃和失败等诊断字段，便于后续排障。导入 demo 包后，系统日志应能留存 `system_updater_log`、`system_updater_stage`、`system_installer` 等关键行类记录。
* **文件事件前缀过滤器设计**:
  * **挂载事件**: 挂载到 `FILE` 类型对应的全部审计事件，包括既有 `FILE`、`FILE_INTERCEPTED`、`KIA_CREATE`、`KIA_READ`、`KIA_VARIANT`、`KIA_INTERCEPT`，以及首批 API26 已验证文件事件 `FILE_CREATE(0x1C001104)`、`FILE_CLOSE(0x1C001106)`、`FILE_COPY(0x1C001109)`、`FILE_SETOWNER(0x1C00110A)`、`FILE_SETMODE(0x1C00110B)`、`FILE_SETEXTATTR(0x1C00110C)`、`FILE_DELETEEXTATTR(0x1C00110D)`。
  * **过滤器类型**: `securityAudit.FilterType.FILE_PATH_PREFIX`。
  * **过滤器语义**: `isInclude = true`，只接收指定路径前缀下的文件审计事件。
  * **前缀来源**: `LogAuditSource.prepareFileFilterContext()` 获取前台系统用户 ID，再调用 `LogPathUtils.buildHmdfsAccountFilesPrefix(userId)`。
  * **前缀格式**: `/data/service/el2/{foregroundUserId}/hmdfs/account/files/`。
  * **用户 ID 获取**: `osAccount.getAccountManager().getForegroundOsAccountLocalId()`；获取失败时回退 `DEFAULT_FOREGROUND_USER_ID = 100`。
  * **典型值**: 当前用户为 `100` 时，过滤前缀为 `/data/service/el2/100/hmdfs/account/files/`。
  * **挂载顺序**: 每个文件审计事件先 `client.subscribe([eventId])`，再 `client.addFilter(eventId, filter)`；若过滤器挂载失败，会立即取消该事件订阅并将该事件计入失败列表。
  * **失败策略**: 文件过滤上下文不可用、客户端无 `addFilter`、或系统返回 `FILE_FILTER_UNSUPPORTED_CODE = 1012000005` 时，该文件事件不进入有效订阅；若仍有其他事件成功，整体返回 `partial`。

## 5. 异常处理与系统依赖 (Dependencies & Errors)
- **关键系统 API**:
  - `@kit.DeviceSecurityKit.securityAudit`: 系统审计事件客户端、事件订阅、事件取消订阅和文件前缀过滤器。
  - `@kit.BasicServicesKit.osAccount`: 获取前台系统用户 ID，用于构造 HMDFS 文件审计前缀。
  - `@ohos.data.relationalStore`: 本地 RDB 存储。
  - Ability `common.UIAbilityContext`: 初始化存储和导出文件时使用。
- **系统权限**:
  - `ohos.permission.QUERY_AUDIT_EVENT`: 订阅系统审计事件。
  - `ohos.permission.GET_LOCAL_ACCOUNTS`: 获取前台系统用户 ID。
  - `ohos.permission.PREPARE_APP_TERMINATE`: 应用终止前准备能力。
  - 其他企业管理权限以 `entry/src/main/module.json5` 当前声明为准，不在日志管理模块内独立扩展。
- **异常兜底策略**:
  - 审计客户端不可用：`LogAuditSource.subscribe()` 返回 `unsupported`，采集器进入降级态。
  - 审计权限不足或订阅失败：返回 `permission_denied`，记录错误码，保留其他可用采集源。
  - 部分审计事件失败：返回 `partial`，`failedEventIds` 记录失败事件，成功事件继续工作。
  - 文件前缀过滤器不可用：文件事件不保留裸订阅，避免接收未限定路径范围的文件审计事件。
  - 审计日志事件订阅部分失败：`AUDIT` 对应的其它 `NotifyEvent` 沿用逐事件订阅策略，单个事件订阅失败只计入 `failedEventIds`；至少一个审计或崩溃来源可用时采集器进入 `partial` 降级态并保留成功事件。
  - 前台用户 ID 获取失败：回退到用户 `100` 构造前缀。
  - 审计事件解析失败：丢弃无法映射为 `LogEntry` 的事件，不写入存储。
  - 扩展 JSON 解析失败：返回空扩展字段，保留主记录展示。
  - 初始化失败：返回 `storage_init_failed` 或 `query_failed`，页面保持未初始化态并记录错误日志。
  - 查询刷新失败：返回 `query_failed`，页面保持现有状态并记录错误日志。
  - 分页刷新失败：返回 `query_failed`，保留最近一次成功分页列表和页码状态，并记录错误日志，不向 UI 事件回调抛出未处理异常。
  - 时间范围筛选为空结果：保持正常查询成功状态，列表展示“无匹配结果”空态，不视为异常。
  - 开机日志写入失败：记录 `appendBootEntry failed` 和维护服务错误码，不阻断关机事件订阅和应用启动。
  - 关机事件 marker 写入失败：记录 `markPendingShutdown failed`，仍尝试写入关机日志；重启后可能无法做关机补偿。
  - 关机日志写入失败：保留 pending marker 且 `pending_shutdown_logged = 0`，下次启动按 marker 时间补写关机日志。
  - 关机补偿写入失败：不清理 pending marker，保留给下次启动继续补偿。
  - 清理失败：返回 `clear_failed`，不主动清空 UI 列表。
  - 导出无数据：返回空数据结果，由 UI 给出提示。
  - 导入 Picker 取消：返回取消结果，页面保持当前状态。
  - 导入非 zip 外层文件：返回非法压缩包结果，不解压、不入库。
  - 导入解压失败：返回解压失败结果，保留已选目录中已写入文件但不写入日志数据库。
  - 导入规则解析失败：单文件或单行失败计入失败或丢弃统计，不影响其它文件继续导入。
  - 导入过期记录：按当前留存配置的自然日 cutoff 在入库前跳过，计入过期跳过统计，不进入维护服务入库链路。
  - 导入重复记录：按确定性 ID 跳过，不更新已有数据库记录。
  - 设置保存失败：保持脏状态，允许用户重试。

### 5.1 实施步骤与测试验收 (Implementation & Acceptance)

- **实施步骤**:
  - 涉及采集源、过滤条件、分页、详情、清理、留存配置、导出、RDB schema 或权限变化时，先更新本文档的目标范围、数据模型、状态流、职责边界、异常兜底和验收口径。
  - 涉及离线导入格式、规则端口、内核规则、hash 去重、压缩处理或导入结果口径变化时，先更新本文档。
  - 数据结构和枚举先同步 `LogModels.ets`、`LogSourcePorts.ets`、相关字符串和筛选常量，再调整采集、仓储或页面。
  - 系统审计、崩溃、生命周期等输入先落在 Source / Collector / Maintenance 边界，再由 Repository 和 ViewModel 统一提供页面状态；页面不得直接解析审计事件或访问 RDB。
  - 新增或调整 `securityAudit.NotifyEvent` 映射时，必须先更新 `LogAuditEventIds.ets` 的事件 ID 清单和说明映射，再更新 `LogAuditSource.ets` 订阅映射及 `LogAuditEventMapper.ets` 详情行。
  - 开关机日志调整时，优先保证启动日志直接入库；关机 common event 仅负责关机日志和补偿 marker，不得把启动日志完全依赖上次关机 marker。
  - RDB 字段、索引、扩展 JSON 或留存策略变化时，需要同步清理策略、导出字段和历史数据处理口径。
  - 新增或调整审计权限时，同步 `entry/src/main/module.json5`、`hapsigner/UnsgnedDebugProfileTemplate.json` 并按签名流程重新生成 p7b。
- **测试覆盖**:
  - UT: `entry/src/test/log-manage/*.test.ets` 覆盖采集源、审计映射、仓储查询、留存配置、导出、ViewModel 状态和异常分支。
  - ohosTest: `entry/src/ohosTest/ets/test/simple/RouteAction.test.ets` 通过 `route_action` 场景覆盖日志管理入口和基础页面可达性。
  - E2E: `scripts/e2e/cases/logs/` 下 `page_details.json`、`filters_visible.json`、`stats_visible.json`、`storage_settings_visible.json`、`export.json`、`event_after_policy_change.json` 覆盖查询、筛选、统计、存储配置、导出和联动事件。
- **验收口径**:
  - 审计能力不可用或权限不足时，采集器进入可解释降级态，页面仍可查看已有日志和其它来源日志。
  - 官方 `securityAudit.NotifyEvent` 中当前项目未归类的通知事件应订阅并映射为 `AUDIT`，列表筛选中展示为“审计日志”；详情页只展示“事件说明”和“事件原始JSON”两类结构化信息。
  - 企业设备管理事件详情在保留“事件说明”和“事件原始JSON”的基础上，若原始 JSON 存在 `objectInfo`，必须展示“操作对象”；外设管理已实机确认的 `disable_usb`、`usb_read_only`、`disabled_bluetooth`、`disable_wifi`、`disabled_printer`、`disable_microphone`、`disable_camera`、`disabled_hdc`、`disallowed_tethering`、`get_network_interfaces`、`disallowed_usb_devices`、`allowed_usb_devices`、`set_switch_status` 必须展示对应的中性中文“操作备注”，不得从策略对象名推断已经执行启用或禁用。`disallowed_usb_devices` 展示为“USB 禁止设备类型策略”，`allowed_usb_devices` 展示为“USB 设备信任列表”；缺少公开语义依据的 `disallowed_permissive_usb_devices` 必须回退展示原始对象值，不生成中文备注。若原始 JSON 存在 `type`，必须展示“操作类型”，格式为 `type=0`、`type=1` 等原始值；仅当 `type=0` 时展示“操作类型备注：读取/查询策略状态”，仅当 `type=1` 时展示“操作类型备注：设置/变更策略状态”，其它未知值只展示原始操作类型。实机样本中无有效值的 `targettInfo`、`extra` 和固定为 `0` 的 `subType` 不生成占位详情行。EDM 新入库日志的 `LogEntry.detail` 必须使用详情结构化字段生成列表摘要，优先展示“操作备注”和“操作类型备注”，未知对象或未知 `type` 分别回退到 `objectInfo` 和 `type=N`；摘要不得拼接“事件原始JSON”。该增量展示不得改变 RDB schema、审计订阅范围或历史日志迁移口径，也不得将 `type=1` 展示为“用户手动操作”。
  - `FILE`、`ACCOUNT`、`PERMISSION`、`CRASH` 已有映射不得因新增 `AUDIT` 发生变化，未知且未列入订阅清单的裸事件 ID 仍不入库。
  - 文件操作日志的 `LogEntry.filePath` 与详情页“文件路径”必须来自同一文件审计路径解析结果；重命名事件必须展示目标/新路径作为“文件路径”，并额外展示“原文件路径”和“新文件路径”。样机中 `file_newpath=/data/service/el2/100/hmdfs/account/files/Docs/Download/333.txt`、`file_oldpath=/data/service/el2/100/hmdfs/account/files/Docs/Download/新建 文本文档.txt`、`file_path=''` 的事件，详情必须展示 `/storage/Users/currentUser/Download/333.txt`。
  - 已验证 API26 文件事件必须全部归类为 `LogEventType.FILE`，并复用 `FILE_PATH_PREFIX` 用户目录过滤；过滤上下文不可用、`addFilter` 不可用或挂载失败时，不得裸订阅该事件。
  - `FILE_SETMODE` 详情必须展示 `FILE_SETMODE：文件权限模式变更事件`；日志存在 `file_newmode` 时必须展示新文件权限。`FILE_SETOWNER`、`FILE_SETEXTATTR`、`FILE_DELETEEXTATTR` 等事件的已验证字段同样遵循“有值才展示、无值不占位”的结构化详情规则。
  - `FILE_COPY` 详情必须以 `file_newpath` 作为文件路径，并保留 `file_oldpath`。
  - 筛选、分页、详情和统计之间状态一致；分页失败保留最近一次成功列表，不向 UI 抛出未处理异常。
  - 留存配置保存失败保持脏状态，导出无数据给出明确提示，清理失败不主动清空 UI 列表。
  - RDB 查询、计数或统计失败时，刷新流程返回 `query_failed`，不得把异常查询结果展示为 0 条空列表。
  - 选择 zip 后，导入服务能自动使用应用私有临时工作根目录处理外层 zip、嵌套 zip 和内部 gz；导入命中内核规则的 `KERNEL` 记录，并将 `remoteLog/faultlogger` 下 appfreeze/jscrash/cppcrash 文件一文件一条导入为 `CRASH` 记录；UTF-8 中文内容跨读取块边界时不得乱码。
  - 导入服务不再弹出目录 Picker；导入成功、解析失败、入库失败或异常退出后，必须清理 `context.filesDir/log_import_tmp` 临时工作根目录；清理失败只记录错误日志，不覆盖导入主结果；下次导入前必须清理历史残留目录。
  - 重复导入同一日志包时，已有记录不更新，导入结果以重复跳过计数体现；已超过当前留存自然日 cutoff 的记录不入库，导入结果以过期跳过计数体现，新增条数只统计通过过期和重复过滤后提交入库的记录。
  - `SYSTEM` 类型可展示和筛选，导入 demo zip 后能看到命中白名单路径的系统日志记录；`remoteLog/hilog/hilog.*` 不产生系统日志导入数据。
  - 时间范围筛选默认“全部时间”不追加时间查询条件；选择最近 1 小时、最近 12 小时、最近 24 小时或最近 3 天时，列表、分页、过滤后数量、详情保留判断和导出结果均只包含 `timestamp >= startTimestamp` 的日志。
  - 应用冷启动后，即使没有上次关机 pending marker，也应产生一条 `POWER` / `设备启动` 日志；存在上次 pending marker 时，先完成关机补偿判断，再产生本次启动日志。
  - 收到 `usual.event.SHUTDOWN` 后，应先写入 pending marker，再写入 `POWER` / `设备关闭` 日志；若关机日志入库失败，下次启动保留补偿机会。
