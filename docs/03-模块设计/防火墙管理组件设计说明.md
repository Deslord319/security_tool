## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 为设备上所有可用系统用户提供防火墙开关、预设模式、自定义规则和用户级默认策略管理能力。
- **入口路由 (Route Entry)**:
  - Shell Entry: `pages/MainPage`
  - Route ID: `firewall`、`firewall-rules`
  - View Entry: `entry/src/main/ets/views/firewall/overview/FirewallPage.ets`、`entry/src/main/ets/views/firewall/rules/FirewallRulesPage.ets`
- **对外暴露能力 (Public APIs)**:
  - `entry/src/main/ets/services/firewall/FirewallService.ets`: 防火墙模块业务编排入口。
  - `entry/src/main/ets/services/firewall/providers/SystemUserProvider.ets`: 防火墙可用用户列表与用户策略摘要读取入口。
- **业务边界**:
  - ✅ **包含**: 全用户防火墙开关、`public/private/custom` 首页模式切换、自定义规则增删改查、规则冲突检测、规则用户作用域、用户级 `allowlist/denylist` 默认策略下发、API 26 系统账号枚举、账号删除后的本地数据清理，以及账号新增后 `public/private` 当前模式对最新账号集合的补下发、`custom` 当前模式对新增账号的默认 policy 同步。
  - ❌ **不包含**: 系统防火墙能力本身实现、账号管理 UI、PIN 认证规则实现、测试框架实现、签名工具实现。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `FirewallOverviewViewModelState.enabled: boolean`: 管理员/当前用户的防火墙开启状态。当前部署口径下应用只面向管理员用户安装，管理员/当前用户的 policy 状态即首页总开关展示状态，初始值 `false`。
  - `FirewallOverviewViewModelState.currentMode: FirewallPresetMode`: 当前首页模式，取值 `public | private | custom`，默认 `custom`。
  - `FirewallOverviewViewModelState.initialized/isLoading/isToggleProcessing/isModeProcessing: boolean`: 首页初始化、刷新、开关和模式切换处理状态；`isModeProcessing` 为首页模式切换互斥锁，期间不允许触发总开关、其它模式卡片或自定义规则入口。
  - `FirewallRulesViewModel.currentMode: FirewallPresetMode`: 规则页接收到的当前模式；非 `custom` 时不允许编辑规则。
  - `FirewallRulesViewModel.rules: NetFirewallRule[]`: 规则页展示列表，由本地 `FirewallRuleIntent` 和成功下发的 `FirewallRuleDeployment` 回读生成。
  - `FirewallRulesViewModel.defaultPolicyItems: FirewallDefaultPolicyDisplayItem[]`: 规则页统计区默认策略数据源，按用户读取 `allowlist/denylist/custom/unknown` 和入站/出站默认动作，页面用与入站/出站规则相同的静态摘要卡片聚合展示白名单用户、黑名单用户及必要异常摘要。
  - `FirewallRulesViewModel.conflictNotice/currentTab/userOptions/lastSaveResult/lastOperationMessage`: 规则冲突提示、列表分组、可选用户、最近保存结果和操作提示。
  - `MainPage.lastFirewallRoute: string`: 防火墙模块内最近一次子路由，取值 `firewall | firewall-rules`，用于从侧边栏、总览快捷入口或顶部菜单再次进入防火墙模块时恢复上次所在页面。
  - `FirewallUserDispatchViewModel.users/selectedUserId/selectedModeIndex/statusHintText/lastApplyResult`: 模式下发弹窗的用户列表、当前用户、目标模式、状态提示和最近下发结果。
  - `FirewallLocalRepository.desiredEnabled/lastAppliedMode/lastAppliedUserIdsSignature`: 防火墙期望开关状态、最近一次成功处理账号集合的模式和账号集合签名，用于账号新增后判断 `public/private` 当前模式是否需要补下发；`custom` 模式账号级默认 policy 同步成功后也必须更新该签名，表示当前账号快照已完成处理。
- **关键流转路径**:
  - `进入 MainPage` -> `FirewallOverviewViewModel.initialize` -> `FirewallService.loadOverviewState` -> `SystemUserProvider + FirewallSystemRepository + FirewallLocalRepository` -> `首页状态同步`
  - `点击首页总开关` -> `FirewallOverviewViewModel.requestToggle` -> `FirewallService.setFirewallEnabledForAllUsers` -> `PIN 认证 + 全用户 setPolicy(isOpen)` -> `成功后更新 enabled`；如果模式切换正在执行，ViewModel 直接拒绝本次开关请求。
  - `切换首页模式` -> `FirewallOverviewViewModel.requestModeSwitch` -> `FirewallService.switchMode` -> `切换前快照当前模式/本地映射/用户 policy/系统规则 + 清系统规则 + 按模式写 policy 动作但保留 isOpen + 清 deployments + 生成并下发规则 + 全部成功后保存模式` -> `回读首页状态并同步 currentMode/enabled`；任一步骤失败时不保存目标模式，并尝试恢复快照；如果总开关正在执行，ViewModel 直接拒绝本次模式切换请求。
  - `进入规则页` -> `MainPage` 从 `ApplicationRuntimeManager` 取得共享 `FirewallRulesViewModel` 并注入页面 -> `FirewallRulesViewModel.initialize/loadRules` -> `SystemUserProvider.loadAvailableUserIds + FirewallService.loadUserPolicyState` 读取全局默认策略 -> `FirewallService.listRulesForDisplay` -> `FirewallLocalRepository.loadRuleIntentMappingData` -> `展示默认策略摘要、规则和冲突提示`
  - `规则页切到其它模块后再次进入防火墙` -> `MainPage` 根据 `lastFirewallRoute` 恢复到 `firewall-rules` -> `复用 ApplicationRuntimeManager 持有的共享 FirewallRulesViewModel.currentTab` -> `保留上次退出规则页时的分组 Tab` -> `重新加载规则列表和冲突提示`
  - `规则页点击返回防火墙首页` -> `MainPage.setCurrentPage(firewall)` 显式更新 `lastFirewallRoute` -> 后续再进入防火墙模块默认展示首页
  - `打开新增/编辑规则弹窗` -> `FirewallRulesViewModel.prepareRuleEditor` -> `SystemUserProvider.loadAvailableUserIds + FirewallService.listRulesForDisplay` 回读编辑所需账号列表、规则列表和冲突提示，不读取全局默认策略 -> `AddRuleDialog` 按最新用户选项初始化作用域。
  - `新增/编辑规则提交` -> `FirewallRulesViewModel.validateRuleUserScopeForSubmit` 轻量回读当前账号 ID 并校验规则目标用户仍存在 -> `FirewallRulesViewModel` -> `FirewallService.createRule/updateRule` -> `FirewallRuleUtils + FirewallSystemRepository + FirewallLocalRepository` -> 成功后 `refreshPageData` 回读账号、默认策略、规则列表和冲突提示，并保留操作提示。
  - `删除规则` -> `FirewallRulesViewModel` 记录当前删除中的 `localRuleId` -> `FirewallService.deleteRule` -> `FirewallSystemRepository + FirewallLocalRepository` -> 成功后 `refreshPageData` 回读账号、默认策略、规则列表和冲突提示，并保留操作提示；删除过程中不切换规则列表 loading 空间，只在对应规则卡片展示删除中状态。
  - `打开模式下发弹窗` -> `FirewallUserDispatchViewModel.initialize` -> `SystemUserProvider.loadAvailableUserIds` 读取可用用户和系统账号名 -> 非空时触发本地无效用户清理 -> `FirewallService.loadUserPolicyState` -> `优先按本地历史模式回显；无历史模式时再按系统默认动作推导用户模式`
  - `提交用户模式` -> `FirewallUserDispatchViewModel.applySelectedMode` -> 提交前通过 `SystemUserProvider.loadAvailableUserIds` 重新校验目标用户仍存在 -> `FirewallService.applyUserPolicyMode` -> `PIN 认证 + policy-only setPolicy + 保存 userId -> mode` -> `重新读取用户策略状态`
  - `账号新增/删除` -> `EnterpriseAdminAbility` 保留保活注册或移除，并通知公共 `AccountChangeCoordinator` -> 协调层防抖后通过 `SystemUserProvider.loadAvailableUserIds` 全量读取最新账号；`account-added` 场景必须确认快照包含触发账号 ID，若尚不可见则在约 1 秒窗口内短间隔重试，超时仍不可见时不更新上一轮账号签名、不分发 Handler、不发布账号 CommonEvent -> 稳定快照按注册顺序分发给账号变化 Handler；防火墙 Handler 在非空账号列表下清理无效用户本地数据；当前模式为 `public/private` 且账号集合签名变化时，按最新账号集合补下发当前模式 policy 和预置规则；当前模式为 `custom` 时同步账号级默认 policy，但不自动扩展规则作用域，同步成功后保存 `custom` 和当前账号集合签名；稳定账号事实通过账号 CommonEvent 通知 UI 进程，UI 进程由 `AccountRuntimeService` 记录最新账号 payload 并触发已初始化/已加载的防火墙 ViewModel 重新读取完整账号列表和页面数据，`MainPage` 不再按 route 分发账号变化。

## 3. 核心功能场景 (Core Functional Scenarios)

### 3.1 首页模式切换

- **业务目标**: 用户在公共网络、私有网络、自定义模式之间切换时，保证模式卡片状态、系统 policy、系统规则和本地模式记录保持一致；失败时不得把目标模式保存为当前模式。
- **入口/触发**: 防火墙首页点击公共网络、私有网络或自定义模式卡片；`FirewallService.switchMode(context, targetMode)`。
- **涉及能力**: `FirewallOverviewViewModel`、`FirewallService`、`FirewallModeSwitchService`、`FirewallPolicyService`、`FirewallModeStrategy`、`FirewallSystemRepository`、`FirewallLocalRepository`、系统 `netFirewall.setNetFirewallPolicy/addNetFirewallRule/removeNetFirewallRule`、本地 Preferences。

| 条件/分支 | 前置状态 | 处理规则 | 预期表现 | 状态/持久化影响 | 覆盖要求 |
|---|---|---|---|---|---|
| 切换到公共网络 | 防火墙处于开启状态；当前模式不是公共网络 | 执行模式切换事务；按公共网络策略更新系统 policy 和规则 | 首页选中公共网络；模式摘要与公共网络一致 | 保存当前模式为公共网络；系统规则与公共网络策略一致；旧 deployments 被替换 | UT 覆盖成功链路；手工或 E2E 覆盖首页切换 |
| 切换到私有网络 | 防火墙处于开启状态；当前模式不是私有网络 | 执行模式切换事务；按私有网络策略更新系统 policy 和规则 | 首页选中私有网络；模式摘要与私有网络一致 | 保存当前模式为私有网络；系统规则与私有网络策略一致；旧 deployments 被替换 | UT 覆盖成功链路；手工或 E2E 覆盖首页切换 |
| 切换到自定义模式 | 防火墙处于开启状态；当前模式不是自定义模式 | 执行模式切换事务；重新下发本地自定义规则；切换成功后进入规则页 | 首页先完成自定义模式切换，随后进入规则页 | 保存当前模式为自定义模式；本地 rule intents 不变；deployments 更新为最新下发结果 | UT 覆盖 custom intent 下发；手工或 E2E 覆盖规则页联动 |
| 模式切换失败 | 防火墙处于开启状态；用户发起模式切换；系统或本地任一步骤失败 | 不保存目标模式；尝试恢复切换前 policy、系统规则、deployments 和模式；完成后回读系统和本地状态 | 页面提示切换失败；首页展示回读后的实际模式和开关状态 | 目标模式不作为持久化真相；旧状态尽力恢复；恢复失败时以后续回读为准 | UT 覆盖失败回滚；手工关注失败提示 |

### 3.2 自定义防火墙规则新增

- **业务目标**: 在自定义模式下新增防火墙规则时，保证用户提交、系统规则下发、本地规则意图和本地 deployments 保持一致；失败时不得产生半提交状态。
- **入口/触发**: 防火墙规则页点击新增规则并提交；`FirewallService.createRule(context, input)`。
- **涉及能力**: `FirewallRulesViewModel`、`FirewallService`、`FirewallRuleMutationService`、`FirewallRuleDeploymentService`、`FirewallRuleUtils`、`FirewallSystemRepository`、`FirewallLocalRepository`、系统 `netFirewall.addNetFirewallRule`、本地 Preferences。

| 条件/分支 | 前置状态 | 处理规则 | 预期表现 | 状态/持久化影响 | 覆盖要求 |
|---|---|---|---|---|---|
| 打开新增规则弹窗 | 当前模式为自定义模式；规则页可能已打开过，系统设置中可能新增或删除账号 | 打开弹窗前调用 `prepareRuleEditor`，只刷新 `userOptions`、`rules` 和 `conflictNotice`；不读取全局默认策略 | 用户作用域下拉展示最新系统用户；规则冲突判断基于最新规则列表 | 仅刷新编辑上下文；不新增本地规则，不调用系统防火墙写接口 | UT 覆盖弹窗准备阶段刷新账号和规则上下文；手工关注系统设置新增用户后重新打开新增规则 |
| 打开新增规则弹窗时无可用账号 | 当前模式为自定义模式；账号读取失败或返回空用户集合 | 不打开新增规则弹窗，提示无可用用户或读取失败 | 用户不会进入无法提交的空账号规则表单 | 不改变规则、本地映射或系统防火墙状态 | UT 覆盖无账号时拒绝打开编辑器 |
| 提交新增规则前账号已变化 | 弹窗打开后系统账号被删除；规则目标用户包含已不存在账号 | 提交前重新读取当前账号 ID；发现目标用户不存在时提示重新选择用户作用域，不进入冲突判断和系统下发 | 页面提示账号列表已变化，用户需重新选择作用域后再提交 | 不新增 intent/deployment；不调用系统防火墙 API | UT 覆盖提交前账号删除拒绝提交 |
| 自定义模式下新增规则成功 | 当前模式为自定义模式；规则页允许编辑；输入合法；未命中本地冲突；系统下发和本地保存均成功 | 构造本地 rule intent，按目标用户下发系统规则；全部成功后保存 intent 和 deployments | 页面提示新增成功，规则列表刷新并展示新规则 | 新增本地 intent；保存每个目标用户的 deployment；回读后规则与系统一致 | UT 覆盖成功链路；手工或 E2E 覆盖页面新增 |
| 非自定义模式下尝试新增规则 | 当前模式为公共网络或私有网络 | 不允许编辑自定义规则；页面入口禁用或 ViewModel/Service 拒绝提交 | 用户不能新增自定义规则，页面保持当前规则展示 | 不新增 intent；不新增 deployment；不调用系统新增规则 | UT 覆盖非自定义模式拒绝；手工验收入口禁用 |
| 输入不合法 | 用户提交非法名称、目标用户、协议、端口、IP、域名、主 DNS 或备用 DNS | 提交前拦截，不调用系统 API | 表单展示校验提示 | 系统规则、本地 intent 和 deployments 均不变 | UT 覆盖规则校验；手工验收表单提示 |
| 命中本地规则冲突 | 新规则与现有自定义规则存在重复、硬冲突或重叠冲突 | 展示冲突提示；用户未确认替换前不提交普通新增 | 用户看到冲突说明，可按页面流程取消或进入替换 | 默认不改变系统规则、本地 intent 或 deployments | UT 覆盖冲突判断；手工验收冲突提示 |
| DNS 规则重复 `29400007` | 前端本地未拦截，但系统 `addNetFirewallRule` 返回 `29400007` | 将错误映射为 DNS 规则重复；终止新增；回滚本次已成功下发的系统规则 | 页面提示 DNS 规则重复，新增失败 | 不保存目标 intent/deployment；已下发系统规则删除回滚 | UT 覆盖错误码映射和回滚；设备测试或手工关注系统返回 |
| 新增提交失败 | 系统下发失败、本地 intent/deployment 保存失败，或多目标用户部分下发失败 | 记录失败结果；回滚已下发系统规则；不保存本地规则 | 页面提示新增失败，可展示失败原因或失败明细 | 不产生半提交规则；系统和本地状态尽力恢复到新增前 | UT 覆盖部分成功回滚和本地保存失败 |

### 3.3 自定义防火墙规则编辑

- **业务目标**: 在自定义模式下编辑防火墙规则时，保证规则类型变化、用户作用域变化、系统规则更新/新增/删除和本地 intent/deployments 提交保持一致；失败时保留或恢复旧规则。
- **入口/触发**: 防火墙规则页点击编辑规则并提交；`FirewallService.updateRule(context, localRuleId, input)`。
- **涉及能力**: `FirewallRulesViewModel`、`FirewallService`、`FirewallRuleMutationService`、`FirewallRuleDeploymentService`、`FirewallRuleUtils`、`FirewallSystemRepository`、`FirewallLocalRepository`、系统 `netFirewall.updateNetFirewallRule/addNetFirewallRule/removeNetFirewallRule`、本地 Preferences。

> 编辑事务按规则类型和用户集合差量执行：
> - 保留用户：编辑前后都存在的用户。
> - 新增用户：编辑后新增的用户。
> - 移除用户：编辑后删除的用户。
> - 编辑前后均为非 DNS 时，保留用户调用 `updateNetFirewallRule`。
> - 编辑前或编辑后为 DNS 时，保留用户不调用 `updateNetFirewallRule`，统一走 `remove + add`。
> - 任一系统动作或本地保存失败时，不提交新 intent/deployments，并按已执行动作反向回滚。

| 条件/分支 | 前置状态 | 处理规则 | 预期表现 | 状态/持久化影响 | 覆盖要求 |
|---|---|---|---|---|---|
| 打开编辑规则弹窗 | 当前模式为自定义模式；规则页可能已打开过，系统设置中可能新增或删除账号 | 打开弹窗前调用 `prepareRuleEditor`，只刷新 `userOptions`、`rules` 和 `conflictNotice`；不读取全局默认策略 | 用户作用域下拉展示最新系统用户；冲突比较基于最新规则列表 | 仅刷新编辑上下文；不修改当前规则 intent/deployment | UT 覆盖弹窗准备阶段刷新账号和规则上下文 |
| 提交编辑规则前账号已变化 | 弹窗打开后系统账号被删除；编辑后的目标用户包含已不存在账号 | 提交前重新读取当前账号 ID；发现目标用户不存在时提示重新选择用户作用域，不进入冲突判断和系统下发 | 页面提示账号列表已变化，用户需重新选择作用域后再提交 | 不保存新 intent/deployment；不调用系统防火墙 API | UT 覆盖提交前账号删除拒绝提交 |
| 同类型非 DNS 编辑，仅修改规则内容 | 当前模式为自定义模式；规则存在；编辑前后类型均非 DNS；目标用户集合不变 | 对保留用户调用 `updateNetFirewallRule` 原地更新系统规则；全部成功后保存新 intent | 页面提示编辑成功，规则列表展示新内容 | 本地 intent 更新；deployments 的用户和 `systemRuleId` 保持不变 | UT 覆盖 IP->IP、DOMAIN->DOMAIN 原地更新 |
| 非 DNS 规则类型切换 | 当前模式为自定义模式；规则存在；编辑前后均非 DNS，但类型发生变化，例如 IP->DOMAIN、DOMAIN->IP | 对保留用户调用 `updateNetFirewallRule`，按新类型构造系统规则；全部成功后保存新 intent | 页面提示编辑成功，规则类型和展示字段更新 | 本地 intent 更新为新类型；保留用户 deployment 继续指向原 `systemRuleId` | UT 覆盖 IP->DOMAIN、DOMAIN->IP |
| 非 DNS 编辑时增加用户 | 编辑后目标用户包含新增用户，且类型非 DNS | 保留用户执行 update；新增用户执行 add；全部成功后保存新 intent 和新增 deployments | 页面提示编辑成功，规则列表展示更新后的用户范围 | 新增用户写入 deployments；保留用户 deployments 保持一致 | UT 覆盖保留用户 + 新增用户 |
| 非 DNS 编辑时删除用户 | 编辑后目标用户移除部分旧用户，且类型非 DNS | 保留用户执行 update；移除用户执行 remove；全部成功后保存新 intent，并清理移除用户 deployments | 页面提示编辑成功，被移除用户不再受该规则影响 | 移除用户 deployments 被删除；保留用户 deployments 保持一致 | UT 覆盖保留用户 + 删除用户 |
| 非 DNS 编辑时同时增加和删除用户 | 编辑后目标用户集合同时包含新增用户和移除用户，且类型非 DNS | 保留用户 update，新增用户 add，移除用户 remove；任一步失败则回滚已成功动作 | 页面提示编辑成功或失败；成功时用户范围完整更新 | 成功后 deployments 与新用户集合一致；失败时恢复旧 intent/deployments | UT 覆盖 add/update/remove 组合事务 |
| 非 DNS 切换为 DNS | 编辑后目标类型为 DNS，例如 IP->DNS、DOMAIN->DNS | 不调用 update；对保留用户先 remove 旧系统规则再 add 新 DNS 规则；新增用户 add；移除用户 remove | 页面提示编辑成功，规则展示 DNS 主备地址 | intent 更新为 DNS；保留用户 `systemRuleId` 刷新为新 add 返回值 | UT 覆盖 IP->DNS、DOMAIN->DNS |
| DNS 切换为非 DNS | 编辑前目标类型为 DNS，编辑后为 IP 或 DOMAIN | 不调用 update；对保留用户先 remove 旧 DNS 规则再 add 新非 DNS 规则；新增用户 add；移除用户 remove | 页面提示编辑成功，规则展示为新类型字段 | intent 更新为新类型；保留用户 `systemRuleId` 刷新为新 add 返回值 | UT 覆盖 DNS->IP、DNS->DOMAIN |
| DNS 编辑仍为 DNS | 编辑前后均为 DNS | 不调用 update；对保留用户 remove+add；新增用户 add；移除用户 remove | 页面提示编辑成功，主 DNS/备用 DNS 更新 | intent 更新；保留用户 deployments 刷新 `systemRuleId` | UT 覆盖 DNS->DNS、备用 DNS 变化 |
| 编辑后命中本地规则冲突 | 编辑后的规则与其它自定义规则存在重复、硬冲突或重叠冲突 | 系统调用前展示冲突提示；用户未确认替换前不提交普通编辑 | 用户看到冲突说明，可取消或进入替换流程 | 默认不改变系统规则、本地 intent 或 deployments | UT 覆盖冲突判断；手工验收冲突提示 |
| DNS 规则重复 `29400007` | DNS add 阶段系统返回 `29400007` | 映射为 DNS 规则重复；删除已 add 的新规则，add 回已 remove 的旧规则；不保存新 intent | 页面提示 DNS 规则重复，编辑失败 | 旧 intent/deployments 尽力恢复；新 intent 不落盘 | UT 覆盖错误码映射和 DNS 替换回滚 |
| 编辑提交失败 | update/add/remove、remove+add、本地保存任一步失败 | 回滚已成功系统动作；不提交新 intent/deployments；回滚 remove 后若产生新 `systemRuleId`，同步保存恢复后的 deployment | 页面提示编辑失败；规则列表回读后保持旧规则或恢复后的实际状态 | 新 intent 不落盘；旧 intent/deployments 保留或恢复 | UT 覆盖 update/add/remove 失败、本地保存失败 |

### 3.4 自定义防火墙规则删除

- **业务目标**: 在自定义模式下删除防火墙规则时，保证系统规则删除和本地 rule intent/deployments 清理一致；失败时不得丢失本地规则追踪。
- **入口/触发**: 防火墙规则页点击删除规则并确认；`FirewallService.deleteRule(context, localRuleId)`。
- **涉及能力**: `FirewallRulesViewModel`、`FirewallService`、`FirewallRuleMutationService`、`FirewallRuleDeploymentService`、`FirewallSystemRepository`、`FirewallLocalRepository`、系统 `netFirewall.removeNetFirewallRule/addNetFirewallRule`、本地 Preferences。

| 条件/分支 | 前置状态 | 处理规则 | 预期表现 | 状态/持久化影响 | 覆盖要求 |
|---|---|---|---|---|---|
| 自定义模式下删除规则成功 | 当前模式为自定义模式；规则存在；所有系统删除和本地清理均成功 | 逐个删除系统规则；全部成功后删除本地 intent 和 deployments；删除事务执行期间不驱动规则列表 loading 态，ViewModel 记录当前删除中的 `localRuleId` 供规则卡片展示局部动效 | 页面提示删除成功，规则列表移除该规则；删除过程中保留当前列表上下文，不展示“正在加载规则”，仅对应卡片显示删除中进度 | 系统规则被删除；本地 intent/deployments 被清理；删除结束后清空卡片级删除状态 | UT 覆盖删除成功、删除期间不置规则页 loading 和卡片级删除状态；手工或 E2E 覆盖页面删除 |
| 非自定义模式下尝试删除规则 | 当前模式为公共网络或私有网络 | 不允许删除自定义规则；页面入口禁用或 ViewModel/Service 拒绝提交 | 用户不能删除自定义规则 | 本地 intent/deployment 和系统规则不变 | UT 覆盖非自定义模式拒绝；手工验收入口禁用 |
| 规则不存在 | 本地找不到目标 `localRuleId` | 返回删除失败或规则已不存在结果，不调用系统删除 | 页面展示删除失败或规则已不存在提示 | 本地和系统状态不变 | UT 覆盖不存在规则 |
| 删除提交失败 | 系统 remove 失败、本地 intent/deployment 清理失败，或多目标用户部分删除失败 | 记录失败结果；add 回已删除的系统规则；保留本地 intent/deployments | 页面提示删除失败，可展示失败原因或失败明细 | 不丢失本地规则追踪；系统规则尽力恢复；恢复后 deployments 应与系统真实规则一致 | UT 覆盖部分删除失败、恢复失败和本地清理失败 |

### 3.5 账号变化后的防火墙策略同步

- **业务目标**: 系统账号新增或删除后，防火墙模块以最新账号集合为准保持系统 policy、系统规则和本地 Preferences 一致；`public/private` 当前模式需要补下发到新增账号，删除账号需要清理本地引用；`custom` 需要给新增账号同步账号级默认 policy，但不把旧自定义规则自动扩展到新增账号。
- **入口/触发**: `EnterpriseAdminAbility.onAccountAdded/onAccountRemoved` 触发 `AccountChangeCoordinator`；协调层防抖后读取真实系统账号；稳定账号事实通过账号添加/删除 CommonEvent 通知 UI 进程；UI 进程 `AccountRuntimeService` 更新账号状态并刷新已加载的防火墙页面态。防火墙页面进入、打开新增或编辑规则弹窗前、规则提交前校验、首页模式切换前仍按各自用例读取最新账号。
- **涉及能力**: `AccountChangeCoordinator`、`AccountChangeHandler`、`AccountChangeEventBus`、`SystemUserProvider.loadAvailableUserIds`、`FirewallAccountChangeHandler`、`FirewallModeSwitchService`、`FirewallLocalRepository.pruneUnavailableUsers`、本地 Preferences。

| 条件/分支 | 前置状态 | 处理规则 | 预期表现 | 状态/持久化影响 | 覆盖要求 |
|---|---|---|---|---|---|
| 公共网络模式新增账号 | 当前模式为 `public`，账号事件触发后最新账号集合包含新增用户 | 协调层全量读取最新账号；防火墙 Handler 非空账号下先 prune，再重放公共网络模式到最新账号集合；policy 的 `isOpen` 使用本地 `desiredEnabled` | 新账号获得公共网络默认 policy 和预置规则 | 更新 `lastAppliedMode=public` 和最新账号签名；系统规则覆盖最新账号集合 | UT 覆盖账号签名变化触发 public 补下发 |
| 私有网络模式新增账号 | 当前模式为 `private`，账号事件触发后最新账号集合包含新增用户 | 与公共网络一致，按私有网络配置生成预置规则并补下发 | 新账号获得私有网络默认 policy 和预置规则 | 更新 `lastAppliedMode=private` 和最新账号签名 | UT 覆盖 private 补下发 |
| 自定义模式新增账号 | 当前模式为 `custom`，已有自定义规则作用于旧账号集合 | 读取最新账号并执行 prune；按本地用户默认策略记录和 `desiredEnabled` 同步账号级 policy；不把新增账号加入旧 `targetUserIds`，不自动下发旧自定义规则 | 新账号获得自定义模式默认 policy，后续可在用户列表和默认策略展示中出现，旧规则作用域不变 | 本地 rule intents 不因新增账号扩展；不新增 deployments | UT 覆盖 custom 同步 policy 但不自动扩规则 |
| 多用户规则包含已不存在用户 | 本地 rule intent 的 `targetUserIds` 包含不在最新有效账号列表中的用户，且移除后仍有其它用户 | 从 `targetUserIds` 中移除无效用户；删除无效用户对应 deployments；不调用系统规则接口 | 规则刷新后该规则仍展示，但用户作用域不再包含无效用户 | 本地 intent 被裁剪；deployments 仅保留有效用户 | UT 覆盖 intent 裁剪和 deployment 过滤 |
| 规则仅作用于已不存在用户 | 本地 rule intent 的 `targetUserIds` 与最新有效账号列表无交集 | 删除整条本地 rule intent，并删除该规则所有 deployments；不调用系统规则接口 | 规则刷新后该规则消失 | 本地规则和下发映射均移除 | UT 覆盖空作用域规则删除 |
| 用户历史模式存在已不存在用户记录 | 本地 `firewall_user_policy` 中存在无效用户的 `allowlist/denylist` 记录 | 批量删除无效用户本地历史模式记录；系统 policy 由系统账号删除流程自行清理 | 后续用户模式回显不再出现无效用户历史记录 | 本地用户策略模式记录移除 | UT 覆盖用户模式记录批量裁剪 |
| 账号列表读取为空或失败 | `SystemUserProvider.loadAvailableUserIds` 返回空列表或错误空态 | 不执行本地 prune，不重放 `public/private` 模式，避免系统账号 API 临时失败时误删或误下发 | 页面显示无可用用户或对应失败态，本地规则仍保留 | 本地 intent、deployments、用户历史模式和模式签名不变 | UT 覆盖空有效账号列表不清理、不重放 |

### 3.6 规则页账号列表与弹窗数据源

- **业务目标**: 防火墙规则页只维护一份账号列表，账号新增/删除后由规则页 ViewModel 重读展示数据；新增/编辑规则弹窗和用户默认策略弹窗都使用规则页 `userOptions`，不各自重复调用系统账号 API。
- **入口/触发**: 规则页初始化、账号变化协调完成、规则页内新增/编辑/删除或用户默认策略操作完成。

| 场景 | 前置条件 | 系统行为 | 用户可见结果 | 数据状态变化 | 测试覆盖 |
|---|---|---|---|---|---|
| 账号变化时规则页已打开或规则 ViewModel 已加载 | 账号协调已发布账号 CommonEvent，UI 进程已持有共享 `FirewallRulesViewModel` | `AccountRuntimeService` 收到添加/删除事件后记录最新账号 payload，并在 UI 进程调用 `SystemUserProvider.loadAvailableUserIds()` 拉取完整账号列表，再触发 `FirewallRulesViewModel.refreshForAccountChange(context, users)`；ViewModel 更新 `userOptions` 后通过 `FirewallService.reconcileRulesForUsers(context, users)` 按账号 ID 快照对齐本地规则映射，再刷新 `defaultPolicyItems/rules` | 默认策略卡、规则列表、入站/出站计数和规则用户作用域文案同步变化；若规则页尚未加载，后续进入页面时走初始化读取最新账号 | `@Trace userOptions/defaultPolicyItems/rules` 重新赋值；本地 rule intents、deployments、用户历史模式不再引用已删除账号 | UT 覆盖账号列表刷新规则页数据和服务层规则映射对账 |
| 打开新增/编辑规则弹窗 | 规则页已持有最新 `userOptions` | 弹窗使用 `rulesViewModel.userOptions` 构造表单；不直接调用 `SystemUserProvider` | 用户作用域下拉展示规则页当前账号列表 | 弹窗不维护独立账号真相 | UT 覆盖 `prepareRuleEditor` 不重复读取系统账号 |
| 打开用户默认策略弹窗 | 规则页已持有最新 `userOptions` | `UserFirewallControlDialog` 接收 `userOptions` 并初始化 `FirewallUserDispatchViewModel`；弹窗 ViewModel 不直接调用 `SystemUserProvider` | 目标用户下拉展示规则页当前账号列表 | 默认策略弹窗消费规则页账号列表 | UT 覆盖用户策略弹窗 ViewModel 使用外部账号列表 |
| 连续账号事件 | 短时间内连续新增/删除账号，或同步运行中再次收到事件 | `AccountChangeCoordinator` 防抖合并事件；同一时间只运行一个同步任务；运行中收到新事件则排队再基于最新账号集合跑一轮 | 不出现并发清规则/下发规则互相覆盖 | 防火墙只按最终最新账号集合保存签名 | UT 覆盖 pending rerun 或防抖合并 |

## 4. 模块结构与组件设计 (Module Components)

### 【核心层】(Core MVVM & Domain Layers)

#### 4.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `FirewallPresetMode`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 首页预设模式类型。
    * **关键字段**: `public`、`private`、`custom`。
  * `FirewallUserPolicyMode`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 用户级默认策略模式。
    * **关键字段**: `allowlist` 映射为默认入站/出站拒绝，`denylist` 映射为默认入站/出站允许。
  * `FirewallRuleIntent`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 自定义规则的本地业务意图，是规则模板的长期真相。
    * **关键字段**: `localRuleId` (本地稳定身份), `targetUserIds` (固定用户集合), `type/direction/action` (规则匹配和冲突判断), `createdAt/updatedAt` (展示与排序辅助)。
  * `FirewallRuleDeployment`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 记录本地规则成功下发到系统后的映射。
    * **关键字段**: `localRuleId` (关联本地意图), `userId` (目标系统用户), `systemRuleId` (系统规则 ID)。
  * `FirewallDefaultPolicyDisplayItem`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 规则页只读展示用户级默认 policy，不作为可删除规则参与增删改、统计或冲突判断。
    * **关键字段**: `userId/label`, `mode` (`allowlist/denylist/custom/unknown`), `isOpen`, `inAction/outAction`。
  * `FirewallRuleIntentMappingData`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 防火墙自定义规则本地持久化结构。
    * **关键字段**: `version`, `ruleIntents`, `ruleDeployments`。
  * `FirewallScopedRule`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 页面/弹窗层对 `NetFirewallRule` 附加用户作用域的兼容类型。
    * **关键字段**: `userIds`, `targetUserIds`, `localRuleId`。

#### 4.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/firewall/`
* **核心业务用例 (Use Cases)**:
  * `FirewallService.loadOverviewState(context)`: 读取管理员/当前用户的系统策略和本地当前模式，生成首页状态。
    * **副作用**: 无。
    * **失败策略**: 无可用用户时返回关闭态；管理员/当前用户系统策略读取失败时视为未开启。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；系统 policy 读取和当前模式组合由 `FirewallPolicyService` 承接。
  * `FirewallService.setFirewallEnabledForAllUsers(context, enabled)`: PIN 认证后，对所有可用用户只修改 `policy.isOpen`。
    * **副作用**: 调用系统 `netFirewall.setNetFirewallPolicy`。
    * **失败策略**: 认证失败返回 `pin_unavailable/pin_failed`；PIN 认证成功但无可用用户时返回 `no_available_users`，不写系统 policy；部分用户写入失败返回 `policy_apply_failed` 和 `failedItems`。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；PIN 认证、用户列表校验和 policy 写入由 `FirewallPolicyService` 承接。
  * `FirewallService.switchMode(context, targetMode)`: 对所有可用用户执行首页模式切换。
    * **副作用**: 清理系统规则、写用户 policy 动作并保留用户原有 `isOpen`、清本地 deployments、下发目标模式规则、保存当前模式。
    * **失败策略**: 切换前快照当前模式、本地 deployments、每个用户的系统 policy 和系统规则；累积 `failedItems`，只要存在失败项则返回 `success=false`，不保存 `targetMode`，并尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式，避免系统策略半成功时只回退 UI。调用方必须在返回失败或异常中断后回读 `loadOverviewState`，用本地已确认模式和系统 policy 同步 UI。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；清规则、policy 应用、预置规则生成、deployment 保存和模式保存由 `FirewallModeSwitchService` 承接，其中 policy 写入继续委托 `FirewallPolicyService`。
  * `FirewallService.listRulesForDisplay(context)`: 从本地规则意图和 deployments 构建规则页展示数据。
    * **副作用**: 无。
    * **失败策略**: 本地读取失败由 Repository 兜底为空数据。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；本地规则意图组合、展示规则构造、展示规则冲突汇总由 `FirewallRuleDisplayService` 承接。
  * `FirewallService.createRule/updateRule/deleteRule(context, ...)`: 新增、更新、删除自定义规则。
    * **副作用**: 调用系统规则新增、更新、删除，并更新本地 `ruleIntents/ruleDeployments`。
    * **失败策略**: 输入校验失败不调用系统 API；新增、编辑、删除均采用 all-or-nothing 事务语义，任一系统操作失败时回滚已成功系统动作，只有全部系统操作成功后才提交本地 intent/deployment。新增规则任一用户下发失败时删除已新增系统规则且不保存本地；编辑规则按用户差量执行，目标类型非 DNS 时保留用户调用 `netFirewall.updateNetFirewallRule`，新增用户调用 `addNetFirewallRule`，移除用户调用 `removeNetFirewallRule`；目标类型为 DNS 时不调用 `updateNetFirewallRule`，保留用户先 `removeNetFirewallRule` 删除旧系统规则，再 `addNetFirewallRule` 按新 DNS 意图创建新系统规则，并用新增返回的 `systemRuleId` 替换本地 deployment，新增用户直接 add，移除用户直接 remove。编辑失败时分别回滚已 update、已 add、已 remove 的系统动作；DNS 替换式编辑失败时删除已 add 的新系统规则，并 add 回已 remove 的旧系统规则。删除规则任一系统删除失败时 add 回已删除系统规则并保留本地 intent。回滚 remove 后系统会生成新的 `systemRuleId`，必须同步写回本地 deployments，确保本地维护表与实际系统规则一致。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；事务编排由 `FirewallRuleMutationService` 承接，页面和 ViewModel 不直接依赖该内部 service。
  * `FirewallService.replaceConflictRule(context, conflictLocalRuleId, nextLocalRuleId, input, isEditMode)`: 替换冲突规则。
    * **副作用**: 删除冲突规则后新增或更新目标规则。
    * **失败策略**: 操作前捕获冲突规则 intent 和 deployments；目标规则新增/更新失败时回滚目标规则已创建部署并尝试恢复旧冲突规则和本地映射，避免页面层先删后加导致旧规则丢失。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；替换、回滚和恢复旧规则的编排由 `FirewallRuleMutationService` 承接。
  * `FirewallService.applyUserPolicyMode(context, userId, mode)`: 用户级 policy-only 模式下发。
    * **副作用**: PIN 认证、写目标用户 `NetFirewallPolicy`、保存用户历史模式。
    * **失败策略**: 认证失败、系统 policy 写入失败或本地保存失败均返回失败；该链路不筛规则、不删规则、不下发规则。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；policy 构造、写入和历史模式保存由 `FirewallPolicyService` 承接。
  * `FirewallService.loadUserPolicyState(context, userId)`: 读取用户系统策略和本地历史模式。
    * **副作用**: 无。
    * **失败策略**: 系统策略读取失败时返回非受管状态和默认允许动作。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；系统 policy 读取和历史模式回填由 `FirewallPolicyService` 承接。
  * `FirewallAccountChangeHandler.reconcile(context, snapshot)`: 账号变化后的防火墙业务同步。
    * **副作用**: 读取最新账号集合后清理无效用户本地数据；当前模式为 `public/private` 且账号集合签名变化时重放当前模式，写系统 policy、清理旧系统规则并下发预置规则。
    * **失败策略**: 账号列表为空或读取失败时不 prune、不重放；模式重放失败时沿用模式切换快照回滚语义，不保存新的账号集合签名。
    * **实现边界**: 账号事件由公共协调层触发，防火墙 Handler 只处理防火墙本模块数据和系统防火墙能力，不处理权限管理或 UI 强制刷新。
* **内部架构设计**:
  * **Repository**: `entry/src/main/ets/services/firewall/FirewallSystemRepository.ets` - 封装 `netFirewall` policy/rule 查询、设置、新增、更新、删除、清空。
  * **Repository**: `entry/src/main/ets/services/firewall/FirewallLocalRepository.ets` - 通过 Preferences 保存当前模式、期望开关状态、最近应用账号集合签名、规则意图/下发映射、用户历史模式；账号删除后的本地规则清理复用 `loadRuleIntentMappingData/saveRuleIntentMappingData` 整体读写规则映射，用户历史模式清理由批量保存过滤后的记录完成。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleMutationService.ets` - 自定义规则 create/update/delete/replaceConflictRule 事务边界，负责事务顺序、输入校验、用户差量计算、本地 intent 提交和失败项组合；目标类型为 DNS 的编辑在该层编排 remove+add 替换式事务，绕开系统 `updateNetFirewallRule` 的 DNS 重复误判；不被页面或 ViewModel 直接依赖。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleDeploymentService.ets` - 自定义规则 deployment 边界，负责系统规则下发、系统规则原地更新、旧部署删除、失败回滚、旧部署恢复和本地 deployments 保存/清理；不处理页面弹窗和规则冲突判断。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallPolicyService.ets` - 防火墙 policy 边界，负责首页总开关、首页状态 policy 读取、模式切换 policy 应用、用户 allowlist/denylist policy-only 下发和用户 policy 状态读取；不处理规则 CRUD、规则冲突或页面弹窗。
  * **Service**: `entry/src/main/ets/services/account/AccountChangeCoordinator.ets` - 账号变化协调器，只负责账号事件防抖、全量账号读取、账号快照生成、注册 Handler 分发和稳定账号事实事件发布；不写死防火墙、权限或页面逻辑。某个 Handler 失败不得阻断账号变化事实通知 UI 进程。
  * **Service**: `entry/src/main/ets/services/account/AccountChangeEventBus.ets` - 账号变化跨进程通知契约，封装账号添加/删除 CommonEvent 的发布、订阅和参数解析；事件只作为 UI 进程刷新触发器，不执行防火墙业务副作用。
  * **Service**: `entry/src/main/ets/services/account/AccountRuntimeService.ets` - UI 进程账号事件运行时订阅器，负责订阅账号 CommonEvent、记录最新账号变化 payload 并调用运行时层账号刷新回调；不按页面 route 分发模块逻辑，不下发策略、不访问系统防火墙 API。
  * **Service**: `entry/src/main/ets/services/account/AccountChangeHandler.ets` - 账号变化 Handler 接口，后续权限管理、目标应用等模块可注册独立 Handler。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallAccountChangeHandler.ets` - 防火墙账号变化 Handler，接收公共账号协调层产出的账号快照；负责无效用户本地清理、`public/private` 当前模式补下发和账号集合签名保存；`custom` 模式同步账号级默认 policy，但不扩展旧规则作用域，同步成功后同样保存 `custom` 模式和账号集合签名。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallModeSwitchService.ets` - 首页模式切换边界，负责切换前快照、清系统规则、清本地 deployments、调用 policy service 应用模式策略、按 `FirewallModeStrategy` 下发预置/自定义模式规则、保存当前模式，并在失败时尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleDisplayService.ets` - 规则展示边界，负责从本地 intent/deployment 组合展示规则、展示规则到 intent 的临时转换、重复/冲突检查和冲突汇总。
  * **Strategy**: `entry/src/main/ets/services/firewall/FirewallModeStrategy.ets` - 只负责 `public/private/custom` 模式规则生成，不下发、不写本地数据。
    * **配置来源**: `entry/src/main/resources/rawfile/firewall_preset_config.json`；`private` 模式读取失败时使用空配置兜底。`fixedPublicIp` 字段继续保留，但 `public` 模式不再读取或使用该字段生成远程访问放行规则。
    * **配置格式**: `privateTcpPorts` 和 `privateUdpPorts` 必须使用 JSON 数字数组；字符串端口不做隐式转换，避免预设配置契约出现多类型入参。
    * **public 模式规则族**: 允许 ICMPv4 入站；允许 HTTP/HTTPS 服务入站；无条件生成一条“禁止远程访问”入站 TCP 拒绝规则，不设置 `remoteIps`，对每个可用系统用户禁止所有来源 IP 访问本机 22、23、3389、5900 端口；拒绝 SMB 相关 TCP 端口入站；拒绝 NetBIOS 相关 UDP 端口入站。协议号、预置端口组和 IPv4 参数取值必须使用 `FirewallModels` 与 `FirewallModeStrategy` 内命名常量，不在规则生成流程继续散落裸数字。
    * **private 模式规则族**: 仅在存在 `privateCidrs` 时生成规则；允许私网 CIDR 内 ICMP、远程管理、共享和发现端口入站；允许配置的私网 TCP/UDP 扩展端口入站。自定义扩展端口仍按 `FIREWALL_PORT_MIN / FIREWALL_PORT_MAX` 校验。
    * **custom 模式规则来源**: 从 `FirewallLocalRepository.listRuleIntents(context)` 读取本地自定义规则意图，并按每条规则的 `targetUserIds` 生成系统规则。
  * **Utils**: `entry/src/main/ets/services/firewall/FirewallRuleUtils.ets` - 防火墙规则领域纯函数集合，不访问系统 `netFirewall` API，不访问 Preferences，不持有页面状态。
    * **输入校验**: `validateRuleIntentInput` 校验规则名称、目标用户、DOMAIN/DNS 必填项等。
    * **用户作用域归一化**: `normalizeUserIds` 对 `targetUserIds` 去重、过滤非法值并升序排序。
    * **规则意图转换**: `buildIntentFromInput`、`buildIntentInputFromRule` 在页面输入、展示规则和本地 `FirewallRuleIntent` 之间转换。
    * **系统规则构造**: `buildSystemRuleFromIntent`、`normalizeRuleForCreate` 将本地规则意图转换为可下发的 `NetFirewallRule`。
    * **展示规则构造**: `cloneDisplayRuleFromIntent` 根据本地 intent 和 deployments 构建规则页展示对象。
    * **冲突判断**: `findRuleConflict`、`summarizeConflicts` 负责重复、硬冲突、重叠冲突识别与汇总。
    * **IP/端口语义**: 内部归一化逻辑按规则方向选择本地/远端 IP 和端口，空值或用户输入 `ALL` 表示 ALL，并用于 IP 范围和端口范围重叠判断。
    * **冲突判断口径**: 仅当两条规则的 `targetUserIds` 存在交集时进入冲突判断；类型或方向不同则不冲突；同一规则 key 且动作相同为 `duplicate`，同一规则 key 但动作不同为 `hard_conflict`，动作不同且 IP/DOMAIN 规则存在范围重叠时为 `overlap_conflict`。
    * **规则 key 归一化**: IP 规则按方向选择主字段，入站使用本地 IP/本地端口，出站使用远端 IP/远端端口；空 IP 按 `0.0.0.0/0`，空端口按 `0-65535` 参与比较；DOMAIN 规则按精确域名和单级通配符域名归一化比较；DNS 规则按 `primaryDns` 小写归一化比较。
  * **Provider**: `entry/src/main/ets/services/firewall/providers/SystemUserProvider.ets` - 通过 API 26 `getOsAccountLocalIds()` 读取全部非系统可见账号，通过 `getForegroundOsAccountLocalId()` 标记当前账号，并用 `getOsAccountNameByLocalId(localId)` 读取账号显示名；账号列表不再回退 `getActivatedOsAccountLocalIds()` 或本地缓存，账号名称读取失败仅按 localId 展示。该 Provider 只作为账号真相源，不执行防火墙策略下发或本地清理副作用。

#### 4.3 ViewModel (视图模型层)
* **真实文件路径**: `entry/src/main/ets/viewmodels/firewall/`
* **状态分发逻辑**:
  * `overview/FirewallOverviewViewModel.ets`: 首页状态入口；调用 `FirewallService.loadOverviewState/setFirewallEnabledForAllUsers/switchMode`，并维护开关和模式处理状态。开关和模式切换完成后以 Service 回读结果同步页面状态；模式切换异常中断时也必须返回失败结果并尝试回读，避免 View 层维护额外业务镜像。开关与模式切换互斥执行，任一处理状态未结束时，另一类操作必须在 ViewModel 层返回失败结果。
  * `rules/FirewallRulesViewModel.ets`: 规则页状态入口；由 `ApplicationRuntimeManager` 持有，`MainPage` 仅负责从运行时取得并注入 `FirewallRulesPage`，在防火墙首页与规则页之间切换时保留 `currentTab` 等页面筛选状态。只在 `custom` 模式下加载规则，调用 `FirewallService` 完成规则增删改，并用持久化回读结果刷新页面规则；删除规则时维护当前删除中的 `localRuleId`，供卡片级删除动效消费，不复用整页 `isLoading`。

    * **默认策略展示来源**: `defaultPolicyItems` 来源于当前 `userOptions` 和 `FirewallService.loadUserPolicyState(context, userId)`。用户标签由 `SystemUserProvider` 通过 API 26 `getOsAccountNameByLocalId(localId)` 读取系统账号名称，读取失败时只回退用户 ID 文案；该列表只展示系统 policy 的默认动作，页面按 `allowlist/denylist` 聚合为白名单用户、黑名单用户静态摘要卡；方向自定义、读取失败或关闭态只在必要时以一行异常摘要提示。默认策略不进入 `rules`，不参与入站/出站统计和冲突判断，也不提供删除入口。

    * **用户选项来源**: `userOptions` 由 ViewModel 统一加载，规则弹窗只接收该入参，不直接访问 `SystemUserProvider`。
  * `user-dispatch/FirewallUserDispatchViewModel.ets`: 模式下发弹窗状态入口；加载用户、优先按本地 `bindingMode` 回显用户模式，没有本地记录时再按系统 `inAction/outAction` 推导真实模式，调用 `FirewallService.applyUserPolicyMode`。

#### 4.5 Components (可复用组件层)

## 5. 异常处理与系统依赖 (Dependencies & Errors)
- **关键系统 API**:
  - `@kit.NetworkKit/netFirewall`: 读取和写入防火墙 policy、规则列表、规则新增、规则删除、规则更新。
  - `@kit.BasicServicesKit.osAccount`: 读取全部非系统可见本地用户、前台用户和系统账号名称；账号列表使用 API 26 `getOsAccountLocalIds()`，账号名称使用 API 26 `getOsAccountNameByLocalId(localId)`。空列表及读取失败统一按“未找到可用本地用户”展示，不使用“已激活用户”口径。
  - `@kit.MDMKit/EnterpriseAdminExtensionAbility`: 企业管理员账号事件来源。
- **系统权限**:
  - `entry/src/main/module.json5`: `ohos.permission.MANAGE_NET_FIREWALL`、`ohos.permission.GET_NET_FIREWALL`、`ohos.permission.ENTERPRISE_MANAGE_NETWORK`、`ohos.permission.GET_LOCAL_ACCOUNTS`、`ohos.permission.GET_LOCAL_ACCOUNT_IDENTIFIERS`。
  - `hapsigner/UnsgnedDebugProfileTemplate.json`: 上述权限必须同步出现在 `acls.allowed-acls` 与 `permissions.restricted-permissions`。
  - **是否需要重新生成 p7b**: 涉及系统账号管理权限变更时，需要重新生成 p7b 后再签名。
- **异常兜底策略**:
  - 用户列表读取失败时返回空用户和提示文案，页面/弹窗进入不可操作或提示状态。
  - 总开关在无可用用户时返回失败，不把“0 个用户成功写入”视为切换成功。
  - policy 读取失败时首页按管理员/当前用户视为未开启，用户策略状态返回非受管默认允许态；规则页全局默认策略卡片按用户展示读取失败，不把失败用户误判为白名单或黑名单。
  - 总开关和用户模式下发必须先 PIN 认证；认证失败不写系统 policy。
  - 首页总开关和首页模式切换互斥执行；模式切换期间页面不得触发总开关、其它模式卡片或自定义规则入口，ViewModel 也必须拒绝并发请求。
  - DNS 规则重复以系统 `netFirewall.addNetFirewallRule` 返回为最终裁决。前端本地冲突提示仍只覆盖当前自定义规则列表中 `primaryDns` 相同的直观重复；不复刻系统源码中 `primaryDns` 或 `standbyDns` 任一相同即重复的完整判定。系统返回 `29400007` 时，新增/编辑链路将错误映射为 DNS 规则重复提示。
  - 规则新增、编辑、删除均不允许部分提交本地状态；任一系统操作失败时返回失败并回滚已成功系统动作。新增失败删除已新增系统规则；目标类型非 DNS 的编辑失败回滚已 update 规则、删除已 add 规则、add 回已 remove 规则；目标类型为 DNS 的编辑不调用系统 update，保留用户以 remove+add 替换旧系统规则，失败时删除已 add 的新规则、add 回已 remove 的旧规则；删除失败 add 回已 remove 规则。回滚 remove 后产生的新 `systemRuleId` 必须同步保存到本地 deployments。
  - 模式切换和规则增删改以 `failedItems` 记录失败明细；模式切换存在失败项时不保存目标模式，并基于切换前快照尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式；规则增删改存在失败项时不提交目标本地 intent/deployment，只保留或恢复与系统真实状态一致的旧本地映射；冲突替换由 Service 捕获旧规则并在目标规则失败时尝试恢复。涉及本地模式保存或 deployment 恢复的链路需要回读页面状态同步 UI。
  - 账号变化同步以事件触发、统一防抖和全量账号读取为准；账号列表为空或读取失败时不 prune、不重放模式。`account-added` 事件的触发账号 ID 只作为快照稳定性校验条件，不替代完整快照；如果防抖后的账号快照尚不包含触发账号 ID，协调层按 200ms 间隔最多重试 5 次，仍不可见时记录 warn 并返回失败，不更新 `previousUserIds`，不分发 Handler，不发布 `ACCOUNT_CHANGE` CommonEvent。删除账号事件不根据事件 ID 人工过滤账号快照；删除用户会从所有本地 `targetUserIds` 中移除，空作用域规则连同 deployments 一并删除，并移除该用户的本地 `allowlist/denylist` 历史模式记录；多用户规则删除其中一个用户后继续保留剩余用户作用域。该本地清理不对被删账号调用 `netFirewall` 系统接口。新增用户在 `public/private` 当前模式下需要补下发当前模式 policy 和预置规则，policy 的 `isOpen` 使用 `firewall_desired_enabled`，成功后由模式重放链路保存账号签名；新增用户在 `custom` 当前模式下只同步账号级默认 policy，不继承旧自定义规则，同步成功后由账号 Handler 保存 `custom` 和当前账号签名。
  - 账号变化后的 UI 不采用 MainPage 强制刷新四卡或 route 分支作为修复手段；四卡只是 ViewModel 数据读取后的展示结果。策略和本地数据必须先由 Handler 校准，UI 进程账号运行时服务收到账号添加/删除事件后重新读取完整账号列表并更新已加载 ViewModel，随后 `defaultPolicyItems/rules/userOptions` 自然联动。
  - 本地 Preferences 读取失败时返回空规则/默认模式；保存失败时返回失败结果并记录日志。

  - 总开关在无可用用户时必须失败并提示，不能把“0 个用户成功写入”视为成功；有用户时按认证和系统返回结果展示成功、失败或部分失败。首页顶部总开关展示只读取管理员/当前用户 `policy.isOpen`，不因新增账号 policy 尚未补齐而显示关闭。
  - 模式切换后需要回读系统状态并同步首页摘要；模式切换只在全部系统与本地步骤成功后保存目标模式，失败时必须尝试恢复切换前 policy、规则和本地 deployments。规则新增、编辑、删除只允许全部成功或全部失败；新增任一用户下发失败时不得保存本地规则；目标类型非 DNS 的编辑按保留用户 update、新增用户 add、移除用户 remove 的差量事务执行；目标类型为 DNS 的编辑按保留用户 remove+add、新增用户 add、移除用户 remove 的差量事务执行，全部系统动作成功后才保存新 intent/deployment，且保留用户的本地 deployment 必须刷新为 add 返回的新 `systemRuleId`；删除任一系统规则失败时必须恢复已删除系统规则并保留本地规则。冲突替换目标规则失败时应尽力恢复旧冲突规则。
  - 重新应用 `public` 模式后，每个可用系统用户必须获得一条“禁止远程访问”规则；该规则为启用的 `RULE_IP` 入站 TCP 拒绝规则，本地端口精确为 22、23、3389、5900，不设置 `remoteIps`，不得再生成基于 `fixedPublicIp` 的 SSH/RDP 放行规则。`private/custom` 模式规则语义保持不变。
  - 模式切换执行中，总开关、其它模式卡片和自定义规则入口不可触发；总开关执行中，模式卡片不可触发，ViewModel 层需要返回失败结果而不是进入并发系统写入。
  - 规则页必须始终让用户看到“未命中显式规则时的默认动作”。白名单模式不是一条可删除规则，而是 `NetFirewallPolicy.inAction/outAction = RULE_DENY` 的系统默认 policy；黑名单模式对应默认允许。多用户默认动作不一致时必须按白名单/黑名单聚合展示用户归属，优先展示系统账号真实名称，方向自定义或读取失败再以异常摘要补充。打开新增/编辑规则弹窗和用户默认策略弹窗时，账号列表必须来自 `FirewallRulesViewModel.userOptions`，不得在弹窗内重复读取系统账号。
  - 新增/编辑规则提交前必须重新校验目标用户仍存在；如果弹窗打开后账号被删除，页面提示用户重新选择作用域并阻止系统防火墙写入。“全部用户”仍保存为弹窗内当时选中的具体账号 ID，不引入动态 ALL 持久化语义。
  - 规则新增、编辑、删除和冲突替换成功后必须刷新账号、默认策略、规则列表和冲突提示，但不得清空本次成功操作提示。
  - 账号删除后，本地防火墙规则和用户历史模式不得继续引用被删用户；后续切回自定义模式不得因本地残留的已删除用户 ID 触发规则下发失败。若规则仅作用于被删用户，该规则应从本地列表消失；若规则还作用于其它用户，应保留并只展示剩余用户作用域。
  - `public/private` 模式下新增账号后，账号事件必须触发防火墙账号同步；协调层必须先确认账号快照包含触发账号 ID，避免将旧账号集合误判为成功同步。新账号应获得当前模式的 policy 和预置规则，`isOpen` 与本地 `desiredEnabled` 一致；同步成功后保存最新账号集合签名。
  - `custom` 模式下新增账号必须获得账号级默认 policy，但不得自动扩展旧规则作用域，也不得自动下发旧自定义规则；用户需要显式编辑规则作用域后新账号才获得对应自定义规则。
  - 从防火墙规则页切到其它模块后再次进入防火墙，应恢复最近子路由；只有规则页返回按钮显式切回防火墙首页。
  - DNS 规则新增/编辑弹窗必须提供主 DNS 与备用 DNS 输入；主 DNS 必填，备用 DNS 可选且非空时必须为合法 IPv4。提交后本地意图和系统规则必须保留非空 `standbyDns`，规则列表必须展示主 DNS 和备用 DNS。系统新增规则返回 `29400007` 时，页面必须提示 DNS 规则重复；前端不得因备用 DNS 相同或同时为空而在本地冲突检查中提前拦截。

### 5.2 大文件职责拆分修改

- **修改前问题**:

- **修改目标**:
  - 保持现有路由、公开用例、Preferences 结构、权限、用户可见文案和系统调用结果不变，只做职责拆分和测试补强。

- **保持不变**:
  - 不改防火墙业务规则、冲突判断口径、默认策略语义、PIN 认证边界、账号事件链路、签名权限或 p7b。DNS 备用地址只补齐系统规则字段输入与展示，不把系统 DNS 重复判定完整搬到前端本地冲突判断。
  - 不引入新的状态管理框架，不把页面事件直接接到 Repository / SystemRepository，不让新组件持有系统 API。
  - 不在本轮引入 LazyForEach DataSource；规则列表仍可继续使用现有 ForEach 数据源，除非后续性能验证单独立项。
- **当前结构**:
  - `services/firewall/FirewallService.ets`: facade，只保留公开 use case 编排入口和少量参数转换。
  - `services/firewall/FirewallRuleMutationService.ets`: 自定义规则 create/update/delete/replace 的事务边界，负责成功/失败结果组合，不直接处理页面弹窗。

  - `services/firewall/FirewallRuleDisplayService.ets`: 规则展示边界，负责本地 intent/deployment 组合、展示规则构造、展示规则冲突检查和冲突汇总。
  - `components/firewall/rules/FirewallRuleFormModel.ets`: 弹窗表单解析、校验、回填、规则构造纯逻辑；不依赖 ArkUI，不调用系统 API。
  - `components/firewall/rules/FirewallRuleList.ets`、`FirewallRulesSummaryCards.ets`: 规则页展示组件，接收 ViewModel 已准备好的规则、默认策略和事件回调；规则卡片作为列表组件内部实现，不单独暴露文件。

- **兼容要求**:
  - `FirewallService` 原有公开方法签名保持可用；ViewModel 和页面仍通过 facade 调用业务能力。
  - 每个拆分项独立可构建、可回滚；不得在一个提交里同时改 service 事务边界和页面 UI 行为。
  - 拆分后如果业务行为发生变化，必须先补充本节对应验收口径，再实施代码。

  - `FirewallRulesPage.ets` 不再内联完整规则卡片和摘要行渲染，页面主体只负责 SubPageHeader、Tabs/列表装配和弹窗事件。
  - `AddRuleDialog.ets` 不再内联 IP/端口/域名解析和规则构造细节，相关逻辑有直接 UT。
  - 现有 `entry/src/test/firewall/*.test.ets`、`entry/src/test/views/SettingsPagesState.test.ets`、HAP 构建继续通过。
