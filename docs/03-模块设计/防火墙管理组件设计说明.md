## 1. 业务概述与对外接口 (Overview & Public Interfaces)
- **核心目标**: 为设备上所有可用系统用户提供防火墙开关、预设模式、自定义规则和用户级默认策略管理能力。
- **入口路由 (Route Entry)**:
  - Shell Entry: `pages/MainPage`
  - Route ID: `firewall`、`firewall-rules`
  - View Entry: `entry/src/main/ets/views/firewall/overview/FirewallPage.ets`、`entry/src/main/ets/views/firewall/rules/FirewallRulesPage.ets`
- **对外暴露能力 (Public APIs)**:
  - `entry/src/main/ets/services/firewall/FirewallService.ets`: 防火墙模块业务编排入口。
  - `entry/src/main/ets/services/firewall/providers/SystemUserProvider.ets`: 防火墙可用用户列表与用户策略摘要读取入口。
- **业务边界**:
  - ✅ **包含**: 全用户防火墙开关、`public/private/custom` 首页模式切换、自定义规则增删改查、规则冲突检测、规则用户作用域、用户级 `allowlist/denylist` 默认策略下发、账号事件驱动的用户列表增量跟踪。
  - ❌ **不包含**: 系统防火墙能力本身实现、账号管理 UI、PIN 认证规则实现、测试框架实现、签名工具实现。

## 2. 状态与数据流 (Data Flow & State)
> **设计原则**: 单向数据流 (View -> ViewModel -> Service -> Repository/Storage)

- **核心业务状态 (Core Business State)**:
  - `FirewallOverviewViewModelState.enabled: boolean`: 当前可用用户集合是否全部处于防火墙开启状态，初始值 `false`。
  - `FirewallOverviewViewModelState.currentMode: FirewallPresetMode`: 当前首页模式，取值 `public | private | custom`，默认 `custom`。
  - `FirewallOverviewViewModelState.initialized/isLoading/isToggleProcessing/isModeProcessing: boolean`: 首页初始化、刷新、开关和模式切换处理状态；`isModeProcessing` 为首页模式切换互斥锁，期间不允许触发总开关、其它模式卡片或自定义规则入口。
  - `FirewallRulesViewModel.currentMode: FirewallPresetMode`: 规则页接收到的当前模式；非 `custom` 时不允许编辑规则。
  - `FirewallRulesViewModel.rules: NetFirewallRule[]`: 规则页展示列表，由本地 `FirewallRuleIntent` 和成功下发的 `FirewallRuleDeployment` 回读生成。
  - `FirewallRulesViewModel.defaultPolicyItems: FirewallDefaultPolicyDisplayItem[]`: 规则页统计区默认策略数据源，按用户读取 `allowlist/denylist/custom/unknown` 和入站/出站默认动作，页面用与入站/出站规则相同的静态摘要卡片聚合展示白名单用户、黑名单用户及必要异常摘要。
  - `FirewallRulesViewModel.conflictNotice/currentTab/userOptions/lastSaveResult/lastOperationMessage`: 规则冲突提示、列表分组、可选用户、最近保存结果和操作提示。
  - `MainPage.lastFirewallRoute: string`: 防火墙模块内最近一次子路由，取值 `firewall | firewall-rules`，用于从侧边栏、总览快捷入口或顶部菜单再次进入防火墙模块时恢复上次所在页面。
  - `FirewallUserDispatchViewModel.users/selectedUserId/selectedModeIndex/statusHintText/lastApplyResult`: 模式下发弹窗的用户列表、当前用户、目标模式、状态提示和最近下发结果。
- **关键流转路径**:
  - `进入 MainPage` -> `FirewallOverviewViewModel.initialize` -> `FirewallService.loadOverviewState` -> `SystemUserProvider + FirewallSystemRepository + FirewallLocalRepository` -> `首页状态同步`
  - `点击首页总开关` -> `FirewallOverviewViewModel.requestToggle` -> `FirewallService.setFirewallEnabledForAllUsers` -> `PIN 认证 + 全用户 setPolicy(isOpen)` -> `成功后更新 enabled`；如果模式切换正在执行，ViewModel 直接拒绝本次开关请求。
  - `切换首页模式` -> `FirewallOverviewViewModel.requestModeSwitch` -> `FirewallService.switchMode` -> `切换前快照当前模式/本地映射/用户 policy/系统规则 + 清系统规则 + 按模式写 policy 动作但保留 isOpen + 清 deployments + 生成并下发规则 + 全部成功后保存模式` -> `回读首页状态并同步 currentMode/enabled`；任一步骤失败时不保存目标模式，并尝试恢复快照；如果总开关正在执行，ViewModel 直接拒绝本次模式切换请求。
  - `进入规则页` -> `MainPage` 注入共享 `FirewallRulesViewModel` -> `FirewallRulesViewModel.initialize/loadRules` -> `SystemUserProvider.loadAvailableUserIds + FirewallService.loadUserPolicyState` 读取全局默认策略 -> `FirewallService.listRulesForDisplay` -> `FirewallLocalRepository.loadRuleIntentMappingData` -> `展示默认策略摘要、规则和冲突提示`
  - `规则页切到其它模块后再次进入防火墙` -> `MainPage` 根据 `lastFirewallRoute` 恢复到 `firewall-rules` -> `复用共享 FirewallRulesViewModel.currentTab` -> `保留上次退出规则页时的分组 Tab` -> `重新加载规则列表和冲突提示`
  - `规则页点击返回防火墙首页` -> `MainPage.setCurrentPage(firewall)` 显式更新 `lastFirewallRoute` -> 后续再进入防火墙模块默认展示首页
  - `新增/编辑/删除规则` -> `FirewallRulesViewModel` -> `FirewallService.createRule/updateRule/deleteRule` -> `FirewallRuleUtils + FirewallSystemRepository + FirewallLocalRepository` -> `持久化回读刷新规则列表`
  - `打开模式下发弹窗` -> `FirewallUserDispatchViewModel.initialize` -> `SystemUserProvider.loadAvailableUserIds` 读取可用用户和系统账号名 -> `FirewallService.loadUserPolicyState` -> `优先按本地历史模式回显；无历史模式时再按系统默认动作推导用户模式`
  - `提交用户模式` -> `FirewallUserDispatchViewModel.applySelectedMode` -> `FirewallService.applyUserPolicyMode` -> `PIN 认证 + policy-only setPolicy + 保存 userId -> mode` -> `重新读取用户策略状态`
  - `账号新增/删除` -> `EnterpriseAdminAbility 发布 CommonEvent` -> `EntryAbility 接收` -> `SystemUserProvider.trackAddedUser/trackRemovedUser` -> `下次打开弹窗时重新读取用户列表`

## 3. 模块结构与组件设计 (Module Components)

### 【核心层】(Core MVVM & Domain Layers)

#### 3.1 Model & Types (核心数据模型与类型)
* **核心实体与依赖路径**:
  * `FirewallPresetMode`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 首页预设模式类型。
    * **关键字段**: `public`、`private`、`custom`。
  * `FirewallUserPolicyMode`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 用户级默认策略模式。
    * **关键字段**: `allowlist` 映射为默认入站/出站拒绝，`denylist` 映射为默认入站/出站允许。
  * `FirewallRuleIntent`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 自定义规则的本地业务意图，是规则模板的长期真相。
    * **关键字段**: `localRuleId` (本地稳定身份), `targetUserIds` (固定用户集合), `type/direction/action` (规则匹配和冲突判断), `createdAt/updatedAt` (展示与排序辅助)。
  * `FirewallRuleDeployment`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 记录本地规则成功下发到系统后的映射。
    * **关键字段**: `localRuleId` (关联本地意图), `userId` (目标系统用户), `systemRuleId` (系统规则 ID)。
  * `FirewallDefaultPolicyDisplayItem`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 规则页只读展示用户级默认 policy，不作为可删除规则参与增删改、统计或冲突判断。
    * **关键字段**: `userId/label`, `mode` (`allowlist/denylist/custom/unknown`), `isOpen`, `inAction/outAction`。
  * `FirewallRuleIntentMappingData`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 防火墙自定义规则本地持久化结构。
    * **关键字段**: `version`, `ruleIntents`, `ruleDeployments`。
  * `FirewallScopedRule`
    * **所在文件**: `entry/src/main/ets/services/firewall/FirewallModels.ets`
    * **业务作用**: 页面/弹窗层对 `NetFirewallRule` 附加用户作用域的兼容类型。
    * **关键字段**: `userIds`, `targetUserIds`, `localRuleId`。

#### 3.2 Service / Domain (领域业务层)
* **真实文件路径**: `entry/src/main/ets/services/firewall/`
* **核心业务用例 (Use Cases)**:
  * `FirewallService.loadOverviewState(context)`: 读取全部可用用户的系统策略和本地当前模式，生成首页状态。
    * **副作用**: 无。
    * **失败策略**: 无可用用户时返回关闭态；系统策略读取失败时视为未全部开启。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；系统 policy 读取和当前模式组合由 `FirewallPolicyService` 承接。
  * `FirewallService.setFirewallEnabledForAllUsers(context, enabled)`: PIN 认证后，对所有可用用户只修改 `policy.isOpen`。
    * **副作用**: 调用系统 `netFirewall.setNetFirewallPolicy`。
    * **失败策略**: 认证失败返回 `pin_unavailable/pin_failed`；PIN 认证成功但无可用用户时返回 `no_available_users`，不写系统 policy；部分用户写入失败返回 `policy_apply_failed` 和 `failedItems`。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；PIN 认证、用户列表校验和 policy 写入由 `FirewallPolicyService` 承接。
  * `FirewallService.switchMode(context, targetMode)`: 对所有可用用户执行首页模式切换。
    * **副作用**: 清理系统规则、写用户 policy 动作并保留用户原有 `isOpen`、清本地 deployments、下发目标模式规则、保存当前模式。
    * **失败策略**: 切换前快照当前模式、本地 deployments、每个用户的系统 policy 和系统规则；累积 `failedItems`，只要存在失败项则返回 `success=false`，不保存 `targetMode`，并尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式，避免系统策略半成功时只回退 UI。调用方必须在返回失败或异常中断后回读 `loadOverviewState`，用本地已确认模式和系统 policy 同步 UI。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；清规则、policy 应用、预置规则生成、deployment 保存和模式保存由 `FirewallModeSwitchService` 承接，其中 policy 写入继续委托 `FirewallPolicyService`。
  * `FirewallService.listRulesForDisplay(context)`: 从本地规则意图和 deployments 构建规则页展示数据。
    * **副作用**: 无。
    * **失败策略**: 本地读取失败由 Repository 兜底为空数据。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；本地规则意图组合、展示规则构造、展示规则冲突汇总由 `FirewallRuleDisplayService` 承接。
  * `FirewallService.createRule/updateRule/deleteRule(context, ...)`: 新增、更新、删除自定义规则。
    * **副作用**: 调用系统规则新增/删除，并更新本地 `ruleIntents/ruleDeployments`。
    * **失败策略**: 输入校验失败不调用系统 API；全部下发失败不保存本地意图；部分成功返回成功 deployments 与失败项；编辑规则时先成功创建新系统部署，再删除旧部署并提交本地新意图，若新部署全部失败则保留旧系统部署和旧本地意图，避免 UI 与系统状态脱节；删除规则时如果任一系统规则删除失败，保留本地 intent 和失败 deployment，方便页面刷新后继续追踪并重试。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；事务编排由 `FirewallRuleMutationService` 承接，页面和 ViewModel 不直接依赖该内部 service。
  * `FirewallService.replaceConflictRule(context, conflictLocalRuleId, nextLocalRuleId, input, isEditMode)`: 替换冲突规则。
    * **副作用**: 删除冲突规则后新增或更新目标规则。
    * **失败策略**: 操作前捕获冲突规则 intent 和 deployments；目标规则新增/更新失败时回滚目标规则已创建部署并尝试恢复旧冲突规则和本地映射，避免页面层先删后加导致旧规则丢失。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；替换、回滚和恢复旧规则的编排由 `FirewallRuleMutationService` 承接。
  * `FirewallService.applyUserPolicyMode(context, userId, mode)`: 用户级 policy-only 模式下发。
    * **副作用**: PIN 认证、写目标用户 `NetFirewallPolicy`、保存用户历史模式。
    * **失败策略**: 认证失败、系统 policy 写入失败或本地保存失败均返回失败；该链路不筛规则、不删规则、不下发规则。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；policy 构造、写入和历史模式保存由 `FirewallPolicyService` 承接。
  * `FirewallService.loadUserPolicyState(context, userId)`: 读取用户系统策略和本地历史模式。
    * **副作用**: 无。
    * **失败策略**: 系统策略读取失败时返回非受管状态和默认允许动作。
    * **实现边界**: 公开方法保留在 `FirewallService` facade；系统 policy 读取和历史模式回填由 `FirewallPolicyService` 承接。
* **内部架构设计**:
  * **Repository**: `entry/src/main/ets/services/firewall/FirewallSystemRepository.ets` - 封装 `netFirewall` policy/rule 查询、设置、新增、删除、清空。
  * **Repository**: `entry/src/main/ets/services/firewall/FirewallLocalRepository.ets` - 通过 Preferences 保存当前模式、规则意图/下发映射、用户历史模式。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleMutationService.ets` - 自定义规则 create/update/delete/replaceConflictRule 事务边界，负责事务顺序、输入校验、本地 intent 提交和失败项组合；不被页面或 ViewModel 直接依赖。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleDeploymentService.ets` - 自定义规则 deployment 边界，负责系统规则下发、旧部署删除、失败回滚、旧部署恢复和本地 deployments 保存/清理；不处理页面弹窗和规则冲突判断。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallPolicyService.ets` - 防火墙 policy 边界，负责首页总开关、首页状态 policy 读取、模式切换 policy 应用、用户 allowlist/denylist policy-only 下发和用户 policy 状态读取；不处理规则 CRUD、规则冲突或页面弹窗。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallModeSwitchService.ets` - 首页模式切换边界，负责切换前快照、清系统规则、清本地 deployments、调用 policy service 应用模式策略、按 `FirewallModeStrategy` 下发预置/自定义模式规则、保存当前模式，并在失败时尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式。
  * **Service**: `entry/src/main/ets/services/firewall/FirewallRuleDisplayService.ets` - 规则展示边界，负责从本地 intent/deployment 组合展示规则、展示规则到 intent 的临时转换、重复/冲突检查和冲突汇总。
  * **Strategy**: `entry/src/main/ets/services/firewall/FirewallModeStrategy.ets` - 只负责 `public/private/custom` 模式规则生成，不下发、不写本地数据。
    * **配置来源**: `entry/src/main/resources/rawfile/firewall_preset_config.json`，读取失败时使用空配置兜底。
    * **配置格式**: `privateTcpPorts` 和 `privateUdpPorts` 必须使用 JSON 数字数组；字符串端口不做隐式转换，避免预设配置契约出现多类型入参。
    * **public 模式规则族**: 允许 ICMPv4 入站；允许 HTTP/HTTPS 服务入站；若配置 `fixedPublicIp`，允许来自该 IP 的 SSH/RDP 远程管理入站；拒绝 SMB 相关 TCP 端口入站；拒绝 NetBIOS 相关 UDP 端口入站。协议号、预置端口组和 IPv4 参数取值必须使用 `FirewallModels` 与 `FirewallModeStrategy` 内命名常量，不在规则生成流程继续散落裸数字。
    * **private 模式规则族**: 仅在存在 `privateCidrs` 时生成规则；允许私网 CIDR 内 ICMP、远程管理、共享和发现端口入站；允许配置的私网 TCP/UDP 扩展端口入站。自定义扩展端口仍按 `FIREWALL_PORT_MIN / FIREWALL_PORT_MAX` 校验。
    * **custom 模式规则来源**: 从 `FirewallLocalRepository.listRuleIntents(context)` 读取本地自定义规则意图，并按每条规则的 `targetUserIds` 生成系统规则。
  * **Utils**: `entry/src/main/ets/services/firewall/FirewallRuleUtils.ets` - 防火墙规则领域纯函数集合，不访问系统 `netFirewall` API，不访问 Preferences，不持有页面状态。
    * **输入校验**: `validateRuleIntentInput` 校验规则名称、目标用户、DOMAIN/DNS 必填项等。
    * **用户作用域归一化**: `normalizeUserIds` 对 `targetUserIds` 去重、过滤非法值并升序排序。
    * **规则意图转换**: `buildIntentFromInput`、`buildIntentInputFromRule` 在页面输入、展示规则和本地 `FirewallRuleIntent` 之间转换。
    * **系统规则构造**: `buildSystemRuleFromIntent`、`normalizeRuleForCreate` 将本地规则意图转换为可下发的 `NetFirewallRule`。
    * **展示规则构造**: `cloneDisplayRuleFromIntent` 根据本地 intent 和 deployments 构建规则页展示对象。
    * **冲突判断**: `findRuleConflict`、`summarizeConflicts` 负责重复、硬冲突、重叠冲突识别与汇总。
    * **IP/端口语义**: 内部归一化逻辑按规则方向选择本地/远端 IP 和端口，空值或用户输入 `ALL` 表示 ALL，并用于 IP 范围和端口范围重叠判断。
    * **冲突判断口径**: 仅当两条规则的 `targetUserIds` 存在交集时进入冲突判断；类型或方向不同则不冲突；同一规则 key 且动作相同为 `duplicate`，同一规则 key 但动作不同为 `hard_conflict`，动作不同且 IP/DOMAIN 规则存在范围重叠时为 `overlap_conflict`。
    * **规则 key 归一化**: IP 规则按方向选择主字段，入站使用本地 IP/本地端口，出站使用远端 IP/远端端口；空 IP 按 `0.0.0.0/0`，空端口按 `0-65535` 参与比较；DOMAIN 规则按精确域名和单级通配符域名归一化比较；DNS 规则按 `primaryDns` 小写归一化比较。
  * **Provider**: `entry/src/main/ets/services/firewall/providers/SystemUserProvider.ets` - 读取激活用户、前台用户、tracked users，并输出防火墙可用用户列表。

#### 3.3 ViewModel (视图模型层)
* **真实文件路径**: `entry/src/main/ets/viewmodels/firewall/`
* **状态分发逻辑**:

## 4. 异常处理与系统依赖 (Dependencies & Errors)
- **关键系统 API**:
  - `@kit.NetworkKit/netFirewall`: 读取和写入防火墙 policy、规则列表、规则新增、规则删除、规则更新。
  - `@ohos.account.osAccount`: 读取激活用户、前台用户和系统账号 `localName`。
  - `@kit.BasicServicesKit/commonEventManager`: 传递账号新增/删除事件。
  - `@kit.MDMKit/EnterpriseAdminExtensionAbility`: 企业管理员账号事件来源。
- **系统权限**:
  - `entry/src/main/module.json5`: `ohos.permission.MANAGE_NET_FIREWALL`、`ohos.permission.GET_NET_FIREWALL`、`ohos.permission.ENTERPRISE_MANAGE_NETWORK`、`ohos.permission.GET_LOCAL_ACCOUNTS`、`ohos.permission.MANAGE_LOCAL_ACCOUNTS`。
  - `hapsigner/UnsgnedDebugProfileTemplate.json`: 上述权限必须同步出现在 `acls.allowed-acls` 与 `permissions.restricted-permissions`。
  - **是否需要重新生成 p7b**: 涉及系统账号管理权限变更时，需要重新生成 p7b 后再签名。
- **异常兜底策略**:
  - 用户列表读取失败时返回空用户和提示文案，页面/弹窗进入不可操作或提示状态。
  - 总开关在无可用用户时返回失败，不把“0 个用户成功写入”视为切换成功。
  - policy 读取失败时首页视为未全部开启，用户策略状态返回非受管默认允许态；规则页全局默认策略卡片按用户展示读取失败，不把失败用户误判为白名单或黑名单。
  - 总开关和用户模式下发必须先 PIN 认证；认证失败不写系统 policy。
  - 首页总开关和首页模式切换互斥执行；模式切换期间页面不得触发总开关、其它模式卡片或自定义规则入口，ViewModel 也必须拒绝并发请求。
  - 规则编辑的新部署全部失败时不删除旧部署、不覆盖旧本地意图；旧部署删除或本地提交失败时返回失败并尝试回滚新部署或恢复旧部署映射。
  - 模式切换和规则增删改以 `failedItems` 记录部分失败；模式切换存在失败项时不保存目标模式，并基于切换前快照尝试恢复旧 policy、旧系统规则、旧 deployments 和旧模式；删除规则存在系统删除失败时不删除本地 intent；冲突替换由 Service 捕获旧规则并在目标规则失败时尝试恢复。涉及本地模式保存的链路需要回读首页状态同步 UI。
  - 本地 Preferences 读取失败时返回空规则/默认模式；保存失败时返回失败结果并记录日志。

### 4.1 实施步骤与测试验收 (Implementation & Acceptance)

- **实施步骤**:
  - 涉及防火墙总开关、模式策略、规则增删改、子路由恢复、用户作用域或权限变化时，先更新本文档的目标范围、状态流、职责边界、异常兜底和验收口径。
  - 数据模型、预设规则、冲突判断和展示构造先同步 `FirewallModels.ets`、`FirewallPresetConfig.ets`、`FirewallModeStrategy.ets`、`FirewallRuleUtils.ets`，再调整 Service 或页面。

  - 页面交互按 `FirewallPage.ets`、`FirewallRulesPage.ets`、`FirewallAddRuleDialog.ets`、对应 ViewModel 的顺序落地；敏感写操作必须保留 PIN 认证、用户列表校验和部分失败展示。规则页全局默认策略卡片是 policy 可视化，不应复用普通规则增删改链路。
  - 新增权限或调整受限权限时，同步 `entry/src/main/module.json5`、`hapsigner/UnsgnedDebugProfileTemplate.json`，并按 AGENTS.md 重新生成 p7b 后签名。
- **测试覆盖**:
  - UT: `entry/src/test/firewall/*.test.ets` 覆盖模式策略、规则校验、规则仓储、Service 写入边界、默认策略展示派生、ViewModel 状态和弹窗入参。

  - E2E: `scripts/e2e/cases/firewall/` 下 `status_toggle.json`、`mode_cards.json`、`rule_create.json`、`rule_delete.json`、`rules_tabs.json`、`rules_page_visible.json`、`subroute_restore.json`、`domain_browser.json` 覆盖主流程和路由恢复。
- **验收口径**:
  - 总开关在无可用用户时必须失败并提示，不能把“0 个用户成功写入”视为成功；有用户时按认证和系统返回结果展示成功、失败或部分失败。

  - 从防火墙规则页切到其它模块后再次进入防火墙，应恢复最近子路由；只有规则页返回按钮显式切回防火墙首页。

### 4.2 大文件职责拆分修改
