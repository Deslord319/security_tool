# 安全总览改造设计

## 1. 背景

当前安全总览页存在以下问题：

- 顶部健康标签与下方卡片存在信息重复，页面主次不够清晰。
- 总览页四张卡片偏日志统计视角，未体现“模块入口 + 模块状态”的首页职责。
- 总览页与日志页依赖定时轮询刷新，实时性、性能和可维护性一般。
- 首页、日志页、防火墙规则页存在多套相近但未统一抽象的卡片样式。

结合当前产品状态，安全总览页更适合作为“安全管理中心首页”，承担以下职责：

- 概览四大核心能力模块的当前状态。
- 作为模块主入口，支持快速进入目标页面。
- 对重要状态给出一眼可读的摘要，而不是堆叠细颗粒统计。

## 2. 目标

本次改造目标如下：

1. 删除总览页顶部“管理员 / 日志管理 / 外设管理”健康标签区域。
2. 将总览页主区统一为四张模块摘要卡：
   - 管理员
   - 防火墙
   - 日志
   - 外设
3. 每张卡片同时承担“状态概览 + 入口跳转”职责。
4. 将总览页卡片与日志页顶部四张卡片统一到同一套摘要卡片体系。
5. 为防火墙自定义模式中的“入站 / 出站”卡片提供同风格紧凑变体。
6. 用监听驱动替代主要页面轮询，至少覆盖日志与外设链路。

## 3. 非目标

本次改造默认不包含以下内容：

- 重做侧边栏信息架构。
- 调整模块路由定义。
- 重构防火墙规则列表主体布局。
- 新增日志图表、趋势图、复杂分析面板。
- 引入新的权限模型或新的系统事件源。

## 4. 当前实现摘要

### 4.1 总览页

当前 `DashboardPage` 由三部分组成：

- 顶部标题区
- 顶部健康标签区：管理员 / 日志管理 / 外设管理
- 四张统计卡片：防火墙状态 / 今日审计 / 已阻止事件 / 日志总量
- 六张快捷入口卡片：防火墙 / 日志 / 外设 / 身份 / 工具设置 / 帮助反馈

当前数据刷新方式：

- 总览页通过 `setInterval` 每 3 秒刷新一次。
- 刷新内容包括日志统计和运行状态探测。

### 4.2 日志管理页

日志管理页不是占位页，已经具备完整功能：

- 日志列表
- 筛选
- 导出
- 清空
- 存储设置
- 顶部四张统计卡

当前数据刷新方式：

- 页面通过 `setInterval` 每 1.5 秒刷新一次。

### 4.3 外设管理页

外设管理页不是占位页，已经具备完整功能：

- 接口控制
- 设备连接记录
- 设备策略管理
- 导出
- 清空

当前刷新机制：

- 已接入 `PeripheralTraceRepository` 与 `PeripheralDevicePolicyRepository` 的变更监听。
- ViewModel 已使用监听驱动页面刷新。

### 4.4 卡片实现

当前卡片相关组件包括：

- `StatCard`：总览页统计卡
- `ToolCard`：总览页快捷入口卡
- `DirectionCard`：防火墙规则页入站 / 出站摘要卡

问题在于：

- 三者视觉风格接近，但抽象层次不统一。
- 首页摘要卡与日志页顶部统计卡未共用组件。
- 防火墙方向卡与摘要卡之间缺少可复用的中间层。

## 5. 目标信息架构

### 5.1 页面结构

安全总览页改造后建议保留以下结构：

1. 标题区
2. 四张模块摘要卡
3. 次级入口区

说明：

- 原顶部健康标签区删除。
- 原“六张快捷入口卡”建议缩减为次级入口区，仅保留非核心模块入口。
- 四张摘要卡成为首页主视觉和主交互区。

### 5.2 首页模块分层

建议首页表达以下层级：

- 第一层：管理员 / 防火墙 / 日志 / 外设
- 第二层：身份鉴别 / 工具设置 / 帮助反馈

原因：

- 第一层是当前安全能力闭环中最适合首页突出展示的四个模块。
- 第二层更偏辅助或配置能力，不适合与核心状态卡竞争注意力。

## 6. 模块摘要卡设计

### 6.1 卡片职责

每张模块摘要卡承担三类信息：

1. 模块名称
2. 核心主状态
3. 辅助说明或次级指标

同时支持点击跳转到对应模块页。

### 6.2 四张卡的建议字段

#### 管理员卡

- 标题：管理员
- 主状态：
  - 已激活
  - 未激活
- 辅助说明：
  - 已具备企业管理能力
  - 需先激活企业管理员
- 状态语义：
  - 成功态：已激活
  - 警示态：未激活
- 点击行为：
  - 跳转到管理员相关页面或当前暂跳转到身份鉴别页

说明：

- 如果后续没有独立管理员页面，可先保留跳转到身份鉴别页，或仅作为状态卡不跳转。
- 该点需要在实施时做一次产品确认。

#### 防火墙卡

- 标题：防火墙
- 主状态：
  - 已开启
  - 已关闭
- 辅助说明：
  - 当前模式：公共 / 专用 / 自定义
- 状态语义：
  - 成功态：已开启
  - 警示态：已关闭
- 点击行为：
  - 跳转到防火墙页面

#### 日志卡

- 标题：日志
- 主状态建议二选一：
  - 今日日志数
  - 总日志数
- 辅助说明：
  - 采集中 / 未采集
  - 已拦截事件数
- 状态语义：
  - 成功态：采集中
  - 警示态：未采集
- 点击行为：
  - 跳转到日志管理页

推荐方案：

- 主值使用“今日日志数”
- 副说明展示“采集中 / 未采集”
- 次级说明可在卡片底部展示“拦截 X 条”

这样能同时体现规模和运行态。

#### 外设卡

- 标题：外设
- 主状态建议二选一：
  - 今日事件数
  - 最近接入设备数
- 辅助说明：
  - 监听中 / 未监听
  - 最近一次事件时间或受控设备数
- 状态语义：
  - 成功态：监听中
  - 警示态：未监听
- 点击行为：
  - 跳转到外设管理页

推荐方案：

- 主值使用“今日事件数”
- 副说明展示“监听中 / 未监听”

## 7. 次级入口区设计

首页下半区建议保留简化版入口区，承接非核心模块：

- 身份鉴别
- 工具设置
- 帮助反馈

不再重复展示：

- 管理员
- 防火墙
- 日志
- 外设

原因：

- 四张摘要卡已经具备入口能力，再重复一套入口会稀释首页重心。

## 8. 卡片组件设计

## 8.1 设计原则

组件设计建议采用“基础容器 + 两个摘要模板”的方式，而不是强行合并为一个超大组件。

目标结构如下：

- `BaseSummaryCard`
- `MetricSummaryCard`
- `CompactSummaryCard`

### 8.2 BaseSummaryCard

职责：

- 统一背景、边框、圆角、阴影、悬停、点击、主题色切换。
- 提供头部区域、主体区域、尾部区域插槽。
- 负责是否可点击、hover 动效、禁用态透明度。

建议属性：

- `clickable: boolean`
- `disabled: boolean`
- `accentStart: string`
- `accentEnd: string`
- `onPress?: () => void`
- `content: () => void`

### 8.3 MetricSummaryCard

适用场景：

- 首页四张模块摘要卡
- 日志管理页顶部四张统计卡

建议属性：

- `title: string`
- `primaryText: string`
- `secondaryText: string`
- `statusText?: string`
- `statusKind?: 'success' | 'warning' | 'neutral'`
- `icon?: Resource`
- `accentStart: string`
- `accentEnd: string`
- `onPress?: () => void`

布局建议：

- 左上：标题
- 中部：主值或主状态
- 下部：状态标签 + 次级说明
- 图标可选，不强依赖

### 8.4 CompactSummaryCard

适用场景：

- 防火墙自定义模式的“入站 / 出站”摘要卡
- 其他局部小摘要区

建议属性：

- `title: string`
- `valueText: string`
- `description?: string`
- `icon?: Resource`
- `iconText?: string`
- `accentStart: string`
- `accentEnd: string`
- `onPress?: () => void`

布局建议：

- 左侧为小图标或字母标记
- 右侧为标签与数值
- 整体高度低于首页模块卡

### 8.5 组件归并建议

建议处理方式如下：

- `StatCard` 逐步下线，迁移为 `MetricSummaryCard`
- `ToolCard` 保留或收敛为更简单的入口卡
- `DirectionCard` 逐步迁移为 `CompactSummaryCard`，或内部基于 `BaseSummaryCard` 重写

说明：

- 不建议在第一阶段就直接删除旧组件。
- 建议采用“新建组件 -> 新页面接入 -> 老组件逐步替换”的平滑迁移方式。

## 9. 数据模型设计

### 9.0 MVVM 总体结构

本次改造建议严格参考当前项目已在外设、工具设置、身份鉴别中逐步形成的 MVVM 结构，不建议继续采用“页面直接拼装多个 Service 返回值”的方式。

建议分层如下：

- `View`
  - 只负责页面布局、组件组合、交互事件绑定、生命周期注册与注销。
  - 不直接编排多源数据。
  - 不直接承载复杂状态聚合逻辑。
- `ViewModel`
  - 负责页面展示状态聚合。
  - 将底层领域状态转换为页面可直接渲染的 UI State。
  - 负责监听注册、状态刷新、跳转所需的卡片数据准备。
- `Repository`
  - 负责持久化数据的读写、缓存、变更通知。
  - 例如日志存储、外设 trace、设备策略等。
- `Service`
  - 负责系统能力调用、规则计算、运行时事件接入、跨仓库组合逻辑。
  - 不直接面向 UI 输出页面状态。
- `Model`
  - 负责领域数据结构、枚举、纯函数格式化与映射。

建议安全总览改造后的结构如下：

```text
DashboardPage (View)
  -> DashboardViewModel (ViewModel)
    -> DashboardSummaryService (可选 Service)
      -> EnterpriseAdminService
      -> FirewallService / FirewallModeService
      -> LogStorageService / LogRuntimeCollectorService
      -> PeripheralTraceRepository / PeripheralRuntimeEventService
```

说明：

- 若首页聚合逻辑较轻，可由 `DashboardViewModel` 直接调用已有 Service / Repository。
- 若后续首页摘要逻辑继续增加，建议再抽 `DashboardSummaryService`，把跨模块聚合逻辑从 ViewModel 中收走。
- 首页卡片组件本身属于 `View` 层，不应该读取 Service。

### 9.1 首页视图模型建议

建议为首页引入独立视图模型，例如：

- `DashboardViewModel`

职责：

- 聚合管理员、防火墙、日志、外设四类摘要状态。
- 屏蔽底层服务差异。
- 向页面输出稳定的卡片数据结构。

建议数据结构：

```ts
export interface DashboardModuleCardState {
  id: 'admin' | 'firewall' | 'log' | 'peripheral'
  title: string
  primaryText: string
  secondaryText: string
  statusText?: string
  statusKind: 'success' | 'warning' | 'neutral'
  accentStart: string
  accentEnd: string
  icon?: Resource
  route?: string
  clickable: boolean
}
```

建议再定义总览页整体状态：

```ts
export interface DashboardOverviewState {
  moduleCards: DashboardModuleCardState[]
  quickEntries: DashboardQuickEntryState[]
}
```

建议 `DashboardViewModel` 采用 `@Observed`，由页面以 `@State` 持有，和当前 `PeripheralViewModel`、`ToolSettingsViewModel` 的使用方式保持一致。

### 9.1.1 DashboardViewModel 职责边界

`DashboardViewModel` 建议负责以下内容：

- 初始化首页所需状态
- 聚合四张模块摘要卡状态
- 聚合次级入口数据
- 注册日志与外设监听
- 在监听回调中刷新自身状态
- 对外暴露稳定的 `state`
- 管理 `initialize()` / `destroy()` 生命周期

`DashboardViewModel` 不建议负责以下内容：

- 直接定义卡片具体布局
- 直接访问 ArkUI 组件 API
- 直接弹窗
- 直接处理页面导航跳转

说明：

- 页面跳转仍由 `DashboardPage` 持有 `onNavigate(route)`，ViewModel 只输出卡片上绑定的 `route` 或 `moduleId`。

### 9.1.2 DashboardState 建议

建议为首页定义一个明确的 View State，而不是页面中散落多个 `@State` 基本字段。

参考结构如下：

```ts
export interface DashboardState {
  loading: boolean
  moduleCards: DashboardModuleCardState[]
  quickEntries: DashboardQuickEntryState[]
}
```

如果后续需要保留入场动画，也建议将动画状态继续留在 `DashboardPage`，不要放入 ViewModel。

原因：

- 动画状态属于纯视图状态，不属于业务状态。
- 这样可以避免 ViewModel 混入 UI 细节。

### 9.1.3 DashboardQuickEntryState 建议

建议补齐次级入口数据结构：

```ts
export interface DashboardQuickEntryState {
  id: 'identity' | 'tool-settings' | 'help-feedback'
  title: string
  icon: Resource
  accentStart: string
  accentEnd: string
  route: string
}
```

## 9.3 MVVM 下的分层归属建议

### 9.3.1 View 层

建议归入 View 层的文件：

- `entry/src/main/ets/views/DashboardPage.ets`
- `entry/src/main/ets/components/BaseSummaryCard.ets`
- `entry/src/main/ets/components/MetricSummaryCard.ets`
- `entry/src/main/ets/components/CompactSummaryCard.ets`

职责：

- 渲染 `DashboardViewModel.state`
- 响应点击事件
- 调用 `onNavigate`
- 管理页面动画与局部 hover 态

### 9.3.2 ViewModel 层

建议归入 ViewModel 层的文件：

- `entry/src/main/ets/viewmodels/DashboardViewModel.ets`

职责：

- 统一聚合首页展示数据
- 订阅日志与外设变化
- 将领域状态转换为卡片状态

### 9.3.3 Service 层

建议保留或新增的 Service：

- 现有：
  - `EnterpriseAdminService`
  - `FirewallModeService`
  - `FirewallService`
  - `LogRuntimeCollectorService`
  - `PeripheralRuntimeEventService`
- 可选新增：
  - `DashboardSummaryService`

`DashboardSummaryService` 建议职责：

- 屏蔽首页多模块聚合细节
- 提供：
  - `buildModuleCards()`
  - `loadOverviewState()`
  - `refreshAdminAndFirewall()`
  - `refreshLogSummary()`
  - `refreshPeripheralSummary()`

说明：

- 如果首页逻辑只在一个 ViewModel 中使用，第一阶段可以不新建该 Service。
- 如果后续还有桌面卡片、顶部状态条、二级概览页复用这些摘要逻辑，就应该尽早抽出。

### 9.3.4 Repository 层

建议明确将以下对象视为 Repository：

- `LogStorageService`
- `PeripheralTraceRepository`
- `PeripheralDevicePolicyRepository`

虽然当前命名中 `LogStorageService` 叫 Service，但从职责上更接近 Repository：

- 负责存储
- 负责查询
- 负责配置读写
- 后续还会负责变更通知

因此在设计与实现时，建议按 Repository 思路约束它：

- 不混入页面展示逻辑
- 不输出 UI 直接文案
- 不管理页面刷新节奏

### 9.3.5 Model 层

建议首页相关的 UI State 类型、模块枚举和纯映射函数放入 `DataModels.ets` 或单独的 Dashboard 模型文件中。

如果首页类型继续增长，建议新增：

- `entry/src/main/ets/models/DashboardModels.ets`

避免 `DataModels.ets` 持续膨胀。

### 9.2 数据来源映射

四张卡的数据建议来自以下服务：

- 管理员卡：
  - `EnterpriseAdminService.probeActivation()`
- 防火墙卡：
  - `FirewallService.getFirewallPolicy()`
  - `FirewallModeService.getCurrentMode()`
- 日志卡：
  - `LogStorageService.getStatistics()`
  - `LogRuntimeCollectorService.getStatus()`
- 外设卡：
  - `PeripheralTraceRepository.getStats()`
  - `PeripheralRuntimeEventService.getStatus()`

## 10. 刷新机制设计

### 10.1 目标

将“定时轮询”改造成“首次加载 + 监听驱动 + 手动兜底刷新”。

### 10.2 日志链路改造

当前问题：

- `LogRuntimeCollectorService` 是事件驱动的。
- 但 `LogStorageService` 没有变更监听接口，UI 只能轮询 `getStatistics()` 和 `queryEntries()`。

建议改造：

在 `LogStorageService` 中新增 change listener 机制。

建议接口：

```ts
interface LogStorageChangeEvent {
  totalCount: number
  addedCount: number
  timestamp: number
}

type LogStorageChangeListener = (event: LogStorageChangeEvent) => void
```

建议新增方法：

- `addChangeListener(listener)`
- `removeChangeListener(listenerId)`
- 在 `addEntries()`
- `clearAll()`
- `updateConfig()` 后触发通知

收益：

- `DashboardPage` 可订阅日志变化刷新首页日志卡。
- `LogManagePage` 可订阅日志变化刷新列表和统计栏。
- 可移除日志页定时轮询。

### 10.3 外设链路改造

外设链路已具备监听基础：

- `PeripheralTraceRepository` 已提供变更监听。
- `PeripheralRuntimeEventService` 已提供 USB 事件监听。
- `PeripheralViewModel` 已基于仓库监听驱动刷新。

建议：

- 首页摘要卡优先订阅 `PeripheralTraceRepository` 变更，而不是直接订阅 runtime event。
- runtime event 监听保留给确有实时事件消费需求的场景。

原因：

- 首页更关心“摘要数据已更新”，不关心单个原始事件细节。
- 以仓库变化为主入口，可减少重复刷新。

### 10.4 管理员与防火墙链路

建议暂不强制引入复杂监听：

- 管理员状态：
  - 页面进入时探测
  - 执行相关操作后主动同步
- 防火墙状态：
  - 页面进入时同步
  - 防火墙开关切换或模式切换后主动同步

原因：

- 这两类状态变化频率低。
- 当前收益不足以支撑额外监听复杂度。

### 10.5 总览页刷新策略

建议总览页刷新流程：

1. `aboutToAppear` 时加载一次摘要状态。
2. 注册日志仓库变化监听。

## 12. 文件级改造建议

### 12.0 按当前目录结构落位

结合当前仓库目录：

- `entry/src/main/ets/components`
- `entry/src/main/ets/models`
- `entry/src/main/ets/services`
- `entry/src/main/ets/viewmodels`
- `entry/src/main/ets/views`

本次改造建议严格沿用现有目录分层，不新增新的顶层目录。

这样做的原因：

- 当前项目已经在外设、身份鉴别、工具设置等模块中形成了可复用的目录约定。
- 延续现有目录结构有利于控制认知成本。
- 后续 Review、测试、维护时更容易按同一模式定位文件。

建议首页改造在当前目录中的落位如下：

```text
entry/src/main/ets/
  components/
    BaseSummaryCard.ets
    MetricSummaryCard.ets
    CompactSummaryCard.ets
  models/
    DashboardModels.ets           (推荐新增)
  services/
    DashboardSummaryService.ets   (可选新增)
    LogStorageService.ets         (增强监听能力)
  viewmodels/
    DashboardViewModel.ets        (推荐新增)
  views/
    DashboardPage.ets             (改造接入)
```

### 12.0.1 为什么不放到子目录

当前 `components` 下仅对外设做了子目录拆分，说明项目暂时采用的是：

- 通用组件直接放 `components/`
- 模块内强专用组件再放模块子目录

因此本次首页摘要卡更适合直接放：

- `entry/src/main/ets/components/BaseSummaryCard.ets`
- `entry/src/main/ets/components/MetricSummaryCard.ets`
- `entry/src/main/ets/components/CompactSummaryCard.ets`

而不是新增 `components/dashboard/`。

原因：

- 这三个组件后续不仅首页会用，日志页、防火墙规则页也会复用。
- 它们更像通用摘要卡组件，而不是首页私有组件。

### 12.0.2 为什么建议新增 DashboardModels.ets

当前 `DataModels.ets` 已承载大量跨模块模型。

如果这次继续把首页卡片状态、入口状态、首页枚举都堆进 `DataModels.ets`，后续会带来两个问题：

- 文件继续膨胀，不利于查找。
- 页面级 UI State 与领域模型混在一起，边界会变模糊。

因此更推荐新增：

- `entry/src/main/ets/models/DashboardModels.ets`

建议该文件承载：

- `DashboardModuleCardId`
- `DashboardModuleCardState`
- `DashboardQuickEntryState`
- `DashboardState`
- 首页卡片状态相关纯类型

如果你希望当前阶段尽量少建文件，也可以先放入 `DataModels.ets`，但从中长期维护看，单独文件更合适。

### 12.0.3 为什么 DashboardViewModel 放在 viewmodels

当前项目已有以下模式：

- `PeripheralViewModel`
- `PeripheralPolicyViewModel`
- `LogStorageSettingsViewModel`
- `ToolSettingsViewModel`
- `IdentitySettingsViewModel`

因此首页也应该延续同样方式，新增：

- `entry/src/main/ets/viewmodels/DashboardViewModel.ets`

建议它成为首页唯一的页面级状态入口。

`DashboardPage.ets` 中不建议再继续保留这些散落状态：

- `todayAuditCount`
- `blockedAuditCount`
- `totalLogCount`
- `adminActivated`
- `logManageListening`
- `peripheralRuntimeListening`

这些应迁移为 `DashboardViewModel.state` 的组成部分。

### 12.0.4 Service 层如何与当前目录兼容

当前 `services` 目录中既有真正的 Service，也有 Repository 风格对象，例如：

- `PeripheralTraceRepository.ets`
- `PeripheralDevicePolicyRepository.ets`
- `FirewallAuthStateRepository.ets`
- `LogStorageService.ets`

因此本次不需要强行改目录命名，只需要在设计和实现上控制职责。

建议做法：

- 保留 `LogStorageService.ets` 文件名不变
- 在内部新增变更监听能力
- 在使用上将其视为“日志存储仓库”

如果首页聚合逻辑后续比较重，可以新增：

- `entry/src/main/ets/services/DashboardSummaryService.ets`

但这不是第一阶段硬性要求。

推荐策略：

- 第一阶段：仅新增 `DashboardViewModel.ets`
- 第二阶段：若 ViewModel 逻辑开始膨胀，再抽 `DashboardSummaryService.ets`

### 12.0.5 View 层与 Page 层的边界

当前项目中：

- `pages/MainPage.ets` 负责应用主容器、路由切换、全局初始化
- `views/*.ets` 负责具体页面

这次改造建议保持不变：

- `MainPage.ets` 不承载首页摘要聚合逻辑
- `DashboardPage.ets` 负责将 ViewModel 状态渲染成页面

`MainPage.ets` 仍然负责：

- 首页路由容器
- 运行时服务初始化
- 页面切换

`DashboardPage.ets` 负责：

- 初始化 `DashboardViewModel`
- 注册和销毁页面级监听
- 渲染卡片和次级入口

### 12.0.6 与现有组件的迁移关系

按当前 `components` 目录，建议迁移关系如下：

- `StatCard.ets`
  - 逐步被 `MetricSummaryCard.ets` 替代
- `DirectionCard.ets`
  - 逐步被 `CompactSummaryCard.ets` 替代，或内部改基于 `BaseSummaryCard`
- `ToolCard.ets`
  - 继续保留，作为非核心快捷入口组件
  - 若后续次级入口区简化明显，再决定是否收敛

这意味着当前目录下不用一次性删除旧组件，而是允许新旧组件并存一段时间。

## 14. 风险与注意事项

### 14.1 管理员卡跳转归属

风险：

- 当前没有明确的独立“管理员管理页”。

建议：

- 若近期不新增页面，先将管理员卡作为状态卡处理。
- 若需要跳转，可暂定跳转到身份鉴别页，但需产品确认语义是否合适。

### 14.2 首页信息过载

风险：

- 若四张卡都塞入过多指标，会重新变成统计面板而不是首页摘要。

建议：

- 每张卡只保留 1 个主状态和 1 到 2 个辅助说明。

### 14.3 日志监听的刷新风暴

风险：

- 高频日志写入可能导致 UI 过于频繁刷新。

建议：

- `LogStorageService` 通知可做轻量节流。
- 页面侧收到监听后可采用合并刷新策略。

### 14.4 旧组件迁移周期

风险：

- 一次性替换所有卡片组件，回归面较大。

建议：

- 先改首页和日志页。
- 防火墙方向卡在第二批统一处理。

## 15. 建议结论

建议本次改造采用“MVVM 分层先立住、首页信息架构再收敛、组件体系后统一、刷新机制最后替换”的顺序推进。

推荐实施顺序如下：

1. 定义首页四张模块卡字段、`DashboardState` 与 `DashboardViewModel`
2. 按 MVVM 边界收敛 `DashboardPage` 责任
3. 落地摘要卡组件体系
4. 改造安全总览页
5. 统一日志页顶部统计卡
6. 补充 `LogStorageService` 监听机制
7. 将监听接入 ViewModel，移除轮询
8. 收敛防火墙方向卡

该方案可以在不推翻现有模块能力的前提下，完成首页 MVVM 分层收敛、首页定位校正、组件复用提升和数据刷新机制升级，属于风险可控、收益明确的渐进式改造方案。
