# 日志管理模块 V2 状态收敛与实施设计

## 1. 文档目的

本文档用于说明 SecurityTool 日志管理模块在 ArkUI V2 下的正式实现方案，重点回答以下问题：

- 日志管理页面在 V2 下应当如何组织状态
- 页面、ViewModel、Repository、Collector 的职责边界是什么
- 监听与刷新应如何接入，才能避免首屏不同步和局部不刷新
- 哪些“中间态、镜像态、桥接计算”必须禁止

本文档是日志管理模块的实施说明，不再描述迁移过程中的临时绕行方案。

---

## 2. 设计目标

日志管理模块最终采用以下目标结构：

```text
单一 ViewModel.state
+ @ObservedV2 + @Trace
+ 页面直接读取 state
+ 页面不保存列表/统计/分页/详情副本
+ 监听只驱动 ViewModel 刷新
```

具体目标如下：

1. 页面只负责渲染和事件转发
2. 所有业务状态统一收口到 `LogManageViewModel.state`
3. 日志仓库监听只更新 ViewModel，不直接操作页面
4. 统计、列表、分页、详情、筛选共用一条状态链
5. 页面不得再承担“状态同步中转层”职责

---

## 3. 核心原则

### 3.1 单一真相源

日志列表页唯一真相源是：

- `LogManageViewModel.state`

页面、子组件、弹窗都只允许读取这棵状态树，或读取点击当下的稳定参数快照。

### 3.2 原位更新

V2 状态更新采用原位更新，不采用整体克隆替换：

- 正确：`this.state.summary.totalCount = snapshot.totalCount`
- 错误：`cloneState() -> setState(nextState)`

### 3.3 禁止镜像态

以下对象不得再作为页面或子组件长期状态存在：

- 列表副本
- 分页副本
- 统计卡数组副本
- 详情实体副本
- `renderVersion`
- `uiRefreshListeners`
- `paginationView`
- `summaryCards`
- `entriesCount`

### 3.4 页面只允许保留非业务局部态

页面局部态只允许承担以下职责：

- Dialog Controller
- hover / pressed 这类纯交互视觉态
- 极少量首屏引导态

即使保留页面局部态，它也不得承载任何业务真值。

## 4. 分层设计

### 4.1 View

文件：

- `entry/src/main/ets/views/LogManagePage.ets`

职责：

- 渲染日志列表页
- 转发用户事件到 ViewModel
- 打开或关闭详情弹窗
- 切换日志列表 / 存储设置模式

不负责：

- 查询日志
- 计算统计
- 分页切片
- 拼接筛选提示
- 维护详情业务状态

### 4.2 ViewModel

文件：

- `entry/src/main/ets/viewmodels/log-manage/LogManageViewModel.ets`

职责：

- 初始化仓库与设置子 ViewModel
- 维护日志列表页唯一状态
- 注册和注销仓库监听
- 根据筛选条件构造查询
- 从快照重建 summary / list / pagination / filter hint / detail
- 控制列表页激活态和自动刷新开关

### 4.3 Repository / Collector / Export

文件：

- `entry/src/main/ets/services/log-manage/repository/LogRepository.ets`
- `entry/src/main/ets/services/log-manage/collector/LogCollectorService.ets`
- `entry/src/main/ets/services/log-manage/export/LogExportService.ets`

职责：

- Repository：提供快照与 change listener
- Collector：提供采集运行状态
- Export：负责导出

它们不直接驱动页面，只由 ViewModel 调用。

## 5. 状态模型设计

文件：

- `entry/src/main/ets/models/LogManageModels.ets`

### 5.1 总状态

```ts
@ObservedV2
export class LogManageState {
  @Trace initialized: boolean = false
  @Trace loading: boolean = false
  @Trace active: boolean = false
  @Trace autoRefreshEnabled: boolean = true
  @Trace mode: LogManageMode = 'list'
  @Trace summary: LogSummaryState = new LogSummaryState()
  @Trace collector: LogCollectorState = new LogCollectorState()
  @Trace filter: LogFilterState = new LogFilterState()
  @Trace pagination: LogPaginationState = new LogPaginationState()
  @Trace list: LogListState = new LogListState()
  @Trace detail: LogDetailState = new LogDetailState()
}
```

语义：

- `initialized`：首轮初始化是否结束
- `loading`：当前是否处于初始化或刷新过程
- `active`：当前日志列表页是否处于激活状态
- `autoRefreshEnabled`：当前是否允许监听回调触发刷新
- `mode`：`list | settings`

### 5.2 统计状态

```ts
@ObservedV2
export class LogSummaryState {
  @Trace totalCount: number = 0
  @Trace todayCount: number = 0
  @Trace blockedCount: number = 0
  @Trace currentShownCount: number = 0
}
```

职责：

- 顶部四张卡片的唯一统计来源

约束：

- 页面不允许再基于 `list.pagedEntries` 现场重新算第二套统计

### 5.3 采集状态

```ts
@ObservedV2
export class LogCollectorState {
  @Trace running: boolean = false
  @Trace auditReady: boolean = false
  @Trace crashReady: boolean = false
  @Trace pendingCount: number = 0
  @Trace degraded: boolean = false
  @Trace degradedReason?: 'audit_failed' | 'crash_failed' | 'partial'
}
```

职责：

- 表达运行态、退化态、空态文案辅助语义

### 5.4 筛选状态

```ts
@ObservedV2
export class LogFilterState {
  @Trace eventTypeIndex: number = 0
  @Trace resultIndex: number = 0
  @Trace keyword: string = ''
  @Trace hintMessage: string = ''
}
```

职责：

- 表达筛选输入和筛选提示

约束：

- `eventTypeIndex`、`resultIndex` 只表示 UI 索引
- 不直接混入业务枚举值

### 5.5 分页状态

```ts
@ObservedV2
export class LogPaginationState {
  @Trace currentPage: number = 1
  @Trace pageSizeIndex: number = 0
  @Trace pageInputText: string = '1'
  @Trace totalPages: number = 1
}
```

职责：

- 分页输入与分页显示的唯一状态来源

约束：

- `pageInputText` 只表示输入框文本
- 不额外维护 `currentRangeText` 或分页展示对象

### 5.6 列表状态

```ts
@ObservedV2
export class LogListState {
  @Trace pagedEntries: LogEntry[] = []
}
```

职责：

- `pagedEntries`：当前页结果

说明：

- 总数统一由 `summary.totalCount` 提供
- 当前页显示数量统一由 `summary.currentShownCount` 提供
- 列表状态不再保存第二份“完整筛选结果”镜像

### 5.7 详情状态

```ts
@ObservedV2
export class LogDetailState {
  @Trace visible: boolean = false
  @Trace entryId: string = ''
}
```

职责：

- 记录当前是否打开详情
- 记录当前详情对应的日志 id

约束：

- 详情实体本身不进入状态树
- 弹窗使用点击当下的稳定参数快照展示

## 6. 监听与刷新机制

### 6.1 监听接入点

日志仓库监听只在 ViewModel 内注册：

```text
LogRepository.addChangeListener
  -> ViewModel.refreshIfActive()
```

页面不直接订阅仓库。

### 6.2 激活态控制

列表页进入后：

- `state.active = true`
- `state.autoRefreshEnabled = true`

切到设置页或页面离开后：

- `state.active = false`
- `state.autoRefreshEnabled = false`

这样仓库虽然仍然可能有新日志，但不会在非激活态持续驱动列表刷新。

### 6.3 刷新入口

正式刷新入口只有两个：

- `refreshAll()`
- `refreshIfActive()`

其中：

- `refreshAll()` 负责真正重建状态
- `refreshIfActive()` 负责按激活态做短路判断

## 7. 数据流设计

### 7.1 初始化数据流

```text
LogManagePage.aboutToAppear
  -> viewModel.initialize(context)
  -> repository.init(context)
  -> registerStorageListener()
  -> settingsViewModel.load()
  -> refreshAll()
  -> rebuildStateFromSnapshot()
  -> 页面读取 state 渲染
```

### 7.2 监听刷新数据流

```text
新日志入库
  -> repository change listener 触发
  -> viewModel.refreshIfActive()
  -> refreshAll()
  -> rebuildStateFromSnapshot()
  -> applyCollectorState()
  -> applySummaryState()
  -> applyListState()
  -> applyPaginationState()
  -> rebuildFilterHint()
  -> syncDetailState()
  -> 页面自动重绘
```

### 7.3 用户交互数据流

#### 筛选

```text
页面 onSelect / onChange
  -> viewModel.setEventTypeIndex / setResultIndex / setKeyword
  -> resetToFirstPage()
  -> refreshAll()
```

#### 分页

```text
页面上一页 / 下一页 / 跳转 / 修改页大小
  -> ViewModel 更新 pagination
  -> rebuildStateFromSnapshot()
  -> 页面重绘
```

#### 详情

```text
点击日志项
  -> viewModel.openDetail(entry.id)
  -> 页面使用当前 entry 快照打开 CustomDialog
  -> 关闭时 closeDetail()
```

## 8. ViewModel 内部实现约束

### 8.1 推荐内部步骤

`refreshAll()` 内部统一走以下步骤：

1. `buildQuery()`
2. `repository.snapshot(query)`
3. 计算分页切片
4. `applyCollectorState()`
5. `applySummaryState()`
6. `applyListState()`
7. `applyPaginationState()`
8. `rebuildFilterHint()`
9. `syncDetailState()`

### 8.2 禁止事项

ViewModel 内部禁止：

- `cloneState()`
- `setState(nextState)`
- 返回页面镜像展示对象
- 提供 `summaryCards`、`paginationView`、`currentRangeText` 这类展示态

### 8.3 可接受的辅助方法

允许保留以下纯逻辑辅助方法：

- `normalizeFilterIndex`
- `normalizePageSizeIndex`
- `normalizePage`
- `calcTotalPages`
- `mapResultIndexToValue`

这些方法不保存状态，只服务于 ViewModel 内部计算。

---

## 9. 页面实现约束

### 9.1 页面允许做什么

页面允许：

- 读取 `viewModel.state`
- 调用 ViewModel 公开方法
- 管理 `CustomDialogController`
- 维护 options 常量数组

### 9.2 页面禁止做什么

页面禁止：

- 保存 `entries` 副本
- 保存 `pagedEntries` 副本
- 保存 `summary` 副本
- 保存 `detailEntry` 副本
- 保存 `paginationView` 副本
- 通过 `getSummaryXxx()` 这类方法再包一层统计读取

### 9.3 关于 `initialized`

首屏 loading 的正式控制建议使用：

- `state.initialized`
- `state.loading`

页面不再保留 `listReady` 这类额外引导态。

---

## 10. 子组件接入约束

### 10.1 子组件传参原则

优先传以下两类内容：

- `viewModel.state` 的直接子状态
- 点击当下的基础值

避免：

- 大量包装后的展示对象
- 为了兼容 V1 而做的桥接态

### 10.2 禁止的桥接模式

禁止在关键刷新链上使用以下模式：

- `@Builder actionButton(label, enabled, onTap)` 这种先把状态值转成普通参数再转发
- `getSummaryTotalCount()` 这类中间取值函数
- 页面先把状态算成卡片数组，再让子组件消费

### 10.3 推荐方式

- 统计卡直接读 `state.summary`
- 列表直接读 `state.list`
- 分页直接读 `state.pagination`
- 空态提示直接读 `state.filter.hintMessage`

---

## 11. 详情弹窗方案

日志详情统一采用：

```text
CustomDialogController
+ @CustomDialog
+ 点击当下 entry 快照
```

原因：

- 详情展示需要稳定数据快照
- 列表仍可能在后台刷新
- 不应让弹窗在显示过程中继续依赖实时列表二次解析

因此：

- 状态树只记录 `entryId`
- 弹窗展示使用点击时的 `entry`

---

## 12. 验收标准

满足以下条件，视为日志管理模块 V2 状态收敛完成：

1. 页面不再保存列表、统计、分页、详情镜像态
2. 仓库监听只驱动 ViewModel 刷新
3. 顶部统计卡在当前页停留时可即时刷新
4. 从安全总览或其它 tab 首次进入日志页时，分页启用态正常
5. 切换到设置页后自动刷新暂停，返回列表后恢复
6. 详情弹窗打开与关闭不依赖页面镜像副本

---

## 13. 当前实施建议

后续日志管理若继续微调，优先级如下：

1. 保持 `state` 语义稳定，不再新增页面侧桥接态
2. 若新增统计或提示，先判断是否属于 `summary / filter / collector`
3. 若新增交互弹层，优先沿用 `CustomDialogController + @CustomDialog`
4. 若后续还要扩展其它页面到 V2，应先清宿主页和公共组件的 V1 残留，再迁具体业务页
