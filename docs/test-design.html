SecurityTool 自动化测试设计方案

借鉴 KuiTest 设计理念，构建分层测试架构

HarmonyOS / ArkTS
设计参考
本方案借鉴美团 KuiTest 的核心设计理念 —— "无规则设计" + "AI 理解 UI 交互常识"， 针对 HarmonyOS/ArkTS 企业应用特点，设计出适合 SecurityTool 项目的分层测试架构。

核心设计理念

!
传统测试痛点
KuiTest 要解决的问题
  • 规则维护成本高
  • UI 修改导致测试用例失效
  • 测试覆盖率低
  • 人工编写脚本难以适应快速迭代
*
KuiTest 核心思路
无规则 + AI 理解
  • 组件功能识别：识别可交互组件的功能
  • 交互响应验证：对比预期与实际结果
  • 大模型泛化：无需针对特定应用适配
  • 像素对比 + 语义理解

分层测试架构

E2E E2E 测试 (UI 遍历 + 快照)
基于 AI 的交互验证 / 截图对比 / 页面快照回归
Component 组件测试
StatCard / SideBar / Dialog 等组件的渲染与交互
Service 服务层测试
FirewallService / PeripheralService / IdentityService
Unit 单元测试 (Model + Utils)
DataModels / 解析函数 / 格式化函数 / 业务逻辑

单元测试层

U
纯函数与数据模型测试
不依赖 UI，快速执行
模块 测试内容
DataModels.ets parseUsbDeviceId(), formatUsbDeviceId(), getRuleTypeLabel() 等
FirewallService countByType(), countByDirection() 等纯逻辑方法
describe('DataModels', () => { it('parseUsbDeviceId should parse valid format', () => { let result = parseUsbDeviceId('VID:2B7E PID:B5A2') expect(result?.vendorId).assertEqual(0x2B7E) expect(result?.productId).assertEqual(0xB5A2) }) it('parseUsbDeviceId should return null for invalid', () => { let result = parseUsbDeviceId('invalid') expect(result).assertNull() }) })

服务层测试

S
业务逻辑测试 + Mock 系统 API
测试不依赖真实设备/权限
// services/__mocks__/MockFirewallService.ets export class MockFirewallService { private static mockRules: NetFirewallRule[] = [] static async getAllRules(): Promise<NetFirewallRule[]> { return this.mockRules } static setMockData(rules: NetFirewallRule[]) { this.mockRules = rules } }
可模拟边界条件
不依赖真实设备
快速执行

组件测试层

C
组件快照测试 (Snapshot)
验证组件渲染和交互，减少 UI 修改影响
// __tests__/components/StatCard.test.ets describe('StatCard', () => { it('should render with correct props', () => { let snapshot = renderComponent(StatCard, { label: '活跃规则', value: '12', gradientStart: '#0A84FF', gradientEnd: '#00C6FF' }) expect(snapshot).toMatchSnapshot('StatCard-basic') }) it('should apply dark theme styles', () => { setTheme(ThemeMode.DARK) let snapshot = renderComponent(StatCard, { ... }) expect(snapshot).toMatchSnapshot('StatCard-dark-theme') }) })

E2E 测试层（借鉴 KuiTest 核心）

1
页面快照回归测试
截图对比，检测 UI 破坏性变更
const PAGES = [ { route: 'dashboard', name: 'DashboardPage' }, { route: 'firewall', name: 'FirewallPage' }, { route: 'peripheral-manage', name: 'PeripheralPage' } ] for (let page of PAGES) { let screenshot = await takeScreenshot() let diff = compareWithBaseline(screenshot) expect(diff.mismatch).toBeLessThan(0.1) }
2
交互流程验证
定义关键用户流程，自动验证
// 添加 IP 规则流程 await navigateTo('firewall') await tap({ text: 'IP 规则' }) await tap({ id: 'add-rule-btn' }) await input({ id: 'rule-name' }, 'Test') await tap({ text: '确定' }) await waitFor({ text: 'Test Rule' })
3
组件功能识别表（借鉴 KuiTest）
为关键组件定义"预期行为"，测试时自动验证
// e2e/expectations/ComponentExpectations.ts export const COMPONENT_EXPECTATIONS: Map<string, Expectation[]> = { 'toggle-firewall': [ { action: 'tap', expect: 'state_change', from: 'off', to: 'on' }, { action: 'tap', expect: 'api_call', method: 'setFirewallEnabled' } ], 'add-rule-btn': [ { action: 'tap', expect: 'dialog_open', dialogId: 'AddRuleDialog' } ], 'sidebar-item': [ { action: 'tap', expect: 'navigation', toPage: '{dynamic}' } ] }

减少 UI 修改影响的策略

语义化选择器
使用 testId 而非 CSS 选择器定位元素
布局变化不影响测试
快照容差
像素对比允许小范围差异 (0.1%)
字体/间距微调不触发失败
分层隔离
单元测试不依赖 UI，服务测试 Mock API
80% 测试不受 UI 影响
变更检测
快照 diff 时高亮变更区域，人工确认
快速定位问题
组件契约
定义组件 Props 接口，测试输入输出
内部重构不破坏测试

实施优先级建议

优先级 阶段 内容 收益
P0 立即 DataModels.ets 工具函数单元测试 覆盖解析/格式化逻辑
P1 短期 Service Mock + 服务层测试 业务逻辑覆盖
P2 中期 页面截图快照回归 检测 UI 破坏性变更
P3 长期 E2E 关键流程自动化 端到端质量保障

与 KuiTest 的关键差异

🤖 KuiTest (大模型方案)
  • 大模型推理组件功能
  • 无规则，AI 自动判断交互正确性
  • 计算成本高（需 GPU 资源）
  • 适合大规模 App 遍历测试
  • 泛化能力强，跨应用复用
✅ SecurityTool 方案
  • 人工定义组件预期行为表
  • 规则驱动 + 快照对比
  • 轻量级，可本地运行
  • 适合功能明确的企业应用
  • 维护成本可控
