# 防火墙全局策略 policy-only 规格唯一真相

> 状态：Draft  
> 日期：2026-04-14  
> 适用范围：仅用于“防火墙管理 > 自定义模式 > 全局策略 / 用户模式切换”规格改造  
> 唯一判定依据：本文档

## 0. 背景+目的

当前防火墙模块已经完成“自定义规则按用户生效”能力，规则本身已经可以通过 `userIds` 表达“哪些用户命中这条规则”。

与此同时，现有“全局策略 / 用户模式切换”链路仍沿用旧规格，实际语义仍是：

- 读取共享规则模板
- 按用户和模式筛选规则
- 清理旧规则
- 设置 policy
- 重新下发新规则

这会导致以下问题：

1. 用户级控制真相不唯一
2. 全局策略和规则模板形成不必要耦合
3. 用户模式切换链路持续堆积只服务旧规格的规则编排代码

本次规格调整的目的，是将“全局策略 / 用户模式切换”从“规则式模式切换”重定义为“policy-only 模式切换”，并保留以下约束：

1. 保留 MVVM 架构
2. 保留 PIN 码认证
3. 当前不要求页面回显用户当前模式
4. 本地保存每个用户的历史模式记录
5. 后续全局防火墙重新开启时，按历史模式一次性写完整 policy 恢复

## 1. 规格结论

### 1.1 全局策略的新职责

“全局策略 / 用户模式切换”只负责设置目标用户的默认防火墙策略，不再承担规则筛选、规则清理、规则下发和模板联动职责。

可理解为：

- 规则层：负责精细化控制，规则本身已经支持按用户生效
- 策略层：负责默认行为，即默认允许或默认拒绝

### 1.2 policy-only 的定义

本次规格中的 `policy-only` 固定指：

1. 切换前执行 PIN 认证
2. 根据目标模式生成目标用户完整 `NetFirewallPolicy`
3. 调用 `setNetFirewallPolicy(userId, policy)`
4. 成功后保存本地 `userId -> mode` 历史模式记录

不再允许以下行为进入“全局策略 / 用户模式切换”链路：

1. 读取共享规则模板
2. 按 allowlist / denylist 筛选规则
3. 清理用户现有规则
4. 给用户重新下发规则
5. 因规则模板变化自动重放用户模式

### 1.3 白名单 / 黑名单模式语义

本次规格中，“白名单模式 / 黑名单模式”不再解释为规则筛选语义，而是解释为默认 policy 语义。

固定语义如下：

- 白名单模式：只设置完整 policy，默认入站允许，默认出站允许
- 黑名单模式：只设置完整 policy，默认入站拒绝，默认出站拒绝

### 1.4 页面回显边界

当前阶段不要求页面回显某个用户的当前模式。

因此：

- 不需要为了展示态反推当前模式
- 用户模式切换链路只需关注提交与持久化

### 1.5 历史模式记录

每个用户的历史模式必须本地持久化保存。

建议继续建模为：

- `userId -> mode`

其职责固定为：

1. 记录某个用户最近一次成功设置的模式
2. 为后续全局防火墙重新开启时恢复历史 policy 提供输入

明确不再承担以下职责：

1. 规则模板变更后的自动重下发契约
2. 规则级绑定关系

### 1.6 后续全局防火墙重新开启的恢复规则

后续当首页全局防火墙重新开启时，应根据每个用户的历史模式，一次性写完整 policy 恢复状态，而不是先开启后再额外补写模式。

恢复语义如下：

- 历史为白名单模式：
  - 一次性写完整 policy，`isOpen=true`，默认入站允许，默认出站允许
- 历史为黑名单模式：
  - 一次性写完整 policy，`isOpen=true`，默认入站拒绝，默认出站拒绝

## 2. 不受本轮影响的能力边界

以下能力不属于本次规格调整范围，不应被误伤：

1. 首页 `public / private / custom` 三种模式切换
2. 防火墙开关切换后重新下发历史规则
3. 自定义模式规则的新增 / 删除 / 更新
4. 自定义规则模板持久化与应用
5. 规则冲突判断、规则展示和规则编辑能力

## 3. 腐败代码判定原则

本轮中的“腐败代码”不是泛指旧代码，而是指：

1. 与本次新规格直接冲突的代码
2. 只服务旧规格、已失去业务必要性的代码
3. 制造“规则用户作用域”和“全局策略二次筛选作用域”双重真相的代码
4. 继续保留会误导维护者，以为全局策略仍然是规则集模式切换的代码
5. 仅为旧规格联动链路存在的桥接、回滚和重放代码

认定证据固定看三类：

- 规格证据：本文件中的新规格结论
- 代码证据：实际代码仍在读取规则、筛规则、删规则、加规则、重放规则
- 职责证据：该代码只在“用户模式 = 规则集模式”这一旧前提下成立

## 4. 本轮明确认定的腐败代码范围

### 4.1 用户模式切换链路中的规则依赖

以下逻辑属于腐败代码：

1. 用户模式切换时读取共享规则模板
2. 用户模式切换时按 allowlist / denylist 筛选规则
3. 用户模式切换时清理受管规则并重新下发规则
4. 用户模式切换失败时回滚规则

### 4.2 规则模板变更后的用户模式联动

以下逻辑属于腐败代码：

1. 自定义规则保存成功后，自动按用户模式记录重放用户模式
2. 把本地 `userId -> mode` 记录当成规则模板联动契约

## 5. 本轮不直接删除的能力

以下能力虽然可能被旧链路复用，但本轮不应直接按“腐败代码”整体删除：

1. 首页主模式切换所需的规则构造与应用能力
2. 自定义规则主链路能力
3. 仍可能服务历史规则恢复的底层规则能力

说明：

- 本轮只要求它们退出“全局策略 / 用户模式切换”链路
- 是否在后续继续删除，取决于后续是否仍有合法调用点

## 6. 目标调用链

本轮改造后的目标调用链固定为：

```text
ViewModel.applySelectedMode()
-> FirewallService.applyUserMode(context, userId, mode)
-> FirewallUserDispatchService.applyUserMode(context, userId, mode)
-> FirewallAuthService.executeUserModeApply(context, userId, mode, applyAction)
-> FirewallStore.applyUserMode(context, userId, mode)
-> setNetFirewallPolicy(userId, targetPolicy)
-> save userId -> mode
```

其中：

- 不读取 custom rules
- 不筛规则
- 不删规则
- 不加规则
- 不因规则模板保存而联动重放用户模式

## 7. 验收信号

完成后应满足：

1. 用户模式切换时，不再读取共享规则模板
2. 用户模式切换时，不再筛规则、不删规则、不加规则
3. 用户模式切换前仍必须做 PIN 认证
4. 用户模式切换成功后，本地会保存 `userId -> mode`
5. 自定义规则保存后，不再自动重放用户模式
6. 首页三种模式切换不受影响
7. 自定义模式规则新增 / 删除 / 更新不受影响
8. 后续首页全局重新开启时，可基于历史模式一次性恢复完整 policy
