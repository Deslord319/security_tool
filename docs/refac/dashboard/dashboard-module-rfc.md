# RFC v1：安全总览模块收口与演进设计

> 状态：Draft
> 日期：2026-04-08
> 关联设计：`docs/03-模块设计/安全总览组件设计说明.md`
> 本文定位：安全总览模块的执行层 RFC，聚焦模块边界、演进目标、Story / Round 拆分与验收信号

## 1. 背景

安全总览已经具备首页展示、快捷入口和基础刷新能力，但当前它仍承担两个容易失控的职责：

- 既是首页展示层，又部分承担跨模块状态聚合逻辑；
- 需要消费日志、外设、防火墙等多个模块的摘要信息，但缺少统一聚合边界说明。

这会导致首页容易变成“跨模块临时编排区”，后续一旦任意模块状态模型调整，首页最先失稳。

## 2. 目标

1. 固定安全总览作为“摘要聚合层”而不是业务真相源。
2. 统一摘要卡模型、快捷入口模型和刷新入口。
3. 明确首页对其它模块的依赖方式，避免页面直接拼装各模块 service。
4. 为后续新增模块摘要卡保留稳定扩展位。

## 3. 范围

### 3.1 In Scope

- `entry/src/main/ets/views/dashboard/**`
- `entry/src/main/ets/viewmodels/dashboard/**`
- `entry/src/main/ets/components/CompactSummaryCard.ets`
- `entry/src/main/ets/components/MetricSummaryCard.ets`
- `entry/src/main/ets/components/ToolCard.ets`
- 与首页摘要聚合直接相关的模型和常量

### 3.2 Out of Scope

- 其它业务模块的底层状态重构
- 侧边栏路由体系调整
- 首页视觉风格重做
- 帮助与反馈页面改造

## 4. 当前问题

1. 首页刷新来源多，但刷新优先级与合并口径没有单独沉淀。
2. 摘要卡状态生成逻辑容易与业务模块内部语义耦合。
3. 快捷入口和摘要卡都能导航，但导航元数据可能重复维护。
4. 首页如果继续吸收新逻辑，后续会退化成“大而全”页面。

## 5. 目标设计

### 5.1 固定调用链

```text
DashboardPage
  -> DashboardViewModel
    -> Dashboard module summary providers / adapters
      -> 各模块已收口的 ViewModel / Service 只读能力
```

### 5.2 首页职责

- 展示模块摘要卡
- 展示快捷入口
- 响应页面显示时机做轻量刷新
- 将点击动作转为标准路由跳转

禁止：

- 直接执行防火墙、外设、身份等模块的写操作
- 在页面层拼装跨模块业务规则
- 长期持有其它模块完整业务状态

### 5.3 摘要模型约束

建议把首页所有卡片收敛到统一模型：

```ts
interface DashboardModuleSummary {
  moduleId: string
  title: string
  status: 'healthy' | 'warning' | 'risk' | 'inactive'
  summaryText: string
  actionRoute: string
  updatedAt?: number
}
```

要求：

- 每个模块只给首页暴露“可展示摘要”，不暴露内部编辑态。
- 首页只消费摘要，不反向修改模块内部状态。

## 6. Story 拆分

### Story A：摘要聚合入口收口

- 目标：
  - 统一 DashboardViewModel 的摘要装配入口
  - 把模块摘要生成逻辑收敛到单点
- 验收：
  - 首页初始化与刷新都走统一入口
  - 页面不再分散读取多处摘要状态

### Story B：快捷入口与摘要卡元数据去重

- 目标：
  - 统一导航元数据定义
  - 避免快捷入口和摘要卡分别维护同一路由信息
- 验收：
  - 新增模块入口时最多只改一处元数据定义

### Story C：跨模块刷新边界明确化

- 目标：
  - 定义日志监听、外设监听、页面显示校准三类刷新触发器
  - 明确哪些是增量刷新，哪些是全量校准
- 验收：
  - 刷新来源有固定说明
  - 首页不会因为任一监听抖动而全量重建全部状态

### Story D：首页扩展位规范

- 目标：
  - 为新增模块摘要卡或快捷卡保留规范
  - 固定卡片排序、状态文案和降级写法
- 验收：
  - 文档中有稳定扩展模板
  - 新模块接入不需要重写首页架构

## 7. Round 划分

### Round 1：摘要聚合收口

- 处理 Story A
- 输出统一摘要装配入口
- 不改视觉样式

### Round 2：导航元数据统一

- 处理 Story B
- 收敛快捷入口和摘要卡导航定义

### Round 3：刷新机制定稿

- 处理 Story C
- 补刷新入口、节流口径与页面显示校准说明

### Round 4：扩展位与测试收口

- 处理 Story D
- 补齐文档、最小测试建议与接入清单

## 8. 主要风险

1. 首页如果直接依赖其它模块内部 VM 细节，会放大跨模块耦合。
2. 日志和外设监听都可能带来高频刷新，必须限制首页只消费摘要级变更。
3. 若摘要文案在首页本地硬编码，后续业务语义变化时会与模块真实状态脱节。

## 9. 验收信号

- 首页只消费摘要态，不消费其它模块完整编辑态。
- 快捷入口与摘要卡的路由定义一致。
- 页面显示、监听刷新和手动刷新具有固定触发口径。
- 新增模块摘要卡时，不需要修改首页核心调用链。

## 10. 后续建议

1. 若后续继续推进首页收口，建议新增 `dashboard summary provider` 层，而不是把逻辑回塞到页面。
2. 若模块数量继续增加，建议引入统一摘要注册表，避免首页手写分支持续扩散。
