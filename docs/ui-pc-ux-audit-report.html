SecurityTool UI PC化审查汇报

聚焦 2in1 真机视觉与交互不匹配问题，给出可执行的分阶段优化路线

管理层摘要（3分钟）

  • 当前 UI 主要来自 Web 原型迁移，已具备功能完整性，但信息密度与桌面交互效率不足。
  • 最大影响点集中在：视觉密度、导航语义、窗口态适配与关键操作路径。
  • 建议采用 3 阶段改造：先低风险快改，再结构统一，最后体验强化；预计总体评分从 61 提升至 91。

结论 1：参考链路已清晰

ArkTS 实现与 security-app 原型结构高度一致，当前差异主要在“PC化细节”而非功能缺失。

结论 2：问题可控且可分阶段

Phase 1 可在 1-2 周内完成核心快改，先提升可读性和效率，尽快改善真机观感。

结论 3：ROI 明确

统一模板后可显著降低新增页面设计成本与返工概率，提升交付稳定性与演示效果。

2. 核心结论（3条）

当前模板“可用”，但不“PC化”

布局结构正确（侧栏 + 顶栏 + 内容），但控件密度、层级节奏、操作入口仍偏移动端。

影响等级：高

业务影响：关键页面在 2in1 上信息承载效率偏低，管理类场景操作成本偏高。

证据链完整，改造边界清晰

从 README、原型文件到 ArkTS 页面均可追溯，具备“基线重构”条件。

影响等级：中

业务影响：改造无需推翻业务逻辑，技术风险可控。

建议采用分阶段路线

先快改可视问题，再做组件体系与窗口态策略，避免一次性重构风险。

落地风险：低-中

业务影响：可保持版本节奏，持续释放改进收益。

7. 投入产出与风险

7.1 ROI 预估

指标 当前基线 Phase 1 目标 Phase 3 目标 统计口径
管理任务平均完成时长 100% 92%（下降约8%） 85%（下降约15%） 典型任务：日志筛选、策略配置、规则维护
新页面设计与联调成本 100% 90%（下降约10%） 75%（下降约25%） 采用统一模板、密度规范、组件约束
评审返工率 100% 88%（下降约12%） 70%（下降约30%） 按迭代中 UI 反馈回退次数统计
演示场景可感知专业度 基线 明显提升 稳定达到企业桌面工具标准 由项目评审会打分（1-5）

7.2 风险矩阵与回滚策略

风险 A：一次性样式重构影响范围过大

应对：采用 Phase 方式灰度推进，每阶段可回退至前一稳定样式变量集。

风险 B：2in1 多窗口态下布局错位

应对：引入窗口宽度分档与关键断点验收，在每阶段执行窗口态回归。

风险 C：主题适配出现深浅模式不一致

应对：在 Phase 2 建立颜色 token 对照表，保留统一主题管理入口。

风险 D：团队执行口径不统一

应对：以本报告附录的“证据索引+验收表”为统一评审基线。

7.3 回滚策略（明确）

  • Phase 1 回滚：仅回退样式常量和布局参数，不涉及业务逻辑。
  • Phase 2 回滚：保留新组件接口，回退到旧模板映射层。
  • Phase 3 回滚：按功能开关停用增强体验特性，主流程不受阻断。

8. 里程碑与验收标准（按周）

第1周

Phase 1 启动与基线参数快改

完成字体/间距/侧栏/按钮/表格密度参数调整；输出前后对比截图。

第2周

Phase 1 验收

1366x768 与 1920x1080 通过可读性检查；关键任务路径可操作性提升。

第3-4周

Phase 2 组件与模板统一

统一页面骨架、卡片/表格/工具栏规范；完成桌面分档密度方案。

第5-6周

Phase 3 体验强化

完成窗口态策略、主题一致性、可访问性检查与汇报复盘。

验收测试场景（执行清单）

  • 可读性测试：1920x1080 首屏 30 秒内可读完摘要；1366x768 目录与正文无重叠。
  • 内容完整性：每条核心结论具备至少一条 repo 证据；每阶段含任务、工期、验收标准。
  • 链接有效性：目录锚点可达；官方外链可访问。
  • 打印测试：A4 分页表头不截断，灰度打印仍可辨识。
  • 离线测试：断网打开样式完整，交互（折叠/跳转/评分动画）可用。
