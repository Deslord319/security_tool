# AI 执行原则

本文档定义 SecurityTool 项目中 AI 编码助手执行开发任务时必须遵循的通用原则。本文档不替代 `AGENTS.md`、模块设计文档、测试文档和构建签名流程；当规则存在重叠时，以更具体的项目规则为准。

## 1. 先明确影响面

修改文件前，先明确本次任务是否影响模块行为、用户可见交互、状态流、权限、系统能力调用、数据模型、持久化结构、服务职责或模块级验收口径。

如果影响上述任一内容，必须先阅读并更新对应模块设计文档，再修改代码、测试或配置。

能通过仓库上下文、设计文档、测试、脚本或构建配置确认的问题，应先查证并推进；只有涉及需求取舍、用户可见行为决策、权限/签名策略、数据结构调整或模块设计结论变更时，才向用户确认。

## 2. 简洁优先

只实现当前目标需要的最小改动。

不要为了单次需求新增泛化抽象、配置层、复杂状态机或跨模块公共能力。只有当抽象能明显减少真实重复、降低复杂度，并符合现有代码风格时，才允许新增。

对 HarmonyOS、MDM、签名、权限和设备能力相关逻辑，应优先沿用项目已有封装、流程和文档约束，不引入额外工具链或平行实现。

## 3. 精准修改

每一处改动都必须能对应到用户请求、已确认缺陷、设计文档要求或验证需要。

不顺手重构无关模块，不改无关格式，不扩大影响面。发现无关问题时，先记录或说明；除非用户确认，或者该问题直接阻塞当前目标，否则不直接处理。

精准修改不等于只改代码。如果任务改变了模块行为、状态、交互、权限、系统能力调用或验收口径，模块设计文档更新就是必要改动，并且必须先于代码修改。

## 4. 目标驱动闭环

非平凡任务开始前，应明确成功标准和验证方式。

任务结束时，必须说明验证结果；如果未运行验证，需要说明原因。如果未更新模块设计文档，需要说明本次改动为何属于可不更新场景。

验证方式应按风险选择，优先使用项目已有脚本、单测、构建命令、签名流程、设备测试或手工验收矩阵，不为单次任务新增重复验证链路。

## 5. 错误处理边界

开发态和交付态前置问题不应进入业务代码兜底，包括签名错误、权限清单不一致、`bundleName` 不一致、p7b 过期、构建工具链缺失、HAP 安装失败、未按部署流程激活企业管理员等。

这些问题应由文档、脚本、CI、构建签名流程和部署验收阻断，不应在页面、ViewModel、Service 或 Repository 中设计复杂恢复逻辑。

运行态系统能力调用失败应按模块设计处理，包括设备策略接口失败、设备状态不支持、外设状态变化、审计日志读取失败、账号策略设置失败等。处理原则是明确失败结果、保留必要日志、给出最小必要反馈，不吞错、不伪装成功、不添加投机性兜底。
