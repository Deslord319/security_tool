四层测试架构设计

每一层解决什么问题？为什么需要分层？如何确保 UI 修改不破坏已有功能？

测试金字塔

E2E 测试 ~5% 慢 / 高成本
组件测试 ~15%
服务层测试 ~30%
单元测试 ~50% 快 / 低成本
核心原则

越底层的测试数量越多、执行越快、成本越低。越顶层的测试越接近真实用户场景，但维护成本高。
目标：80% 的测试不依赖 UI，确保 UI 修改时大部分测试仍然有效。

分层详解

1
单元测试 (Unit Tests)
测试最小可测单元：纯函数、工具方法、数据转换
⚠ 解决的问题
  • 工具函数逻辑错误（如 parseUsbDeviceId 解析失败）
  • 枚举映射不完整（新增枚举值忘记更新 switch）
  • 边界条件处理不当（空值、异常输入）
  • 格式化输出不一致（VID:001 vs VID:0001）
  • 重构时意外破坏已有逻辑
✔ 提供的保障
  • 验证每个函数的输入/输出契约
  • 覆盖正常路径 + 边界条件 + 异常输入
  • 毫秒级执行，即时反馈
  • 无需设备，本地即可运行
  • 重构时的安全网

2
服务层测试 (Service Tests)
测试业务逻辑：数据处理、状态管理、API 调用流程
⚠ 解决的问题
  • 业务逻辑错误（规则过滤、统计计算错误）
  • API 调用顺序错误（先删除后查询导致崩溃）
  • 异常处理不当（API 失败时 UI 卡死）
  • 状态不一致（本地状态与远程不同步）
  • 无法在无设备环境下验证业务逻辑
✔ 提供的保障
  • 通过 Mock 隔离系统 API 依赖
  • 模拟各种业务场景（空数据、大量数据）
  • 模拟 API 失败，验证错误处理
  • 验证业务流程的正确性
  • CI 环境可运行，无需真机

组件测试 (Component Tests)
测试 UI 组件：渲染正确性、交互行为、状态变化
⚠ 解决的问题
  • 组件渲染异常（Props 变化导致崩溃）
  • 交互行为失效（按钮点击无响应）
  • 条件渲染错误（某状态下元素缺失）
  • 主题切换时样式异常
  • 组件重构后外观意外变化
✔ 提供的保障
  • 快照测试检测意外 UI 变化
  • 验证组件 Props 到渲染的映射
  • 测试交互触发的状态变化
  • 验证条件渲染逻辑
  • 确保主题切换不破坏布局

4
E2E 测试 (End-to-End Tests)
测试完整用户流程：从操作到结果的端到端验证
⚠ 解决的问题
  • 页面间导航断裂（路由配置错误）
  • 完整业务流程失效（添加规则流程中断）
  • 页面布局整体崩溃（新功能影响旧页面）
  • 多步骤操作的状态传递错误
  • 回归测试覆盖不足
✔ 提供的保障
  • 页面快照回归，检测视觉破坏
  • 模拟真实用户操作流程
  • 验证关键业务路径端到端可用
  • 截图对比发现 UI 意外变更
  • 作为发布前的最后一道防线

分层对比总结

层级 核心问题 测试对象 受 UI 修改影响 执行速度
单元 工具函数、数据转换逻辑错误 纯函数 不受影响 毫秒级
服务 业务逻辑、API 调用流程错误 Service 类 不受影响 秒级
组件 组件渲染、交互行为异常 UI 组件 部分影响 十秒级
E2E 完整流程断裂、页面布局崩溃 整个应用 直接影响 分钟级

设计原则

如何确保 UI 修改不破坏已有功能？
📊
80% 测试不依赖 UI
单元测试 + 服务层测试占 80%，UI 修改时这些测试不受影响
📸
快照对比容差
允许 0.1% 像素差异，字体/间距微调不触发失败
🎯
语义化定位
使用 testId 而非 CSS 选择器定位，布局变化不影响测试
分层的核心价值
⚡
快速反馈
底层测试秒级完成，开发时即时发现问题
🛡
故障隔离
测试失败时能快速定位是哪一层的问题
💰
成本控制
减少昂贵的 E2E 测试，用低成本测试覆盖大部分场景

与 KuiTest 设计思路的关联

🤖
KuiTest 核心理念借鉴
无规则设计 + AI 理解交互常识
💡 KuiTest 解决的问题
  • 传统测试规则维护成本高
  • UI 修改导致测试脚本大量失效
  • 测试覆盖率难以提升
✔ 本方案的对应策略
  • 分层测试：80% 不依赖 UI
  • 快照容差 + 语义化定位
  • 组件预期行为表（类似 KuiTest 的功能识别）
关键差异

KuiTest 依赖大模型推理组件功能，计算成本高但泛化能力强。
本方案采用人工定义预期行为表 + 规则驱动，适合功能明确的企业应用，维护成本可控。
